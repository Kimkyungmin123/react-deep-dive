# React Scheduler — `flushWork`와 `workLoop`: Task에서 Work로

## 1. Task와 Work의 개념 정리

| 개념     | 소유 주체  | 담고 있는 정보                                                                    |
| -------- | ---------- | --------------------------------------------------------------------------------- |
| **Task** | Scheduler  | 언제 Work를 실행할지 (`startTime`, `expirationTime`, `priorityLevel`, `callback`) |
| **Work** | Reconciler | 무엇을 렌더링할지 (`performConcurrentWorkOnRoot` 바인딩)                          |

Task의 `callback` 필드에 Reconciler의 Work 함수(`performConcurrentWorkOnRoot`)가 담겨 있다.  
Scheduler는 Task를 통해 적절한 시점에 이 callback을 호출함으로써 Reconciler에게 Work를 위임한다.

---

## 2. `performWorkUntilDeadline` → `flushWork` 진입 흐름

```
requestHostCallback(flushWork)
  └─ scheduledHostCallback = flushWork
  └─ port2.postMessage()
          │ (비동기, 다음 매크로태스크)
          ▼
performWorkUntilDeadline()
  └─ hasMoreWork = scheduledHostCallback(hasTimeRemaining, currentTime)
  //              └─ 즉, flushWork(hasTimeRemaining, currentTime) 호출
```

`performWorkUntilDeadline`이 `flushWork`를 호출하고, 그 반환값(`hasMoreWork`)으로 추가 Work 여부를 판단한다.

---

## 3. `flushWork` 함수 상세 분석

```javascript
function flushWork(hasTimeRemaining, initialTime) {
  // 1. 중복 실행 방지: 다음 스케줄을 받을 수 있도록 플래그 초기화
  isHostCallbackScheduled = false;

  // 2. 딜레이 타이머가 실행 중이면 취소
  //    (workLoop 내에서 advanceTimers가 직접 처리하기 때문)
  if (isHostTimeoutScheduled) {
    isHostTimeoutScheduled = false;
    cancelHostTimeout();
  }

  // 3. Work 수행 중임을 표시 (중첩 실행 방지)
  isPerformingWork = true;

  const previousPriorityLevel = currentPriorityLevel;
  try {
    // 4. workLoop 호출 → 반환값이 hasMoreWork
    return workLoop(hasTimeRemaining, initialTime);
  } finally {
    // 5. Work 완료 후 정리
    currentTask = null;
    currentPriorityLevel = previousPriorityLevel;
    isPerformingWork = false;
  }
}
```

**`isHostCallbackScheduled = false`로 먼저 초기화하는 이유:**  
`workLoop` 실행 중 새로운 Task가 추가될 경우, Scheduler가 다시 `requestHostCallback`을 호출하여 다음 실행을 예약할 수 있도록 플래그를 미리 해제해 두는 것이다.

## 4. `workLoop` 함수 상세 분석

```javascript
function workLoop(hasTimeRemaining, initialTime) {
  let currentTime = initialTime;
  advanceTimers(currentTime); // timerQueue → taskQueue 이동 먼저 처리

  currentTask = peek(taskQueue); // taskQueue의 최우선 Task 꺼내기

  while (currentTask !== null) {
    // 양보 조건: Task가 아직 만료되지 않았고 시간 예산을 초과했으면 break
    if (
      currentTask.expirationTime > currentTime &&
      (!hasTimeRemaining || shouldYieldToHost())
    ) {
      break; // 브라우저에 스레드 반환
    }

    const callback = currentTask.callback;

    if (typeof callback === "function") {
      currentTask.callback = null;
      currentPriorityLevel = currentTask.priorityLevel;

      const didUserCallbackTimeout = currentTask.expirationTime <= currentTime;
      // didUserCallbackTimeout: Task가 이미 만료되었는가 (true면 강제 처리)

      // callback 호출 → Reconciler의 performConcurrentWorkOnRoot 실행
      const continuationCallback = callback(didUserCallbackTimeout);

      currentTime = getCurrentTime();

      if (typeof continuationCallback === "function") {
        // Work가 아직 남아있음 → 다음 번에 이어서 처리
        currentTask.callback = continuationCallback;
        advanceTimers(currentTime);
        return true; // hasMoreWork = true
      } else {
        // Work 완료 → taskQueue에서 제거
        if (currentTask === peek(taskQueue)) {
          pop(taskQueue);
        }
        advanceTimers(currentTime);
      }
    } else {
      // callback이 null (취소된 Task) → 제거
      pop(taskQueue);
    }

    currentTask = peek(taskQueue); // 다음 Task로
  }

  // while 루프 탈출 후
  if (currentTask !== null) {
    return true; // 아직 처리할 Task 있음 (shouldYield로 break한 경우)
  } else {
    // taskQueue 비워짐 → timerQueue 확인
    const firstTimer = peek(timerQueue);
    if (firstTimer !== null) {
      // 딜레이된 Task가 남아있으면 타이머 재예약
      requestHostTimeout(handleTimeout, firstTimer.startTime - currentTime);
    }
    return false; // 더 이상 처리할 즉시 실행 Task 없음
  }
}
```

> ⚠️ **반환값의 의미 (원문 반대로 정정):**
>
> - `true` 반환 → **더 처리할 Task 있음** (performWorkUntilDeadline이 postMessage 재호출)
> - `false` 반환 → **즉시 실행할 Task 없음** (메시지 루프 종료)

---

## 5. `continuationCallback` — Work의 중단과 재개

`callback(didUserCallbackTimeout)`의 반환값이 핵심이다.

```javascript
// performConcurrentWorkOnRoot 내부 (Reconciler)
function performConcurrentWorkOnRoot(root, didTimeout) {
  // ...렌더링 작업...

  if (root.callbackNode === originalCallbackNode) {
    // Work가 아직 완료되지 않음 → 자기 자신을 continuation으로 반환
    return performConcurrentWorkOnRoot.bind(null, root);
  }
  // Work 완료 → null 반환 (또는 아무것도 반환하지 않음)
}
```

```
workLoop에서:
  continuationCallback = callback(didTimeout)
        │
        ├─ typeof continuationCallback === 'function'
        │     → currentTask.callback = continuationCallback (이어서 처리)
        │     → return true (hasMoreWork)
        │
        └─ typeof continuationCallback !== 'function'
              → pop(taskQueue) (Task 완료, 제거)
              → 다음 Task로 진행
```

## 6. `shouldYieldToHost` 조건의 정확한 해석

```javascript
if (
  currentTask.expirationTime > currentTime && // Task가 아직 만료 안 됨
  (!hasTimeRemaining || shouldYieldToHost())   // 시간 예산 소진됨
) {
  break;
}
```

**두 조건이 모두 필요한 이유:**

| 상황                    | `expirationTime > currentTime` | `shouldYieldToHost()` | 동작                        |
| ----------------------- | ------------------------------ | --------------------- | --------------------------- |
| 시간 초과, 만료 전 Task | true                           | true                  | **break** (브라우저에 양보) |
| 시간 초과, 만료된 Task  | false                          | true                  | 계속 실행 (강제 처리)       |
| 시간 여유, 만료 전 Task | true                           | false                 | 계속 실행                   |

만료된 Task(`expirationTime ≤ currentTime`)는 `shouldYieldToHost()`가 true여도 **강제로 실행**된다.  
이것이 starvation 방지 메커니즘이다.

## 7. 전체 흐름 요약

```
requestHostCallback(flushWork)
        │ MessageChannel.postMessage
        ▼
performWorkUntilDeadline
        │ deadline = currentTime + 5ms
        ▼
flushWork(hasTimeRemaining, currentTime)
  isHostCallbackScheduled = false
  isPerformingWork = true
        │
        ▼
workLoop(hasTimeRemaining, currentTime)
  ├─ advanceTimers (timerQueue → taskQueue 이동)
  ├─ while (currentTask !== null):
  │     ├─ shouldYieldToHost() + 만료 안 됨 → break
  │     ├─ callback 호출 (performConcurrentWorkOnRoot)
  │     │     ├─ continuationCallback 반환 → return true
  │     │     └─ null 반환 → pop(taskQueue)
  │     └─ 다음 Task
  │
  ├─ return true  (더 처리할 Task 있음)
  └─ return false (즉시 실행할 Task 없음)
        │
        ▼
performWorkUntilDeadline
  ├─ hasMoreWork = true  → port.postMessage() 재호출
  └─ hasMoreWork = false → isMessageLoopRunning = false (루프 종료)
```

## 8. React 18+과의 대응 개념

| React 16/17                    | React 18+                | 변화 요약                |
| ------------------------------ | ------------------------ | ------------------------ |
| `flushWork`                    | 동일하게 유지            | 변화 없음                |
| `workLoop`                     | 동일하게 유지            | 내부 구현 일부 개선      |
| `continuationCallback` 패턴    | 동일하게 유지            | 변화 없음                |
| `shouldYieldToHost` (5ms 기준) | `needsPaint` 플래그 추가 | 페인트 필요 시 즉시 양보 |
| `advanceTimers`                | 동일하게 유지            | 변화 없음                |
| `didUserCallbackTimeout`       | 동일하게 유지            | 만료 여부 전달 방식 동일 |

# React 18+ — `workLoop`와 Concurrent 렌더링의 중단·재개 메커니즘

## 1. `workLoop`가 Concurrent 렌더링의 핵심인 이유

React 16의 Stack Reconciler는 한 번 시작한 렌더링을 중단할 수 없었다.  
React 18의 Concurrent 렌더링이 중단·재개 가능한 이유는 `workLoop`의 **`continuationCallback` 패턴** 때문이다.

```
[Stack Reconciler - React 15]
renderTree() 시작 → 완료될 때까지 메인 스레드 독점
        └─ 중단 불가 → 사용자 입력 지연

[Fiber + workLoop - React 18]
workLoop 시작 → 5ms 후 shouldYield → break
        └─ continuationCallback 저장 → 브라우저에 제어권
        └─ 다음 매크로태스크에서 재개
```

---

## 2. `continuationCallback` — 중단과 재개의 핵심

### 2.1 작동 원리

`workLoop`는 `callback`의 반환값으로 중단 여부를 판단한다.

```javascript
const continuationCallback = callback(didUserCallbackTimeout);

if (typeof continuationCallback === "function") {
  // Work 미완성 → 이어서 처리
  currentTask.callback = continuationCallback;
  return true; // hasMoreWork
} else {
  // Work 완성 → Task 제거
  pop(taskQueue);
}
```

### 2.2 Reconciler의 `performConcurrentWorkOnRoot` 반환값

```javascript
// react-reconciler 내부 (단순화)
function performConcurrentWorkOnRoot(root, didTimeout) {
  // Fiber 트리 렌더링 작업 수행
  const exitStatus = renderRootConcurrent(root, lanes);

  if (exitStatus === RootInProgress) {
    // shouldYield()로 인해 중단됨 → 미완성
    return performConcurrentWorkOnRoot.bind(null, root);
    //     └─ 자기 자신을 반환 → workLoop에서 continuationCallback으로 저장
  }

  // 렌더링 완료 → 커밋 페이즈 진입
  finishConcurrentRender(root, exitStatus, lanes);
  return null; // Task 완료
}
```

### 2.3 중단·재개 사이클

```
workLoop 실행 (5ms 예산)
  performConcurrentWorkOnRoot 호출
    → workLoopConcurrent: Fiber 순회
    → shouldYield() = true → break
    → return performConcurrentWorkOnRoot.bind(null, root) ← 미완성 신호
        │
        ▼
workLoop: continuationCallback = function → return true
        │
        ▼
performWorkUntilDeadline: hasMoreWork = true → port.postMessage()
        │ (브라우저: 이벤트 처리, 페인트 기회)
        ▼
performWorkUntilDeadline 재실행
  workLoop: currentTask.callback = continuationCallback (이전에 저장)
  continuationCallback(didTimeout) → 중단된 지점부터 재개
```

## 3. starvation 방지 — 만료된 Task 강제 실행

`startTransition`으로 생성된 TransitionLane Task는 `NormalPriority`(timeout = 5000ms)를 가진다.  
계속 인터럽트되면 5초 후 만료 상태가 된다.

```javascript
// workLoop 양보 조건
if (
  currentTask.expirationTime > currentTime && // 아직 만료 전
  shouldYieldToHost()                          // 시간 예산 소진
) {
  break; // 양보
}

// 만료된 Task (expirationTime ≤ currentTime)는 위 조건 불만족
// → break 없이 강제 실행
```

```
[starvation 방지 시나리오]

startTransition Task 시작 (timeout = 5000ms)
  → 5ms 후 양보 (사용자 입력 처리)
  → 다시 시작 → 5ms 후 양보
  → ... (반복)
  → 5000ms 경과 → expirationTime ≤ currentTime
  → 다음 workLoop 진입 시 강제 실행 (shouldYield 무시)
  → 렌더링 완료
```

## 4. `didUserCallbackTimeout` — Task 만료 여부 전달

```javascript
const didUserCallbackTimeout = currentTask.expirationTime <= currentTime;
const continuationCallback = callback(didUserCallbackTimeout);
```

`performConcurrentWorkOnRoot`는 `didTimeout`을 받아 렌더링 방식을 결정한다.

```javascript
function performConcurrentWorkOnRoot(root, didTimeout) {
  // didTimeout = true: 만료 → 동기적으로 처리 (중단 없이)
  // didTimeout = false: 여유 → Concurrent로 처리 (중단 가능)
  const shouldTimeSlice = !didTimeout && !includesExpiredLane(root, lanes);

  const exitStatus = shouldTimeSlice
    ? renderRootConcurrent(root, lanes) // 중단 가능
    : renderRootSync(root, lanes); // 중단 불가 (만료 Task)
}
```

## 5. `flushWork`의 `isHostCallbackScheduled = false` — 왜 먼저 초기화하는가?

```javascript
function flushWork(hasTimeRemaining, initialTime) {
  isHostCallbackScheduled = false; // ← 작업 시작 전에 먼저 false로
  // ...
  return workLoop(hasTimeRemaining, initialTime);
}
```

**이 순서가 중요한 이유:**

```
workLoop 실행 중 새 setState 발생
        │
        ▼
ensureRootIsScheduled → requestHostCallback(flushWork)
  → isHostCallbackScheduled가 false? YES → scheduleCallback 가능
  → isHostCallbackScheduled = true → 다음 flushWork 예약 완료

만약 false로 초기화 안 했다면:
  → isHostCallbackScheduled가 true → "이미 예약됨" → scheduleCallback 스킵
  → 새 Task 누락 ← 버그!
```

## 6. React 18에서 추가된 `needsPaint`

React 18의 `shouldYieldToHost`는 5ms 외에 `needsPaint` 플래그도 확인한다.

```javascript
function shouldYieldToHost() {
  const timeElapsed = getCurrentTime() - startTime;

  if (timeElapsed < frameInterval) {
    return false; // 5ms 미만 → 계속
  }

  // 5ms 초과 시
  if (enableIsInputPending) {
    if (needsPaint) {
      return true; // 페인트 필요 → 즉시 양보
    }
    // 사용자 입력 pending 확인 (isInputPending API)
    if (timeElapsed < continuousInputInterval) {
      if (isInputPending !== null && isInputPending()) {
        return true; // 입력 대기 중 → 양보
      }
    }
  }
  return true; // 기본적으로 5ms 초과 시 양보
}
```

## 7. 전체 흐름 — Scheduler → Reconciler → Scheduler

```
[Scheduler → Reconciler → Scheduler 완전한 사이클]

Scheduler: workLoop
  currentTask.callback() 호출
        │ Reconciler로 진입
        ▼
Reconciler: performConcurrentWorkOnRoot
  workLoopConcurrent: Fiber 순회
  shouldYield() = true → 중단
        │ return performConcurrentWorkOnRoot.bind(root)
        ▼ Scheduler로 복귀
Scheduler: workLoop
  continuationCallback = function → return true
        │ hasMoreWork = true
        ▼
Scheduler: performWorkUntilDeadline
  port.postMessage() → 브라우저에 제어권
        │ (이벤트 처리, 페인트)
        ▼
Scheduler: performWorkUntilDeadline (재실행)
  workLoop → currentTask.callback = continuationCallback
        │ Reconciler로 재진입 (중단 지점부터)
        ▼ [렌더링 완료 시]
Reconciler: return null
        ▼
Scheduler: pop(taskQueue) → 다음 Task 또는 루프 종료
```

### 핵심 체크리스트

- [ ] `workLoop`가 `continuationCallback`으로 Concurrent 렌더링의 중단·재개를 구현함을 이해했다.
- [ ] `performConcurrentWorkOnRoot`가 미완성 시 자기 자신을 반환함을 이해했다.
- [ ] `workLoop` 반환값 `true` = 더 처리할 Work 있음, `false` = 즉시 실행할 Task 없음임을 이해했다.
- [ ] 만료된 Task는 `shouldYield()`를 무시하고 강제 실행되어 starvation을 방지함을 이해했다.
- [ ] `flushWork`가 `isHostCallbackScheduled = false`를 먼저 초기화하여 Work 중 새 Task 추가를 허용함을 이해했다.
- [ ] React 18의 `needsPaint` 플래그로 페인트 필요 시 5ms 이전에도 양보함을 이해했다.
