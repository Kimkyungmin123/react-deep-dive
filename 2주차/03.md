# React Fiber — Virtual DOM, 더블 버퍼링, 렌더 페이즈, 커밋 페이즈

## 1. Virtual DOM(VDOM)의 정의와 목적

### 1.1 VDOM은 프로그래밍 컨셉이다

VDOM은 특정 자료구조나 알고리즘을 가리키는 것이 아니라, **"메모리에 UI 관련 핵심 정보를 유지하고, 실제 DOM과 동기화하는 접근 방식"** 을 의미하는 프로그래밍 컨셉이다.

| 구분                         | 특징                                               |
| ---------------------------- | -------------------------------------------------- |
| **실제 DOM 조작**            | 브라우저 렌더링 파이프라인을 거쳐야 함 → 비용이 큼 |
| **VDOM (메모리 내 JS 객체)** | 순수 자바스크립트 연산 → 빠름                      |

React는 변경이 필요한 부분을 VDOM에서 먼저 계산한 뒤, **최소한의 실제 DOM 조작**만 수행하여 비용을 줄인다.

### 1.2 Reconciliation(재조정)의 정확한 범위

Reconciliation은 **VDOM을 실제 DOM과 동기화하는 전체 과정**을 의미한다.  
이 과정은 **Reconciler 패키지**가 담당하며, React DOM 같은 **렌더러 패키지**가 실제 DOM 조작을 수행한다.

```
Reconciler (react-reconciler)
    → VDOM 계산, 변경 사항 파악
Renderer (react-dom)
    → 실제 DOM 조작 수행
```

> ⚠️ **흔한 오해**: Reconciliation = 렌더 페이즈라고 이해하는 경우가 있다.  
> 정확히는 Reconciliation이 렌더 페이즈와 커밋 페이즈를 **모두 포함**하는 더 넓은 개념이다.

## 2. VDOM의 더블 버퍼링 구조

React는 항상 두 개의 Fiber 트리를 유지한다.

### 2.1 두 트리의 정의와 역할

| 트리                           | 정의                                               | 역할                                             |
| ------------------------------ | -------------------------------------------------- | ------------------------------------------------ |
| **Current Tree**               | 현재 화면에 렌더된 상태를 표현하는 Fiber 노드 트리 | 렌더링 중 원본 참조, 화면 안정성 보장            |
| **Work In Progress(WIP) Tree** | Current Tree에서 복제된, 작업 중인 Fiber 노드 트리 | 렌더 페이즈에서 추가·수정·삭제가 이루어지는 트리 |

두 트리의 Fiber 노드는 서로를 `alternate` 필드로 참조한다.

```
CurrentFiber.alternate          = WorkInProgressFiber
WorkInProgressFiber.alternate   = CurrentFiber
```

### 2.2 트리 전환 (더블 버퍼링)

커밋 페이즈가 완료되면 WIP 트리가 Current 트리로 **승격(flip)** 된다.

```
[커밋 페이즈 전]
FiberRoot.current → Current Tree
                    ↕ alternate
                    Work In Progress Tree  ← 렌더 페이즈에서 수정됨

[커밋 페이즈 완료 후]
FiberRoot.current → (기존 WIP) 새 Current Tree
                    ↕ alternate
                    (기존 Current) 다음 렌더의 WIP 후보
```

**더블 버퍼링을 유지하는 이유:**

| 이유             | 설명                                                     |
| ---------------- | -------------------------------------------------------- |
| 화면 안정성      | WIP 계산 중에도 Current는 화면에 표시된 상태 유지        |
| 중단·재시작 가능 | WIP 트리를 버려도 Current를 복제하여 새로 시작할 수 있음 |
| 빠른 트리 전환   | 포인터 하나(FiberRoot.current)만 변경하면 됨             |

> ⚠️ **"자기 복제"의 정확한 의미**: WIP 노드를 새로 만들 때 대응하는 Current 노드의 정보를 복사한다.  
> 트리 전체를 매번 새로 복사하는 것이 아니라, **필요한 노드만** WIP를 새로 생성하고 나머지는 Current를 재사용한다.

### 2.3 Fiber 노드의 트리 연결 구조

Fiber 트리는 일반 트리처럼 부모-자식 관계를 갖지만, 포인터 이름이 특이하다.

```
        Parent Fiber
       ┌──────────────┐
       │ child ───────┼──► Child1 Fiber
       │ return ◄─────┼─── (부모를 가리킴)
       └──────────────┘
                           Child1 Fiber
                          ┌──────────────┐
                          │ sibling ─────┼──► Child2 Fiber
                          │ return ──────┼──► Parent
                          └──────────────┘
```

| 필드      | 역할                        |
| --------- | --------------------------- |
| `child`   | 첫 번째 자식 Fiber를 가리킴 |
| `sibling` | 다음 형제 Fiber를 가리킴    |
| `return`  | 부모 Fiber를 가리킴         |

> ⚠️ **부모는 첫 번째 자식만 `child`로 직접 참조한다.**  
> 두 번째 이후 자식은 첫 번째 자식의 `sibling` 체인을 따라가야 접근할 수 있다.

## 3. 렌더링(Rendering)의 정확한 정의

**React에서 "렌더링"의 범위:**

```
컴포넌트 함수 호출
    │
    ▼
React 엘리먼트 반환 (JSX → React.createElement 결과)
    │
    ▼
Fiber 노드에 반영 (VDOM 업데이트)  ← 여기까지가 "렌더링"
    │
    ▼
실제 DOM 조작 (커밋 페이즈)        ← 렌더링에 포함되지 않음
    │
    ▼
브라우저 페인트                     ← 렌더링에 포함되지 않음
```

---

## 4. 렌더 페이즈 (Render Phase)

### 4.1 역할: Reconciliation 수행

렌더 페이즈는 **WIP 트리를 업데이트**하는 단계이다.  
컴포넌트를 호출하고, 반환된 React 엘리먼트를 기존 Fiber 노드와 비교하여 변경 사항(추가·수정·삭제)을 WIP Fiber 노드에 기록한다.

### 4.2 Work와 Scheduler

렌더 페이즈는 자동으로 시작되지 않는다. `setState` 등의 트리거가 발생하면:

```
setState 호출
    → Reconciler: Work 생성 → Scheduler에 등록
    → Scheduler: 적절한 시점에 Work 실행
    → 렌더 페이즈 시작 (컴포넌트 호출 → Fiber 업데이트)
```

### 4.3 Fiber 아키텍처 vs Stack 아키텍처

| 항목            | Stack Reconciler (React 15 이하) | Fiber Reconciler (React 16+) |
| --------------- | -------------------------------- | ---------------------------- |
| 처리 구조       | LIFO(후입선출) 스택              | 연결 리스트 기반 Fiber       |
| 작업 중단       | 불가                             | 가능                         |
| 우선순위 조정   | 불가                             | 가능                         |
| 작업 재시작     | 불가                             | 가능                         |
| Concurrent 지원 | 불가                             | 가능                         |

Fiber 아키텍처에서 렌더 페이즈는 **중단(interrupt), 폐기(discard), 재시작(restart)** 이 가능하다.  
이것이 React 18 Concurrent 기능(중단 가능한 렌더링, `startTransition`)의 기반이다.

> ⚠️ **렌더 페이즈는 순수해야 한다.**  
> 중단·재시작이 가능하기 때문에, 렌더 페이즈에서 부수효과(side effect)가 있으면 중단 후 재시작 시 두 번 실행될 수 있다.

## 5. 커밋 페이즈 (Commit Phase)

### 5.1 역할: 실제 DOM 적용 + 라이프사이클 실행

커밋 페이즈는 렌더 페이즈에서 계산된 WIP 트리의 변경 사항을 **실제 DOM에 반영**한다.

커밋 페이즈는 세 개의 하위 단계로 구성된다.

| 하위 단계           | 실행 내용                                                      |
| ------------------- | -------------------------------------------------------------- |
| **Before Mutation** | DOM 변경 전 스냅샷 저장, `getSnapshotBeforeUpdate` 호출        |
| **Mutation**        | 실제 DOM 노드 추가·삭제·수정                                   |
| **Layout**          | DOM 변경 후 `useLayoutEffect`, `componentDidMount/Update` 실행 |

### 5.2 커밋 페이즈는 항상 동기적(Synchronous)이다

렌더 페이즈는 중단 가능하지만, **커밋 페이즈는 중단 없이 동기적으로 완료**된다.

**이유:**

```
1. DOM의 일부만 변경된 채로 중단되면 화면이 불일치 상태가 됨
2. 브라우저는 콜 스택이 완전히 비워져야 페인트를 수행함
3. 커밋이 완료되어야만 브라우저가 변경된 DOM을 그릴 수 있음
```

```
커밋 페이즈 시작 (동기 실행)
    → Before Mutation → Mutation → Layout
    → FiberRoot.current 포인터 교체 (WIP → new Current)
    → 콜 스택 반환
브라우저 페인트 (별도 작업, React 관여 없음)
```

### 5.3 `useEffect` vs `useLayoutEffect`

| 훅                | 실행 시점                                     | 동기 여부    |
| ----------------- | --------------------------------------------- | ------------ |
| `useLayoutEffect` | 커밋 페이즈 Layout 단계 (DOM 업데이트 직후)   | 동기         |
| `useEffect`       | 커밋 페이즈 완료 **후**, 브라우저 페인트 이후 | 비동기(지연) |

## 6. React 18+과의 대응 개념

| React 16/17                                | React 18+                                  | 변화 요약                                 |
| ------------------------------------------ | ------------------------------------------ | ----------------------------------------- |
| VDOM 더블 버퍼링 구조                      | 동일하게 유지                              | 변화 없음                                 |
| Fiber 노드 `child`, `sibling`, `return`    | 동일하게 유지                              | 변화 없음                                 |
| 렌더 페이즈 중단 가능                      | 동일, **기본 동작으로 활성화**             | `createRoot` 사용 시 Concurrent 기본 활성 |
| 커밋 페이즈 동기 실행                      | 동일하게 유지                              | 변화 없음                                 |
| Work 우선순위: `expirationTime`            | **Lanes** 비트마스크                       | 우선순위 표현 방식 변경                   |
| `ReactDOM.render` → Current 트리 즉시 생성 | `createRoot` → Concurrent 경로로 트리 생성 | 초기 마운트 경로 변경                     |

📌 **React 18의 핵심 변화**: 렌더 페이즈가 **기본적으로 중단 가능한 Concurrent 방식**으로 동작한다.  
`ReactDOM.render()`(Legacy Mode)는 렌더 페이즈도 동기로 실행했지만, `createRoot()`를 사용하면 Scheduler가 5ms 단위로 브라우저에 제어권을 양보하며 렌더 페이즈를 처리한다.

```
  콜 스택 비워진 후 브라우저가 변경된 DOM을 화면에 그림
  useEffect 실행 (페인트 이후 비동기)
```

# React 18+ — Virtual DOM과 Concurrent 렌더링: 렌더 페이즈와 커밋 페이즈

## 1. Virtual DOM은 무엇인가?

### 1.1 VDOM = 메모리 내 UI 상태 표현

VDOM은 **"실제 DOM과 동기화할 UI 정보를 메모리에 유지하는 프로그래밍 접근 방식"** 이다.

```
컴포넌트 코드 (JSX)
    │ React.createElement 변환
    ▼
React 엘리먼트 (순수 JS 객체)
    │ Reconciler가 처리
    ▼
Fiber 노드 (VDOM의 실제 구현체)
    │ DOM과 동기화
    ▼
실제 DOM
```

React 엘리먼트와 Fiber 노드 모두 **자바스크립트 객체**이므로 DOM 조작보다 훨씬 빠르게 처리할 수 있다.

### 1.2 Fiber 노드의 구조

```javascript
// Fiber 노드 (단순화)
{
  type,           // 컴포넌트 함수 또는 'div' 같은 문자열
  stateNode,      // 실제 DOM 노드 참조
  memoizedState,  // 훅 상태값 (링크드 리스트)
  memoizedProps,  // 현재 props
  pendingProps,   // 업데이트 중인 props
  lanes,          // 처리 대기 중인 Lane (우선순위)
  child,          // 첫 번째 자식 Fiber
  sibling,        // 다음 형제 Fiber
  return,         // 부모 Fiber
  alternate,      // Current ↔ WIP 상호 참조
  flags,          // 변경 유형 표시 (Placement, Update, Deletion 등)
}
```

## 2. 더블 버퍼링 — Concurrent 렌더링의 기반

### 2.1 두 트리가 항상 존재하는 이유

React 18의 Concurrent 렌더링에서는 렌더 페이즈가 **중단되고 재시작**될 수 있다.

```
렌더 페이즈 진행 중 (WIP 트리 수정 중)
        │
        ├─ 더 긴급한 업데이트 도착!
        │
        ▼
WIP 트리 수정 중단
Current 트리는 그대로 → 화면에 변경 없음 ✅
        │
        ▼
긴급 업데이트 처리 → 커밋 → Current 트리 교체
        │
        ▼
이전 WIP 트리 폐기 후 새 WIP 트리 생성하여 재시작
```

Current 트리가 없다면 중단 시점에 화면이 반쯤 업데이트된 상태로 표시될 수 있다.  
더블 버퍼링이 **화면의 안정성**을 보장한다.

### 2.2 `createRoot`와 트리 생성

```jsx
// React 18
const root = ReactDOM.createRoot(document.getElementById("root"));
root.render(<App />);
```

- `createRoot`는 `FiberRootNode`(루트 컨테이너)를 생성한다.
- `FiberRootNode.current`가 Current 트리를 가리킨다.
- 초기 마운트 시 WIP 트리를 생성하여 `<App />`을 렌더링한다.
- 커밋 후 `FiberRootNode.current`가 WIP 트리를 가리키도록 전환된다.

## 3. 렌더 페이즈 — React 18에서 달라진 점

### 3.1 기본이 된 Concurrent 렌더링

React 17(`ReactDOM.render`) vs React 18(`createRoot`)의 렌더 페이즈 차이:

| 항목                 | React 17 `ReactDOM.render` | React 18 `createRoot` |
| -------------------- | -------------------------- | --------------------- |
| 렌더 페이즈 중단     | ❌ 불가                    | ✅ 5ms 단위로 양보    |
| 우선순위 조정        | ❌ 불가                    | ✅ Lane 기반 조정     |
| `startTransition`    | ❌ 효과 없음               | ✅ 작동               |
| 렌더링 도중 인터럽트 | ❌ 불가                    | ✅ 가능               |

### 3.2 렌더 페이즈에서 일어나는 일

```
Scheduler → Work 실행 (performConcurrentWorkOnRoot)
        │
        ▼
workLoopConcurrent()
  while (workInProgress !== null && !shouldYield()) {
    performUnitOfWork(workInProgress); // Fiber 하나 처리
  }
        │
        ├─ shouldYield() = true (5ms 초과) → 중단 → 다음 프레임에서 재개
        └─ workInProgress = null → 모든 Fiber 처리 완료 → 커밋 페이즈 진입
```

각 `performUnitOfWork`에서 수행하는 것:

1. 컴포넌트 함수 호출 (훅 실행 포함)
2. 반환된 React 엘리먼트를 자식 Fiber와 비교 (**diffing**)
3. 변경 사항을 `flags` 필드에 표시 (`Placement`, `Update`, `Deletion`)

### 3.3 렌더 페이즈는 순수해야 한다

렌더 페이즈는 중단·재시작이 가능하므로 **부수효과(side effect)가 있으면 두 번 실행될 수 있다.**

```jsx
// ❌ 렌더 페이즈 부수효과 예시
let count = 0;
function MyComponent() {
  count++; // 렌더 중단 후 재시작 시 두 번 실행됨
  return <div>{count}</div>;
}

// ✅ 부수효과는 useEffect로 분리
function MyComponent() {
  useEffect(() => {
    count++; // 커밋 페이즈 완료 후 실행 → 한 번만 실행 보장
  }, []);
  return <div />;
}
```

**Strict Mode 이중 렌더의 목적:**  
React 18 Strict Mode는 개발 환경에서 컴포넌트를 의도적으로 두 번 렌더링하여 위와 같은 부수효과를 조기에 발견한다.

## 4. 커밋 페이즈 — 변하지 않는 동기적 DOM 적용

### 4.1 커밋 페이즈는 항상 동기

렌더 페이즈가 Concurrent로 바뀌었어도, **커밋 페이즈는 React 18에서도 동기적**이다.

```
렌더 페이즈 (Concurrent, 중단 가능)
        ↓ 완료
커밋 페이즈 (Synchronous, 중단 불가) ← React 18에서도 동일
  1. Before Mutation
  2. Mutation (실제 DOM 조작)
  3. Layout (useLayoutEffect, componentDidMount)
        ↓ 완료
브라우저 페인트
useEffect 실행 (비동기)
```

### 4.2 세 단계의 구체적 내용

**Before Mutation:**

```jsx
// getSnapshotBeforeUpdate 실행 (클래스 컴포넌트)
// useEffect 정리 함수 스케줄링
```

**Mutation:**

```
Fiber 트리를 순회하며 flags 확인:
  Placement flag → DOM 노드 삽입
  Update flag    → DOM 속성 변경
  Deletion flag  → DOM 노드 제거
FiberRoot.current ← WIP 트리 (포인터 교체)
```

**Layout:**

```jsx
// useLayoutEffect 실행 (DOM 업데이트 직후, 페인트 전)
// componentDidMount / componentDidUpdate 실행
```

### 4.3 `useLayoutEffect` vs `useEffect` 실행 타이밍

```jsx
function Example() {
  useLayoutEffect(() => {
    // DOM 조작 완료 직후, 브라우저 페인트 전 실행
    // DOM을 읽거나 즉시 수정해야 할 때 사용
    console.log("1. useLayoutEffect");
  });

  useEffect(() => {
    // 브라우저 페인트 완료 후 실행
    // 대부분의 부수효과는 여기에
    console.log("2. useEffect");
  });

  return <div />;
}
// 출력 순서: 1 → (브라우저 페인트) → 2
```

> ⚠️ `useLayoutEffect`는 커밋 페이즈 내에서 동기로 실행된다.  
> 무거운 연산을 넣으면 브라우저 페인트가 지연될 수 있으므로 DOM 읽기·즉시 수정 외에는 `useEffect`를 사용한다.

## 5. 전체 흐름 요약

```
[React 18 렌더링 전체 흐름]

setState / startTransition 호출
        │
        ▼
Lane 배정 → Scheduler에 Work 등록
        │
        ▼
[렌더 페이즈] ── Concurrent (중단 가능)
  workLoopConcurrent: Fiber 순회
    ├─ 컴포넌트 함수 호출 (훅 실행)
    ├─ React 엘리먼트 생성
    ├─ 기존 Fiber와 비교 (diffing)
    └─ 변경 사항 flags로 표시
  5ms 초과 시 yield → 다음 프레임에서 재개
  더 긴급한 업데이트 시 중단 → 재시작
        │ 완료
        ▼
[커밋 페이즈] ── Synchronous (중단 불가)
  Before Mutation → Mutation → Layout
  실제 DOM 조작 → Current 트리 전환 → useLayoutEffect
        │ 완료
        ▼
브라우저 페인트
        │
        ▼
useEffect 실행 (비동기)
```
