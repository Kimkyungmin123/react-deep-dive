# React Fiber — `useState`의 실제 구현 위치: 코어에서 Reconciler까지

## 1. `import { useState } from 'react'`의 실체

개발자가 작성하는 코드:

```javascript
import { useState } from "react";
```

이것은 `react` 코어 패키지에서 `useState`를 가져오는 것처럼 보인다.  
그러나 `react` 패키지의 `useState`에는 **실제 구현 코드가 없다.**

### 1.1 `react` 패키지 내 `useState`의 실체

```javascript
// react/src/ReactHooks.js
export function useState(initialState) {
  const dispatcher = resolveDispatcher();
  return dispatcher.useState(initialState);
}
```

`resolveDispatcher` 함수는 단순히 `ReactCurrentDispatcher.current`를 반환한다.

```javascript
// react/src/ReactHooks.js
function resolveDispatcher() {
  const dispatcher = ReactCurrentDispatcher.current;
  if (dispatcher === null) {
    throw new Error(
      "Invalid hook call. Hooks can only be called inside of the body of a function component.",
    );
  }
  return dispatcher;
}
```

핵심: `ReactCurrentDispatcher.current`는 **초기값이 `null`** 이다.  
즉, `react` 패키지 자체에는 훅의 구현이 없고, 외부에서 주입된 디스패처 객체에게 위임한다.

---

## 2. 왜 `react` 패키지에 훅 구현이 없는가?

### 2.1 React 엘리먼트 vs Fiber 노드

`react` 패키지는 **React 엘리먼트**를 다루는 패키지이다.

```javascript
// React 엘리먼트: JSX가 변환된 순수 JS 객체
const element = React.createElement(MyComponent, { id: 1 });
// → { type: MyComponent, props: { id: 1 }, key: null, ref: null }
```

React 엘리먼트에는 훅 정보가 **포함되지 않는다.**  
훅 정보는 React 엘리먼트가 **Fiber 노드로 확장될 때** 비로소 생성된다.

```
React 엘리먼트 (react 패키지 영역)
    { type, props, key, ref }
        │ Reconciler가 확장
        ▼
Fiber 노드 (Reconciler 패키지 영역)
    { type, props, memoizedState(훅 링크드 리스트), queue, ... }
```

Fiber로의 확장 작업은 **Reconciler 패키지**가 담당하므로, 훅 구현도 Reconciler에 있다.

### 2.2 패키지 분리의 이유

React가 `react`와 `react-reconciler`를 분리하는 이유:

| 이유          | 설명                                                                                |
| ------------- | ----------------------------------------------------------------------------------- |
| 플랫폼 독립성 | 동일한 `react` 패키지로 웹(react-dom), 앱(react-native), 서버(react-server) 등 지원 |
| 의존성 최소화 | `react` 패키지가 특정 렌더러나 Reconciler에 직접 의존하지 않음                      |
| 커스텀 렌더러 | 개발자가 자체 렌더러를 만들 수 있음                                                 |

## 3. 훅 구현 코드 주입 경로: Dispatcher 패턴

### 3.1 전체 주입 경로

```
[사용자 코드]
  import { useState } from 'react'
        │
        ▼
[react 패키지] ReactHooks.js
  useState → resolveDispatcher() → ReactCurrentDispatcher.current
        │
        ▼ (current는 어디서 채워지는가?)
[shared 패키지] react-shared-internals.js (= ReactSharedInternals)
  → ReactCurrentDispatcher 객체를 공유하는 중간 통로
        │
        ▼
[react-reconciler 패키지]
  렌더 페이즈 진입 시:
  ReactCurrentDispatcher.current = HooksDispatcherOnMount (또는 OnUpdate)
  → current에 실제 훅 구현체 할당
```

### 3.2 `shared` 패키지의 역할

`shared` 패키지는 `react`와 `react-reconciler` 사이의 **공유 통로**이다.

```javascript
// packages/shared/ReactSharedInternals.js
const ReactSharedInternals =
  React.__SECRET_INTERNALS_DO_NOT_USE_OR_YOU_WILL_BE_FIRED;
export default ReactSharedInternals;
```

- `react` 패키지는 `ReactCurrentDispatcher`를 `__SECRET_INTERNALS_DO_NOT_USE_OR_YOU_WILL_BE_FIRED` 객체 안에 노출한다.
- `react-reconciler` 패키지는 `shared/ReactSharedInternals`를 통해 이 객체에 접근한다.
- 두 패키지가 **서로를 직접 import 하지 않으면서** 상태를 공유한다.

> ⚠️ **`__SECRET_INTERNALS_DO_NOT_USE_OR_YOU_WILL_BE_FIRED`**: 이름 그대로 외부에서 직접 사용하면 안 된다.  
> React 팀이 내부 전용임을 강하게 경고하기 위해 이런 이름을 붙였다. 실제로 접근하면 버전 간 호환성이 깨질 수 있다.

### 3.3 Dispatcher가 교체되는 시점

Reconciler는 컴포넌트 렌더링 단계에 따라 `ReactCurrentDispatcher.current`를 **다른 디스패처 객체**로 교체한다.

| 상황                  | 디스패처                    | 역할                             |
| --------------------- | --------------------------- | -------------------------------- |
| 렌더 페이즈 밖 (Idle) | `null`                      | 훅 호출 시 에러 발생             |
| 마운트 (최초 렌더)    | `HooksDispatcherOnMount`    | `mountState`, `mountEffect` 등   |
| 업데이트 (리렌더)     | `HooksDispatcherOnUpdate`   | `updateState`, `updateEffect` 등 |
| 렌더 페이즈 재진입    | `HooksDispatcherOnRerender` | 렌더 페이즈 setState 처리        |
| 서버 렌더링           | `HooksDispatcherOnServer`   | SSR용 구현                       |

```javascript
// renderWithHooks 내부 (단순화)
ReactCurrentDispatcher.current =
  current === null || current.memoizedState === null
    ? HooksDispatcherOnMount // 최초 렌더
    : HooksDispatcherOnUpdate; // 리렌더

// 컴포넌트 함수 호출
const children = Component(props);

// 렌더 완료 후 다시 null로 초기화
ReactCurrentDispatcher.current = ContextOnlyDispatcher;
```

## 4. Dispatcher 패턴이 해결하는 문제

### 4.1 훅 규칙 강제

`ReactCurrentDispatcher.current`가 컴포넌트 함수 실행 중에만 디스패처로 설정되기 때문에:

```javascript
// 컴포넌트 외부에서 useState 호출 시
function notAComponent() {
  const [s, setS] = useState(0); // resolveDispatcher() → current = null → Error!
}
// Error: Invalid hook call. Hooks can only be called inside of the body of a function component.
```

`null` 초기값이 **훅 규칙 위반을 자동으로 감지**하는 장치가 된다.

### 4.2 마운트와 업데이트의 다른 구현

같은 `useState` 호출이 **처음 렌더**에는 `mountState`를, **리렌더**에는 `updateState`를 실행한다.

```javascript
// HooksDispatcherOnMount
const HooksDispatcherOnMount = {
  useState: mountState, // 훅 객체 생성, 초기값 설정
  useEffect: mountEffect,
  // ...
};

// HooksDispatcherOnUpdate
const HooksDispatcherOnUpdate = {
  useState: updateState, // 큐의 업데이트 소비, 새 상태 반환
  useEffect: updateEffect,
  // ...
};
```

## 5. React 18+과의 대응 개념

| React 16/17                                          | React 18+     | 변화 요약 |
| ---------------------------------------------------- | ------------- | --------- |
| `ReactCurrentDispatcher.current`                     | 동일하게 유지 | 변화 없음 |
| `HooksDispatcherOnMount/Update`                      | 동일하게 유지 | 변화 없음 |
| `resolveDispatcher()`                                | 동일하게 유지 | 변화 없음 |
| `shared/ReactSharedInternals`                        | 동일하게 유지 | 변화 없음 |
| `__SECRET_INTERNALS_DO_NOT_USE_OR_YOU_WILL_BE_FIRED` | 동일하게 유지 | 변화 없음 |

📌 **React 18에서 추가된 디스패처**: `HooksDispatcherOnMountInDEV`, `HooksDispatcherOnUpdateInDEV` 등 개발 모드 전용 디스패처가 추가되었다. 개발 환경에서 훅 사용 오류를 더 상세하게 감지하기 위함이다.

## 6. 전체 흐름 요약

```
import { useState } from 'react'
        │
        ▼
react/ReactHooks.js
  useState(initialState)
    → resolveDispatcher()
    → ReactCurrentDispatcher.current.useState(initialState)
        │
        ▼ (current는 Reconciler가 채움)
react/ReactCurrentDispatcher.js
  { current: null }  ← 기본값
        │ shared 패키지를 통해 공유
        ▼
shared/ReactSharedInternals.js
  (react ↔ react-reconciler 공유 통로)
        │
        ▼
react-reconciler
  renderWithHooks() 진입 시:
    ReactCurrentDispatcher.current = HooksDispatcherOnMount (최초)
                                   = HooksDispatcherOnUpdate (리렌더)
        │
        ▼
컴포넌트 함수 실행
  useState() 호출 → current.useState() → mountState() 또는 updateState()
        │
        ▼
renderWithHooks() 종료 시:
    ReactCurrentDispatcher.current = ContextOnlyDispatcher (null과 유사, 호출 시 에러)
```

# React 18+ — Dispatcher 패턴과 훅 아키텍처 이해

## 1. `useState`를 `react`에서 가져오는데 구현은 왜 Reconciler에?

### 1.1 React의 패키지 구조

React는 역할에 따라 여러 패키지로 분리되어 있다.

```
react                  → 엘리먼트 생성, 훅 인터페이스 제공 (구현 없음)
react-dom              → 브라우저 DOM 렌더러
react-reconciler       → VDOM 계산, 훅 실제 구현
react-native-renderer  → React Native 렌더러
scheduler              → 작업 타이밍 관리
shared                 → 패키지 간 공유 객체
```

`react` 패키지가 훅 구현을 갖지 않는 이유:

```
react 패키지가 훅 구현을 가진다면?
    → react가 react-reconciler에 직접 의존
    → react-native는 자체 Reconciler를 사용 불가
    → 커스텀 렌더러 생태계 불가능
```

`react`를 **플랫폼 독립적**으로 유지하기 위해 훅 구현은 Reconciler에 위치한다.

### 1.2 어떤 환경에서도 같은 `useState`

```
[웹 환경]
import { useState } from 'react'  →  react-dom의 Reconciler가 구현 주입

[React Native 환경]
import { useState } from 'react'  →  react-native의 Reconciler가 구현 주입

[커스텀 렌더러]
import { useState } from 'react'  →  커스텀 Reconciler가 구현 주입
```

같은 `useState` API가 어떤 환경에서도 동작하는 것은 이 Dispatcher 주입 패턴 덕분이다.

---

## 2. Dispatcher 패턴 — 런타임 의존성 주입

### 2.1 전체 구조

```
[react 패키지]
  useState → resolveDispatcher() → ReactCurrentDispatcher.current.useState()

[shared 패키지]
  ReactCurrentDispatcher = { current: null }  ← 공유 객체

[react-reconciler 패키지]
  renderWithHooks 진입:
    ReactCurrentDispatcher.current = HooksDispatcherOnMount  (최초 렌더)
    ReactCurrentDispatcher.current = HooksDispatcherOnUpdate (리렌더)
  renderWithHooks 종료:
    ReactCurrentDispatcher.current = ContextOnlyDispatcher   (에러 디스패처)
```

### 2.2 디스패처가 교체되는 정확한 시점

```jsx
function MyComponent({ value }) {
  //  ↑ 컴포넌트 함수 진입 직전:
  //    ReactCurrentDispatcher.current = HooksDispatcherOnMount or Update

  const [state, setState] = useState(0);
  //  useState → current.useState → mountState or updateState

  useEffect(() => {
    /* ... */
  }, [value]);
  //  useEffect → current.useEffect → mountEffect or updateEffect

  return <div>{state}</div>;
  //  ↑ return 직후:
  //    ReactCurrentDispatcher.current = ContextOnlyDispatcher
}
```

## 3. 훅 규칙이 기술적으로 강제되는 방식

### 3.1 컴포넌트 외부에서 훅 호출

```javascript
// 컴포넌트 함수 밖에서 호출
function notAComponent() {
  const [s] = useState(0);
  // resolveDispatcher → ReactCurrentDispatcher.current = null → 에러 발생
}
// Error: Invalid hook call.
// Hooks can only be called inside of the body of a function component.
```

`ReactCurrentDispatcher.current`가 컴포넌트 실행 구간 밖에서는 항상 **에러를 던지는 디스패처**로 설정된다.  
에러 메시지 자체가 Dispatcher 패턴의 산물이다.

### 3.2 조건문 안에서 훅 호출

에러가 즉시 발생하지 않지만, 링크드 리스트 순서가 어긋나 **예측 불가한 상태 불일치**가 발생한다.

```jsx
// ❌ 잘못된 패턴
function Bad({ flag }) {
  if (flag) {
    const [a] = useState(0); // 조건부 실행 → 순서 어긋남
  }
  const [b] = useState("");
}

// Fiber.memoizedState 링크드 리스트:
//   flag=true:  Hook(a) → Hook(b)  (순서 1, 2)
//   flag=false: Hook(b)            (순서 1만 있음)
// → 리렌더 시 Hook(b)를 읽으려는데 Hook(a) 자리를 읽음 → 값 불일치
```

## 4. 개발 모드와 프로덕션 모드의 디스패처 차이

React 18은 개발 환경에서 더 상세한 오류 감지를 위한 **DEV 전용 디스패처**를 추가했다.

| 환경               | 디스패처                       | 특징                    |
| ------------------ | ------------------------------ | ----------------------- |
| 프로덕션, 마운트   | `HooksDispatcherOnMount`       | 최소 구현               |
| 프로덕션, 업데이트 | `HooksDispatcherOnUpdate`      | 최소 구현               |
| 개발, 마운트       | `HooksDispatcherOnMountInDEV`  | 추가 검증·경고 포함     |
| 개발, 업데이트     | `HooksDispatcherOnUpdateInDEV` | 추가 검증·경고 포함     |
| Strict Mode        | 이중 렌더 후 동일 디스패처     | 렌더 페이즈 순수성 검사 |

```javascript
// 개발 모드 디스패처의 useState는 추가 검증 포함 (개념적)
const HooksDispatcherOnMountInDEV = {
  useState(initialState) {
    // 잘못된 훅 사용 패턴 감지
    warnInvalidHookAccess(); // 비동기 컨텍스트에서 호출 여부 확인
    return mountState(initialState);
  },
};
```

## 5. 커스텀 렌더러 — Dispatcher 패턴의 확장성

Dispatcher 패턴 덕분에 React를 다양한 환경에 적용할 수 있다.

```javascript
// 커스텀 렌더러 예시 (react-reconciler 사용)
import ReactReconciler from "react-reconciler";

const hostConfig = {
  createInstance: (type, props) => {
    /* 커스텀 노드 생성 */
  },
  appendChildToContainer: (container, child) => {
    /* 커스텀 마운트 */
  },
  // ...
};

const MyRenderer = ReactReconciler(hostConfig);

// 이 렌더러를 사용하는 컴포넌트에서도 useState가 정상 동작
function MyApp() {
  const [count, setCount] = useState(0); // ← 커스텀 렌더러의 Reconciler가 구현 주입
  return /* 커스텀 노드 */;
}
```

## 6. `__SECRET_INTERNALS_DO_NOT_USE_OR_YOU_WILL_BE_FIRED`

`react`와 `react-reconciler`가 직접 의존하지 않으면서 객체를 공유하는 방식이다.

```javascript
// react 패키지 내부
export const __SECRET_INTERNALS_DO_NOT_USE_OR_YOU_WILL_BE_FIRED = {
  ReactCurrentDispatcher,
  ReactCurrentBatchConfig,
  // ...
};

// react-reconciler가 shared 패키지를 통해 접근
import ReactSharedInternals from "shared/ReactSharedInternals";
const { ReactCurrentDispatcher } = ReactSharedInternals;
```

> ⚠️ 이름에서 알 수 있듯 **외부 코드에서 직접 접근하면 절대 안 된다.**  
> 버전마다 구조가 바뀔 수 있으며, React 팀의 하위 호환성 보장 대상이 아니다.

## 7. 전체 흐름 요약

```
[React 18 훅 실행 전체 경로]

개발자 코드:
  import { useState } from 'react'
  const [s, setS] = useState(0)
        │
        ▼
react/ReactHooks.js:
  resolveDispatcher().useState(0)
        │ ReactCurrentDispatcher.current를 읽음
        ▼
shared/ReactSharedInternals:
  ReactCurrentDispatcher = { current: ??? }
        │ Reconciler가 렌더 직전에 채워줌
        ▼
react-reconciler/ReactFiberHooks.js:
  renderWithHooks() 진입
    current = null? → HooksDispatcherOnMount → mountState(0)
    current 있음?   → HooksDispatcherOnUpdate → updateState()
        │
        ▼
mountState(0):
  Hook 객체 생성 → Fiber.memoizedState에 연결
  Queue 생성 → dispatch 함수 생성
  return [0, dispatch]
        │
        ▼
렌더 완료 후:
  ReactCurrentDispatcher.current = ContextOnlyDispatcher
  (컴포넌트 함수 밖에서 훅 호출 시 에러 발생하도록)
```

### 핵심 체크리스트

- [ ] `react` 패키지의 `useState`에는 실제 구현이 없고, Reconciler가 Dispatcher를 주입함을 이해했다.
- [ ] `ReactCurrentDispatcher.current`가 렌더 페이즈 구간에서만 실제 구현체로 설정됨을 이해했다.
- [ ] 훅 규칙(컴포넌트 최상위 호출)이 링크드 리스트 순서 보장과 직결됨을 이해했다.
- [ ] `shared` 패키지가 `react`와 `react-reconciler`의 직접 의존 없이 객체를 공유하는 통로임을 이해했다.
- [ ] 마운트 시 `HooksDispatcherOnMount`, 업데이트 시 `HooksDispatcherOnUpdate`가 사용됨을 이해했다.
- [ ] 동일한 `useState` API가 웹·앱·커스텀 환경에서 모두 동작하는 이유를 이해했다.
