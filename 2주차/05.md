# React Fiber — `renderWithHooks`: 파이버와 훅을 연결하는 핵심 함수

## 1. `renderWithHooks`의 역할

`renderWithHooks`는 **컴포넌트 함수를 호출하면서 파이버와 훅을 연결**하는 함수이다.

```javascript
function renderWithHooks(current, workInProgress, Component, props, ...) {
  // 1. 렌더링 중인 파이버 설정
  // 2. 마운트/업데이트 판단 → 디스패처 선택
  // 3. 컴포넌트 함수 호출 (훅 실행)
  // 4. 렌더 페이즈 업데이트 처리
  // 5. 안전장치 디스패처로 교체
  // 6. 파이버에 훅 링크드 리스트 연결
  // 7. 전역 변수 초기화 후 children 반환
}
```

---

## 2. 단계별 상세 분석

### 2.1 1단계 — 렌더링 중인 파이버 설정

```javascript
currentlyRenderingFiber = workInProgress;
// 전역 변수에 현재 렌더링할 파이버를 등록
// → dispatchAction(setStateCall)에서 이 값을 참조하여 렌더 페이즈 여부를 판단
```

`currentlyRenderingFiber`는 전역 변수이다 (`let` 또는 `var` 키워드 없이 할당).  
이 값이 `null`이 아닌 구간이 곧 **렌더 페이즈**이다.

### 2.2 2단계 — 마운트/업데이트 판단 → 디스패처 선택

```javascript
// nextCurrentHook 결정
nextCurrentHook = current !== null ? current.memoizedState : null;
```

| `current` 파이버 | 의미                                       | `nextCurrentHook`                    |
| ---------------- | ------------------------------------------ | ------------------------------------ |
| `null`           | 이 Fiber가 처음 생성됨 → **마운트**        | `null`                               |
| `null`이 아님    | 이미 DOM에 반영된 정보 존재 → **업데이트** | `current.memoizedState` (첫 번째 훅) |

```javascript
// 디스패처 선택
ReactCurrentDispatcher.current =
  nextCurrentHook === null
    ? HooksDispatcherOnMount // 마운트: mountState, mountEffect 등
    : HooksDispatcherOnUpdate; // 업데이트: updateState, updateEffect 등
```

**`current.memoizedState`가 첫 번째 훅인 이유:**  
이전 렌더링에서 `renderWithHooks`의 마지막 단계(6단계)에서  
`workInProgress.memoizedState = firstWorkInProgressHook`으로 연결해 두었기 때문이다.  
다음 렌더링에서 이 Fiber가 Current가 되므로, `current.memoizedState`에는 이전 렌더의 첫 번째 훅이 담겨 있다.

### 2.3 3단계 — 컴포넌트 함수 호출

```javascript
let children = Component(props, secondArg);
// 컴포넌트 함수 실행 중 useState, useEffect 등이 호출됨
// → ReactCurrentDispatcher.current의 해당 메서드 실행
// → mountState 또는 updateState 실행
```

이 한 줄이 `renderWithHooks`의 핵심이다.  
컴포넌트 함수가 호출되는 동안 개발자가 작성한 모든 훅이 순서대로 실행된다.

### 2.4 4단계 — 렌더 페이즈 업데이트 처리

```javascript
if (didScheduleRenderPhaseUpdate) {
  // 컴포넌트 함수 실행 중 setState가 호출된 경우 (렌더 페이즈 업데이트)
  // do...while 루프로 재호출하여 업데이트 소비
  // 최대 25회 제한 (초과 시 "Too many re-renders" 에러)
  do {
    didScheduleRenderPhaseUpdate = false;
    numberOfReRenders += 1;
    // ...
    children = Component(props, secondArg);
  } while (didScheduleRenderPhaseUpdate);
}
```

> ⚠️ **원문 오류 정정**: `didScheduleRenderPhaseUpdate`는 "스케줄러 패키지에 Work를 전달했는지"를 나타내는 것이 **아니다.**  
> 이 플래그는 **렌더 페이즈 중 `setState`가 호출되었는지**를 나타낸다.  
> Scheduler와는 무관하며, `renderPhaseUpdates` Map에 업데이트를 저장하는 렌더 페이즈 업데이트 경로에서 `true`로 설정된다.

### 2.5 5단계 — 안전장치 디스패처로 교체

```javascript
ReactCurrentDispatcher.current = ContextOnlyDispatcher;
```

컴포넌트 함수 호출이 끝나면 즉시 `ContextOnlyDispatcher`로 교체한다.

```javascript
// ContextOnlyDispatcher 내부
const ContextOnlyDispatcher = {
  useState: throwInvalidHookError,
  useEffect: throwInvalidHookError,
  useContext: readContext, // context는 허용
  // ...
};
```

이 교체 이유:

- 컴포넌트 함수 외부에서 훅이 호출되면 즉시 에러를 던짐
- 언마운트, 에러 복구 등 예외 상황에서 훅 호출을 방지

### 2.6 6단계 — 파이버에 훅 링크드 리스트 연결

```javascript
workInProgress.memoizedState = firstWorkInProgressHook;
// 렌더 중 생성된 훅 링크드 리스트의 첫 번째 노드를 파이버에 연결
```

```
workInProgress Fiber
  └─ memoizedState
        │
        ▼
     Hook1 (useState)
     ├─ memoizedState: 0
     └─ next → Hook2 (useEffect)
                ├─ memoizedState: { deps, effect }
                └─ next → null
```

이 연결이 완료되어야 다음 렌더링에서 `current.memoizedState`로 이전 훅에 접근할 수 있다.

### 2.7 7단계 — 전역 변수 초기화 및 children 반환

```javascript
// 다음 컴포넌트 렌더링을 위해 전역 변수 초기화
currentlyRenderingFiber = null;
currentHook = null;
workInProgressHook = null;
firstWorkInProgressHook = null;
nextCurrentHook = null;
// ... 기타 전역 변수들

return children; // 컴포넌트 함수의 반환값 (React 엘리먼트)
```

전역 변수를 `null`로 초기화하는 이유: 이 값들이 남아있으면 **다음 컴포넌트의 렌더링에 오염**될 수 있기 때문이다.

## 3. 마운트와 업데이트 시 훅 함수 비교

```javascript
// 마운트: HooksDispatcherOnMount
{
  useState: mountState,        // Hook 객체 새로 생성, 초기값 설정
  useEffect: mountEffect,      // Effect 객체 새로 생성
  useRef: mountRef,            // ref 객체 새로 생성
  useMemo: mountMemo,          // 메모이제이션 값 최초 계산
}

// 업데이트: HooksDispatcherOnUpdate
{
  useState: updateState,       // 큐의 업데이트 소비, 새 상태 반환
  useEffect: updateEffect,     // deps 비교, 변경 시 Effect 재등록
  useRef: updateRef,           // 기존 ref 객체 반환
  useMemo: updateMemo,         // deps 비교, 변경 시 재계산
}
```

같은 `useState(0)` 호출이 마운트에서는 `mountState`를, 업데이트에서는 `updateState`를 실행한다.

## 4. `renderWithHooks` 전체 흐름

```
renderWithHooks(current, workInProgress, Component, props)
        │
        ├─[1] currentlyRenderingFiber = workInProgress
        │
        ├─[2] nextCurrentHook = current?.memoizedState ?? null
        │     ReactCurrentDispatcher.current =
        │       nextCurrentHook === null
        │         ? HooksDispatcherOnMount
        │         : HooksDispatcherOnUpdate
        │
        ├─[3] children = Component(props)
        │       → 훅 순서대로 실행 (mountState / updateState 등)
        │
        ├─[4] didScheduleRenderPhaseUpdate?
        │       YES → do...while 루프로 컴포넌트 재호출 (최대 25회)
        │
        ├─[5] ReactCurrentDispatcher.current = ContextOnlyDispatcher
        │       (컴포넌트 밖 훅 호출 시 에러)
        │
        ├─[6] workInProgress.memoizedState = firstWorkInProgressHook
        │       (파이버에 훅 링크드 리스트 연결)
        │
        ├─[7] 전역 변수 초기화 (null)
        │
        └─ return children
```

## 5. React 18+과의 대응 개념

| React 16/17                                              | React 18+                                                          | 변화 요약                             |
| -------------------------------------------------------- | ------------------------------------------------------------------ | ------------------------------------- |
| `renderWithHooks`                                        | 동일하게 유지                                                      | 함수명·역할 동일                      |
| `nextCurrentHook` 기반 마운트 판단                       | `current === null \|\| current.memoizedState === null` 조건 정밀화 | 판단 로직 일부 개선                   |
| `HooksDispatcherOnMount/Update`                          | 동일 + 개발 모드 전용 DEV 버전 추가                                | `HooksDispatcherOnMountInDEV` 등 추가 |
| `didScheduleRenderPhaseUpdate` 플래그                    | 동일하게 유지                                                      | 변화 없음                             |
| `ContextOnlyDispatcher`                                  | 동일하게 유지                                                      | 변화 없음                             |
| `workInProgress.memoizedState = firstWorkInProgressHook` | 동일하게 유지                                                      | 변화 없음                             |
| 전역 변수 초기화                                         | 동일하게 유지                                                      | 변화 없음                             |

📌 **React 18의 추가 사항**: Strict Mode에서 `renderWithHooks`가 **두 번 호출**된다.  
첫 번째 호출 결과를 버리고 두 번째 결과를 사용하여 렌더 페이즈 부수효과를 감지한다.  
두 번 호출되므로 전역 변수 초기화가 더욱 중요해졌다.

# React 18+ — `renderWithHooks`와 훅-파이버 연결 메커니즘

## 1. `renderWithHooks`가 하는 일 한 줄 요약

> **"컴포넌트 함수를 호출하면서, 마운트/업데이트에 맞는 훅 구현체를 연결하고, 결과를 Fiber에 저장하는 함수"**

---

## 2. 마운트와 업데이트를 구분하는 방법

### 2.1 `current` 파이버로 판단

```javascript
// renderWithHooks 내부
const isMount = current === null || current.memoizedState === null;

ReactCurrentDispatcher.current = isMount
  ? HooksDispatcherOnMount
  : HooksDispatcherOnUpdate;
```

| 상황        | `current` Fiber | `current.memoizedState` | 선택 디스패처             |
| ----------- | --------------- | ----------------------- | ------------------------- |
| 최초 마운트 | `null`          | -                       | `HooksDispatcherOnMount`  |
| 업데이트    | `null`이 아님   | 이전 훅 링크드 리스트   | `HooksDispatcherOnUpdate` |

### 2.2 `current.memoizedState`에 이전 훅이 있는 이유

이전 렌더의 `renderWithHooks` 마지막 단계에서:

```javascript
workInProgress.memoizedState = firstWorkInProgressHook;
```

이 WIP Fiber가 커밋 후 Current Fiber가 되므로,  
다음 렌더에서 `current.memoizedState`에는 **이전 렌더의 첫 번째 훅**이 담겨 있다.

```
이전 렌더 완료
  WIP.memoizedState = Hook1 → Hook2 → Hook3
        ↓ 커밋 후 포인터 전환
  Current.memoizedState = Hook1 → Hook2 → Hook3

다음 렌더 시작
  current.memoizedState !== null → HooksDispatcherOnUpdate 선택
```

---

## 3. 마운트 훅 vs 업데이트 훅의 차이

### 3.1 같은 API, 다른 구현

```jsx
// 개발자 입장: 동일한 호출
const [count, setCount] = useState(0);
```

```javascript
// 마운트 시 내부 실행
mountState(0) {
  const hook = mountWorkInProgressHook(); // 새 Hook 객체 생성
  hook.memoizedState = hook.baseState = 0; // 초기값 설정
  const queue = { pending: null, dispatch: null, ... };
  hook.queue = queue;
  const dispatch = queue.dispatch = dispatchSetState.bind(null, fiber, queue);
  return [hook.memoizedState, dispatch]; // [0, setCount]
}

// 업데이트 시 내부 실행
updateState() {
  return updateReducer(basicStateReducer); // 큐 소비 → 최신 상태 계산
}
```

### 3.2 마운트 시 훅 링크드 리스트 구성

컴포넌트 함수 내 훅이 **호출 순서**대로 링크드 리스트에 연결된다.

```jsx
function Counter() {
  const [count, setCount] = useState(0); // mountState → Hook1 생성
  const [name, setName] = useState(""); // mountState → Hook2 생성
  useEffect(() => {}, [count]); // mountEffect → Hook3 생성
}
```

```
Fiber.memoizedState → Hook1 → Hook2 → Hook3 → null
                      count   name    effect
```

## 4. 컴포넌트 호출과 Dispatcher 보호

### 4.1 컴포넌트 호출 전후 Dispatcher 변화

```javascript
// [컴포넌트 호출 전]
ReactCurrentDispatcher.current = HooksDispatcherOnMount; // 또는 OnUpdate

// [컴포넌트 함수 실행]
const children = Component(props);
// → 훅 호출들이 순서대로 실행됨

// [컴포넌트 호출 후 즉시]
ReactCurrentDispatcher.current = ContextOnlyDispatcher;
// → 이후 훅 호출 시 에러 발생
```

### 4.2 `ContextOnlyDispatcher`의 역할

```javascript
const ContextOnlyDispatcher = {
  useState: throwInvalidHookError,
  useEffect: throwInvalidHookError,
  useReducer: throwInvalidHookError,
  useRef: throwInvalidHookError,
  useContext: readContext, // context 읽기는 어디서든 허용
  // ...
};

function throwInvalidHookError() {
  throw new Error(
    "Invalid hook call. Hooks can only be called inside of the body of a function component.",
  );
}
```

이 장치 덕분에 다음과 같은 잘못된 사용이 즉시 감지된다:

```javascript
// 이벤트 핸들러에서 훅 호출
button.addEventListener("click", () => {
  const [s] = useState(0); // ContextOnlyDispatcher → throwInvalidHookError → Error
});
```

## 5. React 18 Strict Mode — 이중 렌더의 실제 동작

Strict Mode에서는 `renderWithHooks`가 개발 환경에서 **두 번 호출**된다.

```
[Strict Mode 개발 환경 renderWithHooks 흐름]

1차 renderWithHooks 호출
  → currentlyRenderingFiber 설정
  → HooksDispatcherOnMountInDEV 설정
  → Component(props) 호출 → 훅 실행
  → 결과 버림 (훅 객체, 상태 모두 폐기)
  → 전역 변수 초기화

2차 renderWithHooks 호출
  → 동일 과정 반복
  → 결과 실제 사용 (Fiber에 연결)
```

**이중 호출의 목적**: 렌더 페이즈에서 발생하는 부수효과를 조기에 발견하기 위함이다.

```jsx
// 이중 호출로 감지되는 문제
let renderCount = 0;

function MyComponent() {
  renderCount++; // 1차: 1, 2차: 2 → 값이 달라짐 → 부수효과 감지
  return <div>{renderCount}</div>;
}
```

## 6. `workInProgress.memoizedState` 연결 시점

```javascript
// renderWithHooks 종료 직전
renderedWork.memoizedState = firstWorkInProgressHook;
```

이 연결이 일어난 후에야:

1. 커밋 페이즈에서 WIP 트리가 Current 트리로 승격
2. 다음 렌더에서 `current.memoizedState`로 이전 훅에 접근 가능
3. `HooksDispatcherOnUpdate` 선택 → `updateState`로 큐 소비

---

## 7. 전역 변수 초기화의 중요성

`renderWithHooks` 내에서 사용하는 주요 전역 변수들:

| 변수                      | 역할                                | 초기화 필요 이유                 |
| ------------------------- | ----------------------------------- | -------------------------------- |
| `currentlyRenderingFiber` | 현재 렌더링 중인 Fiber              | null이 아니면 렌더 페이즈로 오인 |
| `currentHook`             | 현재 처리 중인 Current Fiber의 훅   | 이전 컴포넌트 훅이 남으면 오염   |
| `workInProgressHook`      | 현재 처리 중인 WIP Fiber의 훅       | 동일                             |
| `firstWorkInProgressHook` | WIP 훅 링크드 리스트의 첫 번째 노드 | 동일                             |

Strict Mode의 이중 렌더에서 1차 실행 후 이 변수들이 초기화되지 않으면 2차 실행이 오염된 상태로 시작된다.

## 8. 전체 흐름 요약

```
[React 18 renderWithHooks 전체 흐름]

renderWithHooks(current, workInProgress, Component, props)
        │
        ├─[1] currentlyRenderingFiber = workInProgress
        │
        ├─[2] current === null? → HooksDispatcherOnMount
        │     아님 → HooksDispatcherOnUpdate
        │     (Strict Mode DEV: InDEV 버전 사용)
        │
        ├─[3] children = Component(props)
        │       useState() → current.useState() → mount/updateState
        │       useEffect() → mount/updateEffect
        │       ...
        │
        ├─[4] didScheduleRenderPhaseUpdate?
        │       YES → do...while (최대 25회) 재호출
        │
        ├─[5] ReactCurrentDispatcher.current = ContextOnlyDispatcher
        │
        ├─[6] workInProgress.memoizedState = firstWorkInProgressHook
        │
        ├─[7] 전역 변수 초기화 (null)
        │
        └─ return children
```

### 핵심 체크리스트

- [ ] `current === null`이면 마운트, 아니면 업데이트임을 `renderWithHooks`에서 판단함을 이해했다.
- [ ] 같은 `useState` 호출이 마운트에서는 `mountState`, 업데이트에서는 `updateState`를 실행함을 이해했다.
- [ ] `ContextOnlyDispatcher`가 컴포넌트 함수 밖 훅 호출을 에러로 막는 안전장치임을 이해했다.
- [ ] `workInProgress.memoizedState = firstWorkInProgressHook`으로 파이버에 훅 링크드 리스트가 연결됨을 이해했다.
- [ ] Strict Mode에서 이중 렌더가 일어나며 1차 결과는 버려짐을 이해했다.
- [ ] 전역 변수 초기화가 다음 컴포넌트 렌더 오염 방지를 위해 필수임을 이해했다.
