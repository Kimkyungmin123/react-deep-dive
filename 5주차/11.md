# React Fiber — Work 스케줄링 사전 작업: `scheduleUpdateOnFiber`와 `markUpdateTimeFromFiberToRoot`

## 1. 사전 작업의 목적과 전체 흐름

`setState`가 호출되면 Reconciler는 Scheduler에 Work를 등록하기 **전에** 반드시 사전 작업을 수행한다.  
핵심은 두 가지이다.

1. **`expirationTime` 계산**: 이 Work를 얼마나 빨리 처리해야 하는지를 나타내는 기준값을 산출한다.
2. **루트까지 마킹**: 이벤트가 발생한 Fiber에서 VDOM 루트까지 올라가며 `expirationTime`을 기록한다.

```
setState 호출
      │
      ▼
scheduleUpdateOnFiber(fiber, expirationTime)
      │
      ├─ [1] markUpdateTimeFromFiberToRoot(fiber, expirationTime)
      │         → Fiber와 상위 노드들에 expirationTime 마킹
      │         → VDOM 루트 반환
      │
      ├─ [2] (Sync 여부 확인)
      │
      └─ [3] ensureRootIsScheduled(root)
                → Scheduler에 Work 등록
```

> ⚠️ **용어 정정**: 원문에서 "Expression Time"이라고 표기한 것은 오기이다.  
> 올바른 용어는 `expirationTime`이다. "Expression(표현식)"과는 무관하며, **만료 시간(Expiration Time)** 을 의미한다.

---

## 2. `expirationTime`이란?

### 2.1 역할과 의미

`expirationTime`은 **이 Work가 얼마나 긴급한가**를 나타내는 숫자 값이다.

| 값                          | 의미                                |
| --------------------------- | ----------------------------------- |
| `Sync` (최댓값, 1073741823) | 즉시 동기 처리 필요                 |
| 큰 값                       | 높은 우선순위 (더 빨리 처리해야 함) |
| 작은 값                     | 낮은 우선순위 (느긋하게 처리 가능)  |

> ⚠️ **원문 오류 정정**: 원문에서 "`expirationTime`은 `lane` 객체 안에 있으며 `lane.expirationTime`으로 접근한다"고 설명했지만, 이는 **틀린 내용**이다.  
> React 16에서 `expirationTime`은 **Fiber 객체에 직접 존재**한다 (`fiber.expirationTime`).  
> `lane`은 React 18에서 도입된 개념으로, React 16에는 존재하지 않는다.

### 2.2 `expirationTime` 계산 시점

`expirationTime`은 `computeExpirationForFiber` 함수에서 계산된다.  
계산 기준은 `requestCurrentTimeForUpdate()`가 반환하는 **현재 시각(`currentTime`)** 이다.

```javascript
// React 16 내부 (개념적 표현)
const currentTime = requestCurrentTimeForUpdate();
const expirationTime = computeExpirationForFiber(
  currentTime,
  fiber,
  suspenseConfig,
);
```

> ⚠️ **원문 오류 정정**: 원문에서 "action time을 `expirationTime`에 할당한다"고 했지만,  
> 정확한 변수명은 **`currentTime`** 이며, `requestCurrentTimeForUpdate()` 함수로 얻는다.

## 3. `markUpdateTimeFromFiberToRoot` — 핵심 사전 작업 함수

### 3.1 함수의 역할

```javascript
function markUpdateTimeFromFiberToRoot(fiber, expirationTime) {
  // 1. 이벤트가 발생한 Fiber에 expirationTime 마킹
  // 2. 부모를 향해 올라가며 childExpirationTime 마킹
  // 3. VDOM 루트(FiberRootNode) 반환
}
```

### 3.2 이벤트 발생 Fiber에 `expirationTime` 마킹

```javascript
// 현재 Fiber의 expirationTime 업데이트 (더 높은 우선순위만 덮어씀)
if (fiber.expirationTime < expirationTime) {
  fiber.expirationTime = expirationTime;
}

// alternate에도 동일 작업 (Current ↔ WIP 양쪽 커버)
const alternate = fiber.alternate;
if (alternate !== null && alternate.expirationTime < expirationTime) {
  alternate.expirationTime = expirationTime;
}
```

**`alternate`에도 동일 작업을 수행하는 이유:**  
Fiber 이중 버퍼링 구조에서 `fiber` 변수가 Current인지 Work in Progress인지 확정할 수 없다.  
`alternate`는 그 반대편을 가리키므로, 양쪽 모두에 마킹해야 어느 경로로 접근하더라도 정확한 값을 읽을 수 있다.

### 3.3 부모 노드를 향해 올라가며 `childExpirationTime` 마킹

> ⚠️ **원문 오류 정정**: 원문에서 "부모 노드를 가져오기 위해 `fiver.liber`를 사용한다"고 했지만,  
> 올바른 필드명은 **`fiber.return`** 이다. Fiber에서 부모 노드를 가리키는 포인터는 `.return`이다.

```javascript
let node = fiber.return; // .return이 부모 Fiber를 가리키는 포인터

while (node !== null) {
  const alternate = node.alternate;

  // 부모 노드의 childExpirationTime 업데이트
  if (node.childExpirationTime < expirationTime) {
    node.childExpirationTime = expirationTime;
    if (alternate !== null && alternate.childExpirationTime < expirationTime) {
      alternate.childExpirationTime = expirationTime;
    }
  }

  // HostRoot 태그에 도달하면 루트 식별
  if (node.tag === HostRoot) {
    return node.stateNode; // FiberRootNode 반환
  }

  node = node.return; // 한 단계 위로
}
```

### 3.4 `childExpirationTime`이 필요한 이유

`childExpirationTime`은 **"이 노드의 자식 중 가장 높은 우선순위의 업데이트"** 를 기록한 값이다.

```
App (childExpirationTime: 100)
  ├─ Header (childExpirationTime: 0)
  └─ Main (childExpirationTime: 100)
       └─ Button (expirationTime: 100) ← 이벤트 발생
```

재조정(Reconciliation) 과정에서 React는 이 값을 보고 **자식 트리에 업데이트가 있는지 빠르게 판단**하여 불필요한 하위 트리 탐색을 건너뛴다.

---

## 4. 루트(FiberRootNode)에 스케줄링 정보를 기록하는 이유

### 4.1 루트의 역할

- VDOM 하나당 루트(`FiberRootNode`) 하나가 존재한다.
- 루트에는 **현재 진행 중인 Work의 스케줄링 정보** (우선순위, callbackNode 등)가 저장된다.
- 여러 컴포넌트에서 동시에 업데이트가 발생해도, 루트 하나에서 **어떤 Work를 먼저 처리할지** 결정한다.

### 4.2 루트 식별 방법

```javascript
// HostRoot 태그를 가진 Fiber의 stateNode가 FiberRootNode
if (node.tag === HostRoot) {
  root = node.stateNode; // ← 이것이 FiberRootNode
  break;
}
```

## 5. React 18+과의 대응 개념

| React 16/17                        | React 18+                       | 변화 요약                                |
| ---------------------------------- | ------------------------------- | ---------------------------------------- |
| `fiber.expirationTime` (직접 필드) | **`fiber.lanes`** (비트마스크)  | Fiber에 저장되는 우선순위 표현 방식 변경 |
| `fiber.childExpirationTime`        | **`fiber.childLanes`**          | 자식 업데이트 추적 단위가 Lane으로 변경  |
| `markUpdateTimeFromFiberToRoot`    | `markUpdateLaneFromFiberToRoot` | 동일 역할, Lane 기반으로 재구현          |
| `expirationTime` 비교 (`>` 연산)   | Lane 비트 연산 (`\|=`, `&`)     | 우선순위 병합·비교 방식 변경             |
| `fiber.return`으로 부모 탐색       | 동일하게 유지                   | 구조 변화 없음                           |
| `HostRoot` 태그로 루트 식별        | 동일하게 유지                   | 구조 변화 없음                           |

📌 **핵심 차이**: React 16에서는 Fiber 하나에 하나의 `expirationTime`만 기록할 수 있었다.  
React 18의 `lanes`는 비트마스크이므로 **하나의 Fiber에 여러 우선순위 업데이트를 동시에** 기록할 수 있다.  
이것이 Automatic Batching과 Concurrent 기능 개선의 기반이 된다.


# React 18+ — Lanes 기반 Fiber 마킹과 스케줄링 사전 작업

## 1. React 16 `expirationTime` → React 18 `Lanes`

React 18은 기존의 `expirationTime` 기반 우선순위 시스템을 **Lanes(비트마스크)** 로 완전히 대체했다.

### 1.1 왜 Lanes로 바꿨나?

**`expirationTime`의 한계:**

```
fiber.expirationTime = 100  // 하나의 숫자 = 하나의 우선순위만 표현 가능
```

하나의 Fiber에 **동시에 여러 우선순위의 업데이트**가 발생하면 표현할 수 없었다.

**Lanes의 해결:**

```
fiber.lanes = 0b000101
//                │ │
//                │ └─ SyncLane (비트 1)
//                └─── DefaultLane (비트 3)
// → 두 가지 우선순위 업데이트가 동시에 pending 상태임을 하나의 값으로 표현
```

### 1.2 React 16 vs React 18 Fiber 필드 비교

| 필드                      | React 16/17                     | React 18+                       |
| ------------------------- | ------------------------------- | ------------------------------- |
| Fiber의 업데이트 우선순위 | `fiber.expirationTime`          | `fiber.lanes`                   |
| 자식 업데이트 추적        | `fiber.childExpirationTime`     | `fiber.childLanes`              |
| 루트까지 마킹 함수        | `markUpdateTimeFromFiberToRoot` | `markUpdateLaneFromFiberToRoot` |
| 비교 연산                 | `>` (숫자 비교)                 | 비트 연산 (`\|=`, `&`, `~`)     |

---

## 2. `markUpdateLaneFromFiberToRoot` — React 18의 사전 작업

### 2.1 함수의 역할 (React 16과 동일)

역할 자체는 React 16과 동일하다. 이벤트가 발생한 Fiber에서 VDOM 루트까지 올라가며 Lane을 마킹한다.

```
setState 호출 (이벤트 발생 Fiber)
        │
        ▼
markUpdateLaneFromFiberToRoot(fiber, lane)
        │
        ├─ [1] fiber.lanes에 lane 추가 (비트 OR 연산)
        │       fiber.alternate.lanes에도 동일 처리
        │
        ├─ [2] fiber.return으로 부모 순회
        │       각 부모의 childLanes에 lane 추가
        │       부모의 alternate.childLanes에도 동일 처리
        │
        └─ [3] HostRoot 도달 → FiberRootNode 반환
```

### 2.2 Lane 마킹 코드 (개념적)

```javascript
// React 18 내부 (개념적 표현)
function markUpdateLaneFromFiberToRoot(sourceFiber, lane) {
  // 이벤트 발생 Fiber에 lane 추가
  sourceFiber.lanes = mergeLanes(sourceFiber.lanes, lane);
  //                   └─ lanes |= lane (비트 OR)

  const alternate = sourceFiber.alternate;
  if (alternate !== null) {
    alternate.lanes = mergeLanes(alternate.lanes, lane);
  }

  // 부모를 향해 올라가며 childLanes 업데이트
  let node = sourceFiber.return;
  while (node !== null) {
    node.childLanes = mergeLanes(node.childLanes, lane);
    if (node.alternate !== null) {
      node.alternate.childLanes = mergeLanes(node.alternate.childLanes, lane);
    }

    if (node.tag === HostRoot) {
      return node.stateNode; // FiberRootNode 반환
    }
    node = node.return;
  }
}
```

### 2.3 `childLanes`가 중요한 이유

재조정(Reconciliation) 과정에서 React는 `childLanes`를 보고 **자식 트리를 탐색할지 건너뛸지** 결정한다.

```javascript
// 재조정 중 자식 건너뛰기 판단 (개념적)
if (!includesSomeLane(workInProgress.childLanes, renderLanes)) {
  // childLanes에 현재 렌더링 중인 Lane이 없음
  // → 이 자식 트리에는 처리할 업데이트 없음 → 건너뜀 (성능 최적화)
  return null;
}
```

이 덕분에 트리 전체를 탐색하지 않고 **업데이트가 있는 경로만** 따라 내려갈 수 있다.

## 3. `startTransition`이 Lane 마킹에 미치는 영향

React 18에서 `startTransition`을 사용하면 업데이트가 **TransitionLane**에 배정된다.

```jsx
import { startTransition } from "react";

function Search() {
  const [query, setQuery] = useState("");

  function handleChange(e) {
    startTransition(() => {
      setQuery(e.target.value); // → TransitionLane으로 마킹
    });
  }
  // ...
}
```

**Lane 마킹 결과 비교:**

```
일반 setState
  fiber.lanes = DefaultLane (0b010000)

startTransition 내부 setState
  fiber.lanes = TransitionLane (0b100000)

// 두 업데이트가 동시에 pending인 경우
  fiber.lanes = DefaultLane | TransitionLane (0b110000)
  //            ← expirationTime으로는 표현 불가능했던 상황
```

## 4. 루트(FiberRootNode)에 기록되는 정보

### 4.1 `pendingLanes` — React 18의 핵심 필드

React 18의 `FiberRootNode`에는 `pendingLanes`라는 필드가 추가되었다.

```
FiberRootNode
  ├─ pendingLanes      : 처리 대기 중인 모든 Lane의 합집합
  ├─ suspendedLanes    : Suspend로 인해 대기 중인 Lane
  ├─ pingedLanes       : 재시도 준비된 Lane
  ├─ expiredLanes      : 만료되어 즉시 처리해야 하는 Lane
  └─ callbackNode      : 현재 스케줄링된 Scheduler 태스크 핸들
```

`markUpdateLaneFromFiberToRoot`가 완료되면 루트의 `pendingLanes`에도 해당 Lane이 추가된다.

### 4.2 `getNextLanes` — 다음 처리할 Lane 결정

`ensureRootIsScheduled`에서 `getNextLanes(root)`를 호출하여 **pendingLanes 중 가장 높은 우선순위의 Lane 집합**을 선택한다.

```javascript
// 개념적 표현
function getNextLanes(root, wipLanes) {
  const pendingLanes = root.pendingLanes;
  // 비트 연산으로 가장 높은 우선순위 Lane 추출
  return getHighestPriorityLanes(pendingLanes);
}
```

## 6. 핵심 정리 체크리스트

- [ ] React 18에서 `expirationTime`이 `lanes`(비트마스크)로 대체된 이유를 이해했다.
- [ ] `fiber.lanes`는 해당 Fiber에 pending된 업데이트의 Lane 집합임을 이해했다.
- [ ] `fiber.childLanes`는 자식 트리에 pending된 업데이트가 있는지 빠르게 확인하기 위한 값임을 이해했다.
- [ ] `fiber.return`이 부모 Fiber를 가리키는 포인터임을 이해했다.
- [ ] `startTransition`이 `TransitionLane`을 배정하며, 일반 `setState`(`DefaultLane`)와 동시에 표현될 수 있음을 이해했다.
- [ ] `FiberRootNode.pendingLanes`에 모든 대기 중인 Lane이 집계됨을 이해했다.
