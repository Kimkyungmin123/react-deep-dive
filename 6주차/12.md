# React Fiber — `expirationTime`의 계산 방식과 의미

## 1. `expirationTime`이란?

`expirationTime`은 **Work의 우선순위를 판단하는 기준값**이다.

| 값의 크기       | 의미                                      |
| --------------- | ----------------------------------------- |
| 값이 클수록     | 더 먼저 발생한 이벤트 → **우선순위 높음** |
| 값이 작을수록   | 나중에 발생한 이벤트 → **우선순위 낮음**  |
| `Sync` (최댓값) | 동기적으로 즉시 처리해야 함               |
| `Idle` (2)      | 유휴 상태, 느긋하게 처리 가능             |
| `NoWork` (0)    | 처리할 Work 없음                          |

---

## 2. `expirationTime` 계산 흐름

### 2.1 전체 계산 경로

```
computeExpirationForFiber(currentTime, fiber)
        │
        ▼
requestCurrentTime() 또는 requestCurrentTimeForUpdate()
        │
        ▼
msToExpirationTime(now())
        │
        ├─ now() = performance.now() 또는 Date.now()
        │   → 브라우저 시작 시점부터의 경과 시간(ms)
        │   → 시간이 흐를수록 값이 커짐
        │
        └─ msToExpirationTime(ms):
            return MAGIC_NUMBER_OFFSET - ((ms / UNIT_SIZE) | 0)
            → 상수에서 경과 시간을 빼므로 시간이 흐를수록 값이 작아짐
```

### 2.2 `now()` 함수의 반환값

```javascript
// scheduler 패키지의 now() 함수
// performance.now()를 지원하면 사용, 아니면 Date.now() 폴백
const now =
  typeof performance.now === "function"
    ? () => performance.now()
    : () => Date.now();
```

- `performance.now()`: 브라우저가 시작된 시점을 `0`으로 잡고 경과 시간(ms)을 반환한다.
- `Date.now()`: Unix 기원(1970-01-01)부터의 경과 시간(ms)을 반환한다.
- 두 함수 모두 **시간이 흐를수록 반환값이 커진다.**

> ⚠️ **원문 오류 정정**: 원문에서 `internal_time_ms` 값이 10000보다 작으면 `scheduler_now`를 사용한다고 설명했지만, 이 분기의 실제 목적은 `performance.now()`의 **정밀도 문제를 회피**하기 위한 것이다. `Date.now()` 기반 폴백과의 전환 조건이며, 학습 단계에서는 "결국 `performance.now()` 또는 `Date.now()`를 사용한다"고 이해하면 충분하다.

### 2.3 `msToExpirationTime` 함수

```javascript
const UNIT_SIZE = 10; // 배칭을 위한 단위 크기
const MAGIC_NUMBER_OFFSET = BATCHED_UPDATES - 1; // 하단 §3 참고

function msToExpirationTime(ms) {
  // ms를 UNIT_SIZE로 나눈 후 정수화, MAGIC_NUMBER_OFFSET에서 뺌
  return MAGIC_NUMBER_OFFSET - ((ms / UNIT_SIZE) | 0);
}
```

**값의 변화 예시:**

```
브라우저 시작 후 100ms  → MAGIC_NUMBER_OFFSET - 10  = 1073741813
브라우저 시작 후 200ms  → MAGIC_NUMBER_OFFSET - 20  = 1073741803
브라우저 시작 후 1000ms → MAGIC_NUMBER_OFFSET - 100 = 1073741723
```

시간이 흐를수록 값이 줄어든다 → **나중에 발생한 이벤트일수록 작은 값** → 낮은 우선순위

## 3. 상수값 계층 구조와 `magicNumberOffset`

### 3.1 상수값 관계

```
maxSigned31BitInt = 1073741823   (2^30 - 1, 31비트 부호 있는 정수 최댓값)
        │
        └─ Sync = maxSigned31BitInt

Sync - 1 = 1073741822
        │
        └─ Batched (배칭 처리를 위한 경계값)

Batched - 1 = 1073741821
        │
        └─ MAGIC_NUMBER_OFFSET (일반 expirationTime 계산의 최댓값)
```

> ⚠️ **원문 오류 정정**: 원문에서 "`maxSigned31BitINT`는 30비트 정수 중 가장 큰 값"이라고 했지만,  
> 정확히는 **31비트 부호 있는 정수(signed 31-bit integer)의 최댓값**이다.  
> 비트 수와 값의 관계: 2^30 - 1 = 1073741823 (부호 비트 1개 + 값 비트 30개).

### 3.2 `MAGIC_NUMBER_OFFSET`을 사용하는 이유

`Sync` 값에서 직접 빼지 않고 `MAGIC_NUMBER_OFFSET`을 사용하는 이유는 **값 충돌 방지** 때문이다.

```
[충돌 발생 시나리오]
만약 MAGIC_NUMBER_OFFSET = Sync라면:
  브라우저 시작 직후 now() = 0ms
  msToExpirationTime(0) = Sync - 0 = Sync

  → 계산된 expirationTime === Sync
  → "이 Work는 동기 처리가 필요한가, 아니면 방금 발생한 일반 이벤트인가?" 구분 불가
```

`MAGIC_NUMBER_OFFSET = Sync - 2`를 사용하면:

```
msToExpirationTime(0) = (Sync - 2) - 0 = Sync - 2

→ 어떤 경우에도 계산된 expirationTime < Sync
→ Sync와 일반 expirationTime이 절대 겹치지 않음 ✅
```

### 3.3 `UNIT_SIZE = 10`의 의미 — 배칭(Batching)

```javascript
return MAGIC_NUMBER_OFFSET - ((ms / UNIT_SIZE) | 0);
//                              ^^^^^^^^^^^^^^^^^^
//                              ms를 10으로 나누고 정수화
```

10ms 이내에 발생한 이벤트들은 **나눗셈 후 같은 정수값**이 된다.  
→ 같은 `expirationTime` 값을 가지게 됨  
→ **하나의 Work로 배칭** 처리된다.

```
100ms에 발생한 이벤트: (100 / 10) | 0 = 10 → expirationTime = OFFSET - 10
105ms에 발생한 이벤트: (105 / 10) | 0 = 10 → expirationTime = OFFSET - 10 (동일!)
→ 두 이벤트가 같은 expirationTime → 배칭
```

## 4. `expirationTime` 특수값 정리

| 상수                  | 값           | 의미                                                    |
| --------------------- | ------------ | ------------------------------------------------------- |
| `Sync`                | `1073741823` | 동기 처리 필요. `expirationTime === Sync`이면 즉시 실행 |
| `Batched`             | `1073741822` | `batchedUpdates` 내부 업데이트                          |
| `MAGIC_NUMBER_OFFSET` | `1073741821` | 일반 expirationTime의 최댓값                            |
| `Idle`                | `2`          | 유휴 상태. 모든 Work가 처리된 후 실행                   |
| `Never`               | `1`          | 절대 처리하지 않음 (Offscreen 등)                       |
| `NoWork`              | `0`          | 처리할 Work 없음                                        |

> ⚠️ **원문 오류 정정**: 원문에서 `Idle = 2`라고 했는데, React 16 소스코드에서의 실제 값은 버전에 따라 다를 수 있다. 정확한 값보다는 **"Idle은 매우 낮은 우선순위를 나타내는 특수값"** 이라는 의미를 이해하는 것이 중요하다.

## 5. React 18+과의 대응 개념

| React 16/17                              | React 18+                           | 변화 요약                    |
| ---------------------------------------- | ----------------------------------- | ---------------------------- |
| `expirationTime` (정수, 시간 기반)       | **`Lane`** (비트마스크)             | 우선순위 표현 방식 전면 교체 |
| `msToExpirationTime(ms)`                 | `requestUpdateLane(fiber)`          | 우선순위 결정 방식 변경      |
| `MAGIC_NUMBER_OFFSET - (ms / UNIT_SIZE)` | 비트 연산 (`\|=`, `&`)              | 계산 방식 변경               |
| `Sync` (숫자 최댓값으로 특수 처리)       | `SyncLane` (비트마스크 최우선 Lane) | 표현 방식만 변경, 역할 동일  |
| `Idle` (숫자 2)                          | `IdleLane`                          | 표현 방식만 변경, 역할 동일  |
| `NoWork` (0)                             | `NoLanes` (0)                       | 동일                         |
| 10ms 단위 배칭 (UNIT_SIZE)               | Automatic Batching (Lane 기반)      | 더 정교한 배칭으로 개선      |

📌 **핵심 변화**: React 16에서는 시간 기반 계산(`performance.now()`)으로 우선순위를 결정했다.  
React 18에서는 이 방식을 완전히 폐기하고 **업데이트가 어떤 컨텍스트에서 발생했는가** (이벤트 핸들러인지, `startTransition`인지, `flushSync`인지)를 기준으로 Lane을 직접 배정한다.  
시간 기반 추론에서 **의도 기반 선언**으로의 전환이다.

## 6. 전체 흐름 요약

```
setState 호출 (이벤트 발생 시점: 예, 브라우저 시작 후 500ms)
        │
        ▼
requestCurrentTimeForUpdate()
        └─ now() → performance.now() → 500 (ms)
        └─ msToExpirationTime(500)
             = MAGIC_NUMBER_OFFSET - ((500 / 10) | 0)
             = 1073741821 - 50
             = 1073741771

        ※ 1000ms에 발생한 이벤트: 1073741821 - 100 = 1073741721 (더 작음 = 낮은 우선순위)
        ※ 500ms 이벤트 > 1000ms 이벤트 → 500ms 이벤트가 먼저 처리됨
        │
        ▼
computeExpirationForFiber(currentTime=1073741771, fiber)
        → 우선순위 레벨에 따라 추가 조정
        │
        ▼
markUpdateTimeFromFiberToRoot(fiber, expirationTime)
        → Fiber ~ 루트까지 expirationTime 마킹
        │
        ▼
ensureRootIsScheduled(root)
        → Scheduler에 Work 등록
```

# React 18+ — Lane 기반 우선순위 모델: `expirationTime`에서 `Lanes`로

## 1. 왜 `expirationTime`을 버렸나?

React 16/17의 `expirationTime`은 **"언제 발생했는가"** 로 우선순위를 결정했다.

```
브라우저 시작 후 100ms 발생 → expirationTime = 1073741811  (높은 우선순위)
브라우저 시작 후 500ms 발생 → expirationTime = 1073741771  (낮은 우선순위)
```

이 방식의 한계:

| 한계           | 설명                                                    |
| -------------- | ------------------------------------------------------- |
| 단일 우선순위  | Fiber 하나에 하나의 숫자 → 여러 우선순위 동시 표현 불가 |
| 시간 기반 추론 | "언제 발생했는지"로 긴급도를 추론 → 부정확한 경우 존재  |
| 배칭 경직성    | 10ms 단위 고정 배칭 → 유연하지 않음                     |

React 18은 이 모든 한계를 **Lanes(비트마스크)** 로 해결했다.

---

## 2. Lanes 모델의 핵심 개념

### 2.1 Lane이란?

Lane은 **비트 하나**로 표현되는 우선순위 슬롯이다.  
비트마스크이므로 **여러 Lane을 동시에** 하나의 값으로 표현할 수 있다.

```javascript
// React 18 Lane 상수 (개념적 표현)
const SyncLane = 0b0000000000000000000000000000001; // 비트 1
const InputContinuousLane = 0b0000000000000000000000000000100; // 비트 3
const DefaultLane = 0b0000000000000000000000000010000; // 비트 5
const TransitionLane1 = 0b0000000000000000000001000000000; // 비트 9
const IdleLane = 0b0010000000000000000000000000000; // 비트 29
const NoLanes = 0b0000000000000000000000000000000; // 0
```

### 2.2 `expirationTime` vs `Lanes` 비교

| 항목               | `expirationTime` (React 16) | `Lanes` (React 18)         |
| ------------------ | --------------------------- | -------------------------- |
| 우선순위 표현 단위 | 정수 (숫자 1개)             | 비트마스크                 |
| 동시 표현 가능 수  | 1개                         | 31개 (비트 수)             |
| 우선순위 결정 기준 | 이벤트 발생 시간            | 업데이트 발생 컨텍스트     |
| 비교 연산          | `>`, `<` (숫자 비교)        | `&`, `\|`, `~` (비트 연산) |
| 배칭 단위          | 10ms 고정                   | Lane 단위 (유연)           |

## 3. 우선순위 결정 방식: 시간 → 컨텍스트

React 18에서는 `requestUpdateLane(fiber)` 함수가 업데이트의 Lane을 결정한다.  
결정 기준은 **어떤 컨텍스트에서 업데이트가 발생했는가**이다.

```javascript
function requestUpdateLane(fiber) {
  // 1. Legacy Sync 모드 (ReactDOM.render 사용 시)
  if (mode === NoMode) return SyncLane;

  // 2. 현재 렌더 중인 Lane 사용 (렌더 페이즈 setState)
  if (isRenderPhaseUpdate(fiber)) return currentlyRenderingLane;

  // 3. 이벤트 핸들러에서의 우선순위
  const updateLane = getCurrentUpdatePriority();
  if (updateLane !== NoLane) return updateLane;

  // 4. 이벤트 시스템의 현재 이벤트 우선순위
  return getCurrentEventPriority();
}
```

### 3.1 컨텍스트별 Lane 배정

| 컨텍스트                      | 배정되는 Lane                         | 비고                    |
| ----------------------------- | ------------------------------------- | ----------------------- |
| `flushSync` 내부              | `SyncLane`                            | 즉시 동기 처리          |
| 클릭, 키보드 이벤트 핸들러    | `SyncLane` 또는 `InputContinuousLane` | 이벤트 종류에 따라 다름 |
| 일반 `setState` (이벤트 외부) | `DefaultLane`                         | Scheduler에 위임        |
| `startTransition` 내부        | `TransitionLane` (1~16)               | 낮은 우선순위           |
| `useDeferredValue`            | `TransitionLane`                      | 파생 업데이트           |
| 유휴 상태 업데이트            | `IdleLane`                            | 가장 낮은 우선순위      |

### 3.2 여러 Lane 동시 pending 예시

```jsx
function App() {
  const [count, setCount] = useState(0);
  const [query, setQuery] = useState("");

  function handleChange(e) {
    setCount((c) => c + 1); // DefaultLane
    startTransition(() => {
      setQuery(e.target.value); // TransitionLane
    });
  }
}
```

```
fiber.lanes = DefaultLane | TransitionLane
           = 0b010000 | 0b100000
           = 0b110000
// 두 우선순위가 하나의 값에 동시 표현됨
// React 16에서는 불가능했던 표현
```

## 4. Lane 비트 연산 기초

React 18 소스코드를 읽을 때 자주 등장하는 비트 연산이다.

```javascript
// Lane 추가 (OR 연산)
fiber.lanes = fiber.lanes | newLane;
// 또는 mergeLanes 함수
function mergeLanes(a, b) {
  return a | b;
}

// Lane 포함 여부 확인 (AND 연산)
function includesSomeLane(lanes, lane) {
  return (lanes & lane) !== NoLanes;
}

// Lane 제거 (AND NOT 연산)
function removeLanes(set, subset) {
  return set & ~subset;
}

// 가장 높은 우선순위 Lane 추출 (최하위 비트)
function getHighestPriorityLane(lanes) {
  return lanes & -lanes; // 비트 트릭: 최하위 set 비트 추출
}
```

## 5. `childLanes` — 자식 트리 업데이트 추적

React 16의 `childExpirationTime`이 React 18에서는 `childLanes`로 대체되었다.

```
App (childLanes: DefaultLane | TransitionLane)
  ├─ Header (childLanes: NoLanes)
  └─ Main (childLanes: DefaultLane | TransitionLane)
       ├─ SearchInput (lanes: DefaultLane)      ← setCount 발생
       └─ Results (lanes: TransitionLane)       ← setQuery 발생
```

재조정 시 React는 `childLanes`를 보고 **해당 Lane의 업데이트가 있는 자식만 탐색**한다.

```javascript
// DefaultLane 렌더링 중 TransitionLane 자식 건너뜀
if (!includesSomeLane(workInProgress.childLanes, renderLanes)) {
  return null; // 이 자식 트리는 현재 렌더링과 무관 → 건너뜀
}
```

## 7. 전체 정리

```
[React 18 우선순위 결정 흐름]

setState 호출
        │
        ▼
requestUpdateLane(fiber)
  → 어떤 컨텍스트인가?
        ├─ flushSync          → SyncLane
        ├─ 이벤트 핸들러       → SyncLane / InputContinuousLane
        ├─ 일반 (외부)         → DefaultLane
        ├─ startTransition    → TransitionLane
        └─ 유휴                → IdleLane
        │
        ▼
markUpdateLaneFromFiberToRoot(fiber, lane)
  → fiber.lanes |= lane
  → 부모들의 childLanes |= lane
  → FiberRootNode.pendingLanes |= lane
        │
        ▼
ensureRootIsScheduled(root)
  → getNextLanes(root) → 가장 높은 우선순위 Lane 선택
  → Scheduler에 Work 등록
```

### 핵심 체크리스트

- [ ] Lane은 비트 하나로 표현되는 우선순위 슬롯임을 이해했다.
- [ ] React 18에서 우선순위는 시간 기반이 아닌 **컨텍스트 기반**으로 결정됨을 이해했다.
- [ ] `startTransition`이 `TransitionLane`을 배정하여 낮은 우선순위로 처리됨을 이해했다.
- [ ] `fiber.lanes`에 여러 Lane이 동시에 표현될 수 있음을 이해했다.
- [ ] `childLanes`가 자식 트리 탐색 최적화에 사용됨을 이해했다.
- [ ] `SyncLane`, `DefaultLane`, `TransitionLane`, `IdleLane`의 차이를 이해했다.
