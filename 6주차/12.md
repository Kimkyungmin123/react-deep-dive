# 리액트 입장에서는 어떤 이벤트를 먼저 처리해야할까? (expirationTime 값의 의미)

### 한 줄 요약

React 16에서 `expirationTime`은 업데이트의 **우선순위와 만료 시점**을 표현하는 내부 스케줄링 값이다.  
단순한 이벤트 발생 시간이 아니라, **현재 시간 + 업데이트 성격 + 배치 정책**을 반영해 계산된 값이다.  
이후 React 18+에서는 이 방식이 **`lane` 기반 우선순위 시스템**으로 크게 변경되었다.

---

## 1. `expirationTime`이란?

React 16에서 `expirationTime`은 업데이트를 스케줄링할 때 사용하는 **내부 우선순위 값**이다.

React는 업데이트가 발생하면 해당 작업에 `expirationTime`을 부여하고,  
이 값을 기준으로:

- 어떤 작업을 더 먼저 처리할지
- 어떤 작업들을 같은 배치로 묶을지
- 루트에 어떤 우선순위의 작업이 쌓여 있는지

를 판단했다.

즉, `expirationTime`은 단순한 숫자가 아니라  
**React 16 스케줄링의 핵심 기준값**이었다.

---

## 2. 오해하면 안 되는 점

### 2-1. `expirationTime`은 단순한 "이벤트 발생 시각"이 아니다

처음 보면 이렇게 이해하기 쉽다.

> `expirationTime` 값이 크면 더 먼저 발생한 이벤트다.

이 설명은 **완전히 틀린 건 아니지만, 정확한 표현은 아니다.**

왜냐하면 `expirationTime`은 단순히 “언제 발생했는가”만 담는 값이 아니라,  
**현재 시각을 기반으로 React가 해당 업데이트를 얼마나 급하게 처리해야 하는지**를 반영해 계산한 값이기 때문이다.

즉 더 정확하게는:

> `expirationTime`은 이벤트 발생 시각 그 자체가 아니라,  
> **우선순위가 반영된 만료 시점(deadline-like value)** 이다.

---

### 2-2. "값이 클수록 먼저 발생한 이벤트"라고 단정하면 위험하다

일부 React 16.x 구현에서는  
**값이 클수록 더 높은 우선순위로 취급되는 구조**가 맞다.

하지만 그 값은 단순 timestamp가 아니라:

- 현재 시간
- 업데이트 종류
- batch window
- expiration 정책

이 함께 반영된 결과다.

따라서 복기할 때는 이렇게 기억하는 게 가장 안전하다.

> React 16의 `expirationTime`은  
> “언제 발생했는가”보다  
> “얼마나 급하게 처리해야 하는가”를 숫자로 표현한 값이다.

---

## 3. 계산 방식의 핵심 흐름

React 16에서는 대략 다음 흐름으로 `expirationTime`이 계산된다.

1. `now()`로 현재 시간(ms)을 구한다.
2. `getCurrentTime()`이 현재 시간을 React 내부 표현으로 변환한다.
3. `computeExpirationForFiber()`가 현재 시간과 업데이트 종류를 바탕으로 최종 `expirationTime`을 계산한다.

핵심은:

- 현재 시간을 그대로 쓰지 않는다.
- React 내부 단위로 변환한다.
- 업데이트 종류에 따라 다른 expiration 정책을 적용한다.

---

### 3-1. `now()`

React 내부의 `now()`는 대체로 현재 시간을 ms 단위로 구하기 위한 함수다.  
환경에 따라 `performance.now()` 또는 이에 준하는 값을 사용한다.

중요한 포인트는:

- 시간이 흐를수록 `now()` 값은 커진다.
- 하지만 최종 `expirationTime`은 이 값을 그대로 저장한 값이 아니다.

---

### 3-2. `getCurrentTime()` / 시간 변환

React는 브라우저 시간(ms)을 바로 우선순위 값으로 쓰지 않고,  
내부 스케줄링에 맞는 단위로 변환해 사용했다.

즉:

- 브라우저 시간 → React 내부 시간 표현
- 그 다음 우선순위 정책 반영
- 최종 `expirationTime` 산출

이라는 흐름이다.

---

### 3-3. `computeExpirationForFiber()`

이 함수는 Fiber에 대한 업데이트의 `expirationTime`을 계산하는 핵심 함수다.

이때 단순히 현재 시간만 보는 것이 아니라,  
업데이트의 성격에 따라 다른 expiration window를 적용한다.  
예를 들어 interactive update와 일반 async update는 같은 시점에 발생해도  
서로 다른 `expirationTime`을 가질 수 있다.

즉:

> 같은 시각에 발생한 업데이트라도  
> 우선순위 정책이 다르면 `expirationTime`도 달라질 수 있다.

---

## 4. `magicNumberOffset`은 무엇인가?

React 16 계열 설명을 보다 보면 `magicNumberOffset`이 등장한다.

이 값은 내부에서 사용하는 특수 상수값들과  
실제 계산된 시간 기반 값이 충돌하지 않도록 하기 위한 보정 개념으로 이해하면 된다.

즉 목적은:

- `Sync`
- `NoWork`
- `Idle`
- `Batched`

같은 예약된 값들과  
실제 계산값이 겹치지 않도록 구분하는 것이다.

복기 포인트:

> `magicNumberOffset`의 핵심 목적은  
> **특수 상태값과 일반 계산값의 충돌 방지**다.

---

## 5. 관련 상수들

React 16의 `expirationTime` 체계에서는 몇 가지 특수값이 함께 등장한다.

### 5-1. `NoWork`

현재 스케줄링할 작업이 없는 상태를 뜻한다.

- 의미: 처리할 일이 없음
- 보통 가장 기본적인 빈 상태 값으로 사용된다.

---

### 5-2. `Sync`

동기적으로 즉시 처리해야 하는 작업을 뜻한다.

- 의미: 가능한 한 바로 처리해야 하는 업데이트
- 가장 높은 우선순위 축에 해당하는 값으로 쓰인다.

---

### 5-3. `Batched`

배치 처리와 관련된 특수값이다.

- 여러 업데이트를 한 번에 묶어 처리하는 상황과 연결된다.
- `Sync` 바로 아래 축에서 설명되는 경우가 많다.

---

### 5-4. `Idle` / `Never`

아주 낮은 우선순위, 혹은 사실상 급하지 않은 작업을 뜻한다.

- 지금 당장 처리할 필요가 없음
- 매우 여유 있게 미뤄질 수 있음

---

### 정리

즉 React 16의 `expirationTime` 체계는 단순히 “숫자 큰 게 빠름”이 아니라,  
이런 특수 상태값들과 함께 작동하는 **스케줄링 체계 전체**로 봐야 한다.

---

## 6. 스케줄링 과정에서 어떻게 사용되나?

업데이트가 발생하면 React는 해당 Fiber에 `expirationTime`을 부여한다.  
그 후 그 Fiber에서 시작해 부모 Fiber를 따라 루트까지 올라가며  
해당 업데이트의 우선순위 정보를 전파한다.

이 과정의 목적은:

- 루트에 어떤 우선순위 작업이 존재하는지 기록하고
- 이후 스케줄러가 어떤 작업을 먼저 처리해야 하는지 판단하게 만드는 것

이다.

즉 `expirationTime`은 개별 Fiber에만 붙는 값이 아니라,  
**루트까지 전파되며 전체 렌더링 우선순위를 결정하는 기준값**이었다.

---

## 7. React 16 기준으로 정확하게 기억할 문장

복기할 때 아래 문장을 기준으로 기억하면 된다.

> React 16에서 `expirationTime`은  
> 업데이트의 단순 발생 시각이 아니라,  
> **현재 시간과 우선순위 정책을 반영해 계산된 내부 스케줄링 값**이다.

그리고 추가로:

> 일부 React 16.x 구현에서는  
> 값이 클수록 더 높은 우선순위로 비교되는 구조를 가진다.

다만:

> 이 값은 “언제 발생했는지”를 그대로 표현한 timestamp가 아니라,  
> “얼마나 빨리 처리해야 하는지”를 반영한 값이다.

---

## 8. React 16 설명에서 자주 하는 실수

### 실수 1. `expirationTime = 이벤트 발생 시간`

이건 틀렸다.

정확히는:

- 현재 시각을 기반으로 계산되지만
- 우선순위 정책이 함께 반영된 결과다.

---

### 실수 2. "값이 클수록 무조건 먼저 발생한 이벤트"

이 표현도 너무 단정적이다.

정확히는:

- 일부 구현에서는 큰 값이 더 높은 우선순위인 건 맞지만
- 그 값은 단순 발생 시각이 아니라
- **만료 시점 + 배치 정책**이 반영된 값이다.

---

### 실수 3. React 18+에서도 같은 개념이라고 생각하기

이건 중요하게 틀린 부분이다.

React 18+에서는 `expirationTime` 중심 모델로 이해하면 안 되고,  
반드시 **lane 중심 모델**로 이해해야 한다.

---

## 9. React 18+에서는 무엇이 달라졌는가?

### 9-1. 가장 큰 변화: `expirationTime` 중심 모델 → `lane` 중심 모델

React 18+에서는 업데이트 우선순위를  
예전처럼 `expirationTime` 숫자 하나로 표현하지 않는다.

대신 **lane 비트마스크 모델**을 사용한다.  
각 업데이트는 하나의 lane에 할당되고, 렌더링은 하나 이상의 lane 집합을 기준으로 진행된다.

즉 React 16이:

- 숫자 하나로 우선순위를 표현했다면

React 18+는:

- 여러 종류의 작업을 비트 단위로 나누어 관리한다.

---

### 9-2. lane이 필요한 이유

기존 `expirationTime` 모델은  
“시간 기반 우선순위 표현”에는 유용했지만,  
동시성 작업이 많아질수록 한계가 있었다.

예를 들어 React 18+에서는 다음을 더 정교하게 구분해야 한다.

- 즉시 반응해야 하는 입력
- 일반 업데이트
- transition 업데이트
- retry 작업
- idle 작업
- offscreen 작업

이런 다양한 작업을 숫자 하나로만 표현하기보다,  
**lane 집합으로 표현하는 방식이 훨씬 유연**하다.

---

### 9-3. React 18의 대표 lane 종류

React 18 소스에는 실제로 여러 lane이 정의되어 있다.

대표적으로:

- `SyncLane`
- `InputContinuousLane`
- `DefaultLane`
- `TransitionLanes`
- `RetryLanes`
- `IdleLane`
- `OffscreenLane`

즉 React 18+에서는 우선순위를 생각할 때  
`expirationTime` 숫자보다는  
**어떤 lane에 속하는 업데이트인가**로 이해하는 편이 맞다.

---

### 9-4. `startTransition`과의 연결

React 18에서는 `startTransition`을 통해  
개발자가 직접 “이 업데이트는 긴급하지 않다”는 의도를 표현할 수 있다.

예를 들어:

- 입력창 타이핑 반영은 urgent
- 검색 결과 렌더링은 transition

처럼 나눌 수 있다.  
transition으로 표시된 작업은 급한 입력이 들어오면 중단되거나 뒤로 미뤄질 수 있다.

이건 React 16과 큰 차이다.

React 16에서는 내부 숫자 우선순위를 이해해야 했다면,  
React 18에서는 개발자도 API 수준에서 작업 성격을 더 직접 표현할 수 있다.

---

### 9-5. Concurrent Rendering과의 관계

React 18의 concurrent rendering은 **interruptible** 하다.

즉 렌더링을:

- 시작했다가
- 중간에 멈추고
- 더 중요한 작업을 먼저 처리한 뒤
- 나중에 이어서 하거나
- 아예 버릴 수도 있다.

이런 환경에서는 숫자 하나짜리 `expirationTime` 모델보다  
lane 기반 모델이 훨씬 적합하다.

---

### 9-6. 그렇다고 expiration 개념이 완전히 사라진 것은 아니다

주의할 점이 있다.

React 18+에서 `expirationTime` 중심 모델은 사라졌지만,  
**lane starvation을 막기 위한 expiration 계산 개념 자체는 일부 남아 있다.**  
즉 특정 lane이 너무 오래 밀리지 않도록 내부적으로 expire 처리를 계산한다.

정리하면:

- React 16: `expirationTime`이 우선순위 모델의 중심
- React 18+: `lane`이 우선순위 모델의 중심
- 다만 React 18 내부에도 starvation 방지를 위한 expiration 계산은 존재

---

## 10. React 16과 React 18+ 비교

| 구분               | React 16                | React 18+                             |
| ------------------ | ----------------------- | ------------------------------------- |
| 핵심 우선순위 모델 | `expirationTime`        | `lane`                                |
| 표현 방식          | 숫자 하나               | 비트마스크                            |
| 중심 개념          | 만료 시점 기반 스케줄링 | 작업 종류별 lane 분리                 |
| 대표 특징          | 시간 기반 우선순위 표현 | concurrent rendering 최적화           |
| 개발자 체감 API    | 내부 개념 위주          | `startTransition` 등 명시적 제어 가능 |

## 11. 복기

### Q. React 16의 `expirationTime`은 무엇인가?

React 16의 `expirationTime`은 업데이트의 단순 발생 시각이 아니라,  
**현재 시간과 우선순위 정책을 반영해 계산한 내부 스케줄링 값**이다.  
React는 이 값을 기준으로 어떤 업데이트를 더 먼저 처리할지 결정했다.

---

### Q. `expirationTime` 값이 클수록 무조건 먼저 발생한 이벤트인가?

완전히 그렇다고 보긴 어렵다.  
일부 React 16.x 구현에서는 값이 클수록 더 높은 우선순위로 비교되지만,  
그 값은 단순 timestamp가 아니라  
**업데이트 종류와 batching 정책이 반영된 만료 시점 값**이다.

---

### Q. React 18+에서는 어떻게 바뀌었는가?

React 18+에서는 `expirationTime` 중심 모델이 아니라  
**lane 기반 우선순위 시스템**을 사용한다.  
업데이트는 `SyncLane`, `DefaultLane`, `TransitionLane` 같은 lane에 할당되며,  
동시성 렌더링 환경에서 더 정교하게 스케줄링된다.

---

## 12. 최종 결론

React 16의 `expirationTime`은  
단순한 “이벤트 발생 시간”이 아니라  
**우선순위가 반영된 내부 스케줄링 값**이다.

그리고 React 18+에서는 이 개념이 더 발전하여  
숫자 하나로 우선순위를 표현하는 방식 대신  
**lane 기반의 동시성 친화적 우선순위 모델**로 전환되었다.

# React 18+에 맞춘 형태로 다시 재구성, React는 어떤 업데이트를 먼저 처리할까?

### React 18+ 관점에서 이해하는 `expirationTime`과 `lane`

### 한 줄 요약

React 16은 업데이트 우선순위를 `expirationTime`이라는 숫자로 표현했다.  
하지만 React 18+에서는 이 방식이 `lane` 기반 모델로 바뀌었고,  
이제는 **"언제 발생했는가"보다 "어떤 성격의 작업인가"** 가 더 중요해졌다.

즉, React 18+를 공부할 때 `expirationTime`은  
**예전 React가 우선순위를 다루던 방식**으로 이해하면 되고,  
현재는 이를 **`lane`, concurrent rendering, transition** 중심으로 연결해서 이해하는 것이 맞다.

---

## 1. React 18+를 공부하는데 왜 `expirationTime`을 알아야 할까?

React 18+를 공부할 때는 사실 `expirationTime`보다 `lane`이 훨씬 중요하다.  
그럼에도 `expirationTime`을 알아두면 좋은 이유는 분명하다.

React는 오래전부터  
**"어떤 업데이트를 먼저 처리해야 하는가?"**  
라는 문제를 내부적으로 계속 다뤄 왔다.

React 16에서는 그 문제를 `expirationTime`으로 풀었고,  
React 18+에서는 더 정교한 `lane` 모델로 발전시켰다.

즉 `expirationTime`은 단순한 옛날 개념이 아니라,  
React가 업데이트 우선순위를 어떻게 설계해 왔는지 이해하게 해 주는 **배경지식**이다.

---

## 2. React 16에서는 어떤 기준으로 업데이트 우선순위를 판단했을까?

React 16에서는 업데이트 우선순위를 `expirationTime`으로 표현했다.

이 값은 단순한 이벤트 발생 시각이 아니라,  
**현재 시간과 업데이트 우선순위 정책을 반영해 계산한 내부 스케줄링 값**이다.

React는 업데이트가 발생하면 각 작업에 `expirationTime`을 부여하고,  
이 값을 기준으로:

- 어떤 작업을 먼저 처리할지
- 어떤 작업들을 같은 배치로 묶을지
- 현재 루트에 어떤 우선순위 작업이 쌓여 있는지

를 판단했다.

즉 React 16에서 `expirationTime`은  
**업데이트 우선순위를 표현하는 핵심 기준값**이었다.

---

## 3. `expirationTime`을 단순한 "이벤트 발생 시간"으로 이해하면 안 되는 이유

처음 보면 이렇게 생각하기 쉽다.

> `expirationTime`이 크면 더 먼저 발생한 이벤트다.

하지만 이건 정확한 표현은 아니다.

`expirationTime`은 단순히 "언제 발생했는가"만 담는 값이 아니라,  
**React가 그 업데이트를 얼마나 급하게 처리해야 하는지**까지 반영한 값이기 때문이다.

즉 더 정확하게는 이렇게 이해해야 한다.

> `expirationTime`은 이벤트 발생 시각 자체가 아니라,  
> **우선순위가 반영된 만료 시점 성격의 값**이다.

따라서 React 16의 `expirationTime`은  
"시간표시"라기보다  
**스케줄링을 위한 우선순위 표현값**으로 보는 것이 맞다.

---

## 4. React 16에서 `expirationTime`은 어떻게 계산되었을까?

React 16에서는 대략 다음 흐름으로 `expirationTime`이 계산되었다.

1. `now()`로 현재 시간(ms)을 구한다.
2. `getCurrentTime()`이 현재 시간을 React 내부 표현으로 변환한다.
3. `computeExpirationForFiber()`가 현재 시간과 업데이트 종류를 바탕으로 최종 `expirationTime`을 만든다.

이 과정에서 중요한 점은 다음과 같다.

- 현재 시간을 그대로 저장하지 않는다.
- React 내부 스케줄링 단위로 변환한다.
- 업데이트 종류에 따라 다른 우선순위 정책을 반영한다.

즉 같은 시점에 발생한 업데이트라도,  
업데이트 성격이 다르면 서로 다른 `expirationTime`을 가질 수 있다.

예를 들어:

- 더 즉각 반응해야 하는 업데이트
- 일반적인 비동기 업데이트

는 같은 시각에 발생해도  
서로 다른 urgency를 갖도록 계산될 수 있다.

---

## 5. React 16에서 중요한 포인트: "언제 발생했는가"보다 "얼마나 급한가"

React 16의 `expirationTime`을 복기할 때 가장 중요한 핵심은 이거다.

> `expirationTime`은  
> **업데이트가 언제 발생했는지**를 그대로 저장한 값이 아니라,  
> **얼마나 빨리 처리해야 하는지**를 반영한 값이다.

즉 React 입장에서는 단순히 "먼저 들어온 업데이트"보다  
**"더 급한 업데이트"를 먼저 처리할 수 있어야 했다.**

이 문제를 React 16은 `expirationTime`이라는 숫자로 표현했다.

---

## 6. React 16에서 함께 등장하는 특수값들

React 16의 `expirationTime` 체계에서는  
몇 가지 특수한 상태값이 함께 사용된다.

### `NoWork`

현재 처리할 작업이 없는 상태를 의미한다.

### `Sync`

가능한 한 즉시 처리해야 하는 동기 작업을 의미한다.

### `Batched`

여러 업데이트를 묶어서 처리하는 배치 처리와 관련된 값이다.

### `Idle` / `Never`

지금 당장 처리하지 않아도 되는 아주 낮은 우선순위 작업을 의미한다.

즉 React 16의 우선순위 체계는  
단순히 큰 숫자/작은 숫자가 아니라,  
이런 특수 상태값들과 함께 작동하는 **하나의 스케줄링 모델**이었다.

---

## 7. React 16에서는 이 값이 실제로 어떻게 사용되었을까?

업데이트가 발생하면 React는 해당 Fiber에 `expirationTime`을 부여한다.  
그 후 부모 방향으로 올라가며 그 우선순위 정보를 루트까지 전파한다.

이 과정을 통해 React는:

- 현재 어떤 루트에 작업이 쌓였는지
- 그중 가장 급한 작업이 무엇인지
- 다음에 무엇을 먼저 처리해야 하는지

를 판단할 수 있게 된다.

즉 `expirationTime`은 개별 Fiber에 붙는 값이면서 동시에,  
**루트 수준의 전체 렌더링 우선순위를 정하는 기준값**이기도 했다.

---

## 8. 그런데 왜 React 18+에서는 이 모델을 버렸을까?

여기서부터가 React 18+ 학습자에게 가장 중요하다.

React 16의 `expirationTime` 모델은  
업데이트 우선순위를 시간 기반 숫자로 표현하는 방식이었다.

이 방식은 단순한 우선순위 표현에는 유용했지만,  
React가 더 복잡한 동시성 작업을 다루게 되면서 한계가 드러났다.

예를 들어 React 18+에서는 다음을 더 정교하게 구분해야 한다.

- 즉시 반응해야 하는 입력
- 일반 상태 업데이트
- transition 업데이트
- retry 작업
- idle 작업
- offscreen 작업

이런 다양한 작업을  
숫자 하나로만 다루는 것보다,  
**작업 종류 자체를 분리해서 관리하는 방식**이 더 유연하다.

그래서 React 18+에서는 `expirationTime` 대신  
**`lane` 기반 모델**을 사용하게 되었다.

---

## 9. React 18+의 핵심: `lane`

React 18+에서는 업데이트 우선순위를  
`expirationTime` 숫자 하나로 표현하지 않는다.

대신 **lane 비트마스크 모델**을 사용한다.

이제 React는 업데이트를 볼 때  
"숫자가 얼마인가?"보다  
**"이 업데이트가 어떤 lane에 속하는가?"** 를 더 중요하게 본다.

예를 들어 React 18+에는 다음과 같은 종류의 lane이 있다.

- `SyncLane`
- `InputContinuousLane`
- `DefaultLane`
- `TransitionLanes`
- `RetryLanes`
- `IdleLane`
- `OffscreenLane`

즉 React 18+에서는  
우선순위를 "시간 기반 숫자"보다  
**"작업 성격별 그룹"** 으로 이해하는 편이 훨씬 정확하다.

---

## 10. React 18+에서는 왜 `lane`이 더 적합할까?

React 18의 핵심 특징은 **concurrent rendering** 이다.

이 환경에서는 렌더링이 다음처럼 동작할 수 있다.

- 시작했다가
- 중간에 멈추고
- 더 중요한 작업을 먼저 처리하고
- 나중에 다시 이어서 하거나
- 아예 버릴 수도 있다

즉 React 18+에서는  
렌더링이 더 이상 "한 번 시작하면 끝까지 가는 일"이 아니다.

이런 환경에서는  
숫자 하나로 우선순위를 표현하는 것보다  
여러 종류의 작업을 구분하고 조합할 수 있는 `lane` 모델이 훨씬 잘 맞는다.

정리하면:

- React 16: 만료 시점 중심
- React 18+: 작업 종류와 우선순위 그룹 중심

이라고 이해하면 된다.

---

## 11. React 18+에서 중요한 것은 "시간"보다 "작업의 성격"

React 16을 공부할 때는  
`expirationTime`이 어떤 값인지, 어떻게 계산되는지가 중요했다.

React 18+에서는 그보다 더 중요한 질문이 바뀐다.

이제는 이렇게 생각하는 편이 맞다.

- 이 업데이트는 즉시 반응해야 하는가?
- 이 업데이트는 transition인가?
- 이 작업은 느려도 괜찮은가?
- 이 렌더는 중간에 멈출 수 있는가?

즉 React 18+에서는  
**"언제 발생했는가"보다 "어떤 종류의 작업인가"** 가 핵심이다.

---

## 12. `startTransition`과 연결해서 이해하기

React 18에서는 `startTransition`을 통해  
개발자가 직접 "이 업데이트는 긴급하지 않다"는 의도를 표현할 수 있다.

예를 들어:

- 입력창에 타이핑한 값을 즉시 보여주는 업데이트는 urgent
- 검색 결과 목록을 다시 렌더링하는 작업은 transition

처럼 나눌 수 있다.

이건 React 16과 매우 큰 차이다.

React 16에서는 내부 숫자 우선순위를 이해해야 했다면,  
React 18에서는 개발자도 API 수준에서  
**작업의 성격을 직접 구분할 수 있게 되었다.**

즉 React 18+를 공부할 때는  
`expirationTime` 자체보다  
**`lane`과 `startTransition`을 함께 이해하는 것이 훨씬 중요하다.**

---

## 13. 그렇다고 React 18+에서 expiration 개념이 완전히 사라진 것은 아니다

여기서 한 가지 주의할 점이 있다.

React 18+는 `expirationTime` 중심 모델을 버렸지만,  
**오래 밀린 lane이 계속 굶주리지 않도록 관리하는 expiration 성격의 내부 계산은 일부 남아 있다.**

즉:

- React 16처럼 `expirationTime` 자체가 우선순위 모델의 중심은 아니지만
- React 18 내부에도 starvation 방지를 위한 expire 개념은 존재한다

이 점을 이해하면  
React 16과 React 18+가 완전히 단절된 것이 아니라,  
**문제를 푸는 방식이 더 정교하게 진화한 것**이라는 흐름이 보인다.

---

## 14. React 16과 React 18+를 어떻게 비교하면 좋을까?

| 구분               | React 16                    | React 18+                             |
| ------------------ | --------------------------- | ------------------------------------- |
| 핵심 우선순위 모델 | `expirationTime`            | `lane`                                |
| 표현 방식          | 숫자 하나                   | 비트마스크                            |
| 중심 관점          | 만료 시점 기반 스케줄링     | 작업 종류별 우선순위 그룹             |
| 핵심 질문          | 얼마나 빨리 처리해야 하는가 | 어떤 종류의 작업인가                  |
| 대표 특징          | 시간 기반 우선순위          | concurrent rendering 최적화           |
| 개발자 체감 API    | 내부 개념 위주              | `startTransition` 등 명시적 제어 가능 |

즉 React 16은  
**시간 기반으로 urgency를 표현한 모델**이고,

React 18+는  
**작업 성격을 중심으로 우선순위를 표현하는 모델**이라고 보면 된다.

---

## 15. React 18+ 학습자 관점에서 기억해야 할 핵심

React 18+를 공부하는 입장에서는 아래처럼 정리하면 가장 좋다.

### 기억할 것 1

`expirationTime`은 예전 React가 업데이트 우선순위를 표현하던 방식이다.

### 기억할 것 2

React 18+에서는 `expirationTime`이 아니라 `lane`이 우선순위 모델의 중심이다.

### 기억할 것 3

이제 중요한 것은 "이벤트가 언제 발생했는가"보다  
"이 업데이트가 urgent인지, transition인지, idle인지"다.

### 기억할 것 4

React 18+의 핵심은 `lane`, concurrent rendering, automatic batching, `startTransition`이다.

즉 React 18+ 학습자에게 `expirationTime`은  
현재 모델의 중심 개념이 아니라,  
**현재 구조를 이해하기 위한 배경지식**이다.

## 16. 복기

### Q. React 16의 `expirationTime`은 무엇인가?

React 16의 `expirationTime`은 업데이트의 단순 발생 시각이 아니라,  
현재 시간과 우선순위 정책을 반영해 계산한 내부 스케줄링 값이다.  
React는 이 값을 기준으로 어떤 업데이트를 더 먼저 처리할지 판단했다.

---

### Q. React 18+에서는 왜 더 이상 `expirationTime` 중심으로 이해하면 안 되는가?

React 18+는 concurrent rendering 환경에서  
더 다양한 종류의 작업을 정교하게 구분해야 하기 때문에,  
숫자 하나로 우선순위를 표현하는 방식보다  
`lane` 기반 모델이 더 적합하다.

---

### Q. React 18+에서 더 중요한 개념은 무엇인가?

React 18+에서는 `expirationTime`보다  
`lane`, concurrent rendering, automatic batching, `startTransition`을 더 중심적으로 이해해야 한다.

---

## 17. 최종 결론

React 16의 `expirationTime`은  
단순한 이벤트 발생 시간이 아니라  
**업데이트의 urgency를 표현한 내부 스케줄링 값**이었다.

하지만 React 18+에서는 이 방식이  
더 정교한 `lane` 기반 모델로 바뀌었다.

따라서 React 18+를 공부하는 사람은  
`expirationTime`을 현재 중심 개념으로 붙잡기보다,

- 예전 React는 `expirationTime`으로 우선순위를 표현했고
- 현재 React는 `lane`으로 더 정교하게 표현한다

는 흐름으로 이해하는 것이 가장 정확하다.
