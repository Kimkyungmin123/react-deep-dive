# React Fiber — `scheduleUpdateOnFiber`의 동기/비동기 작업 분기 처리

> ⚠️ **용어 정정**: 원문에서 `scheduleUpdateOnes`, `scheduleWork`, `expressionTime`, `isScheduled`, `ensureIsScheduled`, `connect time` 등 여러 오기가 등장한다. 아래 표로 정리한다.

| 원문 오기               | 올바른 표현                                                                 |
| ----------------------- | --------------------------------------------------------------------------- |
| `scheduleUpdateOnes`    | `scheduleUpdateOnFiber`                                                     |
| `scheduleWork`          | `scheduleUpdateOnFiber` (React 16 내부에서 `scheduleWork`는 이 함수의 별칭) |
| `expressionTime`        | `expirationTime`                                                            |
| `isScheduled`           | `ensureRootIsScheduled`                                                     |
| `ensureIsScheduled`     | `ensureRootIsScheduled`                                                     |
| `connect time`          | `currentTime`                                                               |
| `insureRootIsScheduled` | `ensureRootIsScheduled`                                                     |

---

## 1. `scheduleUpdateOnFiber`의 역할과 설계 원칙

### 1.1 Reconciler ↔ Scheduler 패키지 간 의존성 분리

React는 `react-reconciler`와 `scheduler` 를 **별도의 독립 패키지**로 분리하여 관리한다.  
두 패키지가 서로의 코드를 직접 호출하지 않도록 설계된 것이 핵심이다.

```
react-reconciler 패키지
    └─ scheduleUpdateOnFiber()
            │
            └─ ensureRootIsScheduled(root)
                    │
                    └─ scheduleCallback(priority, callback)  ← scheduler 패키지 함수
```

`scheduleUpdateOnFiber`는 이 흐름의 **시작점**이다.  
Reconciler가 Scheduler에 작업을 넘기기 전에 **동기/비동기 분기, 컨텍스트 확인, 루트 스케줄링** 등의 판단을 수행한다.

**독립 패키지 설계의 장점:**

- 각 패키지가 독립적으로 테스트·재사용 가능하다.
- Reconciler 없이 Scheduler만 단독으로 사용할 수 있다.
- 패키지 간 결합도(coupling)가 낮아 유지보수가 쉽다.

---

## 2. `scheduleUpdateOnFiber` 내부 분기 구조

```
scheduleUpdateOnFiber(fiber, expirationTime)
        │
        ├─ markUpdateTimeFromFiberToRoot(fiber, expirationTime)
        │   → Fiber ~ 루트까지 expirationTime 마킹
        │   → FiberRootNode 반환
        │
        ├─ [분기 1] expirationTime === Sync?
        │     YES → 동기 처리 경로 (§3)
        │     NO  → 비동기 처리 경로 (§4)
        │
        └─ (양쪽 모두 최종적으로 ensureRootIsScheduled 호출)
```

---

## 3. 동기 처리 경로 (`expirationTime === Sync`)

`expirationTime`이 `Sync`(React 내부 상수, 최댓값 `1073741823`)이면 해당 Work는 **즉시 동기적으로 처리**해야 한다.

### 3.1 동기 경로의 두 가지 하위 분기

```
expirationTime === Sync
        │
        ├─ [A] executionContext가 NoContext (렌더·커밋 컨텍스트 모두 없음)
        │         + VDOM이 아직 생성되지 않은 경우 (unbatchedContext)
        │         → performSyncWorkOnRoot(root) 즉시 호출
        │           (초기 마운트 전용 경로)
        │
        └─ [B] 그 외 (이미 VDOM이 존재하고 업데이트 발생)
                  → ensureRootIsScheduled(root)
                  + executionContext === NoContext이면
                    flushSyncCallbackQueue() 호출
```

### 3.2 분기 A — 초기 마운트 (`performSyncWorkOnRoot` 즉시 호출)

```javascript
// executionContext가 NoContext이고, 아직 아무 렌더링도 없을 때
if (
  (executionContext & LegacyUnbatchedContext) !== NoContext &&
  (executionContext & (RenderContext | CommitContext)) === NoContext
) {
  performSyncWorkOnRoot(root); // VDOM을 즉시 생성
  return;
}
```

- `ReactDOM.render()`가 처음 호출될 때 진입하는 경로이다.
- 스케줄러를 거치지 않고 **즉시 렌더링**을 수행한다.
- 주석에는 `"The initial mount of React DOM render"` 라고 명시되어 있다.

### 3.3 분기 B — 동기 업데이트 스케줄링 후 flush

```javascript
ensureRootIsScheduled(root); // syncQueue에 Work 등록

if (executionContext === NoContext) {
  // Reconciler가 현재 다른 작업을 하고 있지 않으면
  // 즉시 syncQueue를 비워 동기 작업을 처리
  flushSyncCallbackQueue();
}
```

- **`flushSyncCallbackQueue`**: `scheduleSyncCallback`으로 쌓인 `syncQueue`를 즉시 실행한다.
- `executionContext === NoContext` 조건의 의미: 현재 렌더 페이즈나 커밋 페이즈가 진행 중이 아닐 때만 flush한다. 이미 다른 Work가 진행 중이라면 그 Work가 끝난 후 flush된다.

> ⚠️ **흔한 오해**: `flushSyncCallbackQueue`는 Reconciler가 **쉬고 있을 때**(NoContext)만 호출된다.  
> 이미 렌더링 중이라면 호출하지 않는다. 중첩 호출로 인한 상태 오염을 방지하기 위해서이다.

---

## 4. 비동기 처리 경로 (`expirationTime !== Sync`)

```javascript
// Concurrent Work → 스케줄러에 위임
ensureRootIsScheduled(root);
```

비동기 경로는 단순하다. `ensureRootIsScheduled`를 호출하여 Scheduler가 적절한 시점에 Work를 실행하도록 위임한다.  
동기 경로처럼 즉시 실행(`performSyncWorkOnRoot`)하거나 즉시 flush(`flushSyncCallbackQueue`)하지 않는다.

---

## 5. `executionContext`란?

동기 분기 로직에서 핵심 조건으로 등장하는 `executionContext`는 **Reconciler의 현재 실행 상태**를 나타내는 비트마스크이다.

| 플래그                   | 의미                                     |
| ------------------------ | ---------------------------------------- |
| `NoContext` (0)          | 아무 작업도 진행 중이지 않음 (Idle 상태) |
| `BatchedContext`         | `batchedUpdates` 내부                    |
| `LegacyUnbatchedContext` | 초기 마운트 (`ReactDOM.render` 첫 호출)  |
| `RenderContext`          | 렌더 페이즈 진행 중                      |
| `CommitContext`          | 커밋 페이즈 진행 중                      |

```javascript
// 현재 렌더 또는 커밋 중인지 확인
const isAlreadyRendering =
  (executionContext & (RenderContext | CommitContext)) !== NoContext;
```

## 6. 전체 흐름 요약

```
setState 호출
        │
        ▼
scheduleUpdateOnFiber(fiber, expirationTime)
        │
        ├─ markUpdateTimeFromFiberToRoot  →  expirationTime 마킹 + 루트 반환
        │
        ├─ expirationTime === Sync?
        │     │
        │     ├─ YES + LegacyUnbatchedContext
        │     │         → performSyncWorkOnRoot() 즉시 실행 (초기 마운트)
        │     │
        │     └─ YES + 그 외
        │               → ensureRootIsScheduled(root)
        │               → executionContext === NoContext?
        │                     YES → flushSyncCallbackQueue() (즉시 flush)
        │                     NO  → 현재 진행 중인 Work 종료 후 flush
        │
        └─ NO (비동기)
                  → ensureRootIsScheduled(root)
                     (Scheduler가 여유 시간에 처리)
```

## 7. React 18+과의 대응 개념

| React 16/17                               | React 18+                           | 변화 요약                                 |
| ----------------------------------------- | ----------------------------------- | ----------------------------------------- |
| `scheduleUpdateOnFiber`                   | 동일하게 유지                       | 함수명 동일, 내부 로직 변경               |
| `expirationTime === Sync` 조건            | `includesSyncLane(lane)` 비트 연산  | Lane 기반 Sync 판단으로 변경              |
| `LegacyUnbatchedContext` 초기 마운트 경로 | `createRoot` 사용 시 해당 경로 없음 | `createRoot`는 항상 Concurrent 경로 사용  |
| `performSyncWorkOnRoot` 즉시 호출         | Legacy Mode에서만 유지              | `createRoot`에서는 Concurrent 경로로 처리 |
| `flushSyncCallbackQueue`                  | `flushSyncCallbacks`                | 내부 구현 개선, 역할 동일                 |
| `executionContext` 비트마스크             | 동일하게 유지                       | 일부 플래그 추가                          |

📌 **핵심 변화**: React 18에서 `ReactDOM.createRoot()`를 사용하면 **초기 마운트도 Concurrent 경로**로 처리된다.  
`LegacyUnbatchedContext`를 이용한 즉시 동기 마운트 경로는 `ReactDOM.render()`(Legacy Mode)에서만 유지된다.

# React 18+ — `createRoot`와 동기/비동기 Work 분기 처리

## 1. React 17 vs React 18: 초기 마운트 경로의 차이

### 1.1 `ReactDOM.render()` — Legacy Mode (React 17 이하)

```jsx
// Legacy Mode
ReactDOM.render(<App />, document.getElementById("root"));
```

- 초기 마운트 시 `scheduleUpdateOnFiber` 내부에서 **즉시 동기 렌더링**(`performSyncWorkOnRoot`)을 수행한다.
- Scheduler를 거치지 않고 바로 렌더링이 시작된다.
- Concurrent 기능(중단·재개, `startTransition` 등)을 사용할 수 없다.

### 1.2 `ReactDOM.createRoot()` — Concurrent Mode (React 18)

```jsx
// Concurrent Mode
const root = ReactDOM.createRoot(document.getElementById("root"));
root.render(<App />);
```

- 초기 마운트도 **Concurrent 경로**로 처리된다.
- 즉시 실행(`performSyncWorkOnRoot`) 경로를 거치지 않는다.
- `startTransition`, `useDeferredValue`, Automatic Batching 등 모든 Concurrent 기능이 활성화된다.

| 항목               | `ReactDOM.render()` | `ReactDOM.createRoot()` |
| ------------------ | ------------------- | ----------------------- |
| 초기 마운트 경로   | 즉시 동기 실행      | Concurrent 경로         |
| Concurrent 기능    | ❌                  | ✅                      |
| `startTransition`  | ❌                  | ✅                      |
| Automatic Batching | 부분적              | ✅ 완전 지원            |
| React 18 지원 여부 | 유지(하위 호환)     | 권장 방식               |

---

## 2. React 18의 Work 분기 구조

`scheduleUpdateOnFiber`는 React 18에서도 동일하게 존재하지만, `expirationTime` 대신 **Lane**으로 분기를 판단한다.

```
scheduleUpdateOnFiber(fiber, lane, eventTime)
        │
        ├─ markUpdateLaneFromFiberToRoot  →  Lane 마킹 + 루트 반환
        │
        ├─ SyncLane 포함?
        │     YES → 동기 처리 경로
        │     NO  → Concurrent 처리 경로
        │
        └─ ensureRootIsScheduled(root)
```

### 2.1 어떤 업데이트가 SyncLane에 배정되나?

```jsx
// SyncLane → 즉시 동기 처리
flushSync(() => setState(value));

// DefaultLane → Concurrent 경로 (일반 setState)
setState(value);

// TransitionLane → Concurrent 경로 (낮은 우선순위)
startTransition(() => setState(value));
```

> ⚠️ React 18에서 `createRoot`를 사용하면 일반 `setState`도 기본적으로 **Concurrent 경로**를 거친다.  
> React 16/17처럼 무조건 동기 처리되지 않는다. 단, SyncLane은 여전히 동기적으로 처리된다.

---

## 3. `executionContext` — Reconciler의 현재 실행 상태

`executionContext`는 React 18에서도 동일하게 사용되는 **비트마스크 상태 변수**이다.

```javascript
// 현재 렌더 또는 커밋 중인지 확인하는 데 사용
const isAlreadyRendering =
  (executionContext & (RenderContext | CommitContext)) !== NoContext;
```

### 3.1 `flushSync`와 `executionContext`

```jsx
import { flushSync } from "react-dom";

function handleClick() {
  flushSync(() => {
    setCount((c) => c + 1); // SyncLane으로 강제 처리
  });
  // 이 시점에 DOM이 이미 업데이트되어 있음
}
```

`flushSync`는 내부적으로:

1. `executionContext`에 `BatchedContext` 플래그를 추가한다.
2. 콜백 내부 `setState`를 `SyncLane`에 배정한다.
3. 콜백 종료 후 즉시 `flushSyncCallbacks()`를 호출한다.
4. `executionContext`를 원래대로 복구한다.

> ⚠️ **`flushSync` 주의사항**: `flushSync` 내부에서 `startTransition`을 사용하면 무시된다.  
> `flushSync`는 SyncLane을 강제하므로 Transition 우선순위가 덮어씌워진다.

---

## 4. Automatic Batching — Concurrent 경로 덕분에 가능해진 것

React 18의 Automatic Batching은 대부분의 업데이트가 **Concurrent 경로**를 거치기 때문에 가능하다.

```jsx
// React 17 이하 (Legacy Mode): setTimeout 내부는 배칭 안 됨
setTimeout(() => {
  setA(1); // 렌더링 1회
  setB(2); // 렌더링 1회 → 총 2회
}, 1000);

// React 18 (createRoot): 자동 배칭 → 렌더링 1회
setTimeout(() => {
  setA(1);
  setB(2);
  // 두 setState 모두 DefaultLane으로 배정
  // ensureRootIsScheduled 2회 호출 → 2번째 호출 시 같은 우선순위 → 재사용
  // → 렌더링 1회
}, 1000);
```

**배칭이 일어나는 원리:**

```
setA(1) → scheduleUpdateOnFiber → ensureRootIsScheduled
              DefaultLane 스케줄링 → callbackNode 저장

setB(2) → scheduleUpdateOnFiber → ensureRootIsScheduled
              DefaultLane 스케줄링 시도
              → 이미 동일 우선순위 callbackNode 존재 → 재사용 (return)
              → 추가 렌더링 없음
```

### 4.1 배칭을 해제해야 하는 경우: `flushSync`

```jsx
function handleClick() {
  flushSync(() => {
    setCount((c) => c + 1);
  });
  // 이 줄 실행 전에 DOM이 이미 업데이트됨
  console.log(ref.current.textContent); // 최신 값
}
```

## 5. `scheduleUpdateOnFiber` 진입 시나리오별 정리

| 시나리오                             | Lane                            | 경로       | 즉시 실행 여부    |
| ------------------------------------ | ------------------------------- | ---------- | ----------------- |
| `createRoot` 초기 마운트             | `DefaultLane`                   | Concurrent | ❌ Scheduler 위임 |
| 일반 `setState`                      | `DefaultLane`                   | Concurrent | ❌ Scheduler 위임 |
| `startTransition` 내부 `setState`    | `TransitionLane`                | Concurrent | ❌ Scheduler 위임 |
| `flushSync` 내부 `setState`          | `SyncLane`                      | Sync       | ✅ 즉시 flush     |
| Legacy `ReactDOM.render` 초기 마운트 | `SyncLane` + `UnbatchedContext` | Sync       | ✅ 즉시 실행      |

## 7. 핵심 정리 체크리스트

- [ ] `createRoot`를 사용하면 초기 마운트도 Concurrent 경로로 처리됨을 이해했다.
- [ ] 일반 `setState`는 `DefaultLane`으로 배정되어 Scheduler에 위임됨을 이해했다.
- [ ] `flushSync`는 `SyncLane`을 강제하여 즉시 동기 flush가 일어남을 이해했다.
- [ ] Automatic Batching은 `ensureRootIsScheduled`에서 같은 Lane 우선순위를 재사용하기 때문에 가능함을 이해했다.
- [ ] `executionContext`가 NoContext일 때만 즉시 flush가 일어나고, 렌더/커밋 중에는 건너뜀을 이해했다.
- [ ] `ReactDOM.render()`(Legacy)와 `createRoot`(Concurrent)의 초기 마운트 경로 차이를 이해했다.
