# 리액트 입장에서는 어떤 이벤트를 먼저 처리해야할까? (expirationTime 값의 의미)

### `scheduleWork` / `scheduleUpdateOnFiber` / `ensureRootIsScheduled`

### 한 줄 요약

React 16에서 업데이트 스케줄링의 핵심 흐름은  
**업데이트 발생 → `scheduleUpdateOnFiber` 진입 → 루트까지 우선순위 전파 → `ensureRootIsScheduled`로 스케줄 결정 → 동기/비동기 방식에 따라 실행**  
으로 이해할 수 있다.

즉, `scheduleUpdateOnFiber`는 업데이트 스케줄링을 시작하는 진입점이고,  
`ensureRootIsScheduled`는 실제로 루트에 어떤 작업을 어떤 우선순위로 예약할지 결정하는 핵심 함수다.

---

## 1. 전체 흐름 먼저 보기

React 16에서 어떤 컴포넌트에 업데이트가 발생하면, 내부적으로는 대략 다음 순서로 흐른다.

1. 업데이트 발생
2. `scheduleUpdateOnFiber` 호출
3. 현재 Fiber에서 루트까지 올라가며 `expirationTime` 전파
4. 루트(root) 획득
5. `ensureRootIsScheduled` 호출
6. 우선순위에 따라 동기 또는 비동기 작업으로 예약
7. 적절한 시점에 렌더링 작업 수행

이 흐름을 기준으로 각 함수의 역할을 이해하면 전체 구조가 깔끔하게 정리된다.

---

## 2. `scheduleUpdateOnFiber`의 역할

### 2-1. 업데이트 스케줄링의 시작점

`scheduleUpdateOnFiber`는 업데이트가 발생했을 때 가장 먼저 진입하는 핵심 함수 중 하나다.

이 함수의 역할은 단순히 “작업을 등록한다”가 아니라,  
**어느 루트에 어떤 우선순위의 업데이트가 생겼는지 정리하고, 그 루트가 적절히 스케줄되도록 만드는 것**이다.

즉 React 16에서 이 함수는 업데이트 스케줄링의 **입구 역할**을 한다고 볼 수 있다.

---

### 2-2. 이 함수가 하는 핵심 일

`scheduleUpdateOnFiber`는 크게 두 가지를 수행한다.

#### 1) 업데이트 우선순위를 루트까지 전파

업데이트가 발생한 Fiber에서 시작해서 부모 방향으로 올라가며  
해당 업데이트의 `expirationTime` 정보를 반영한다.

이 과정을 통해 React는:

- 어떤 루트에 업데이트가 생겼는지
- 그 업데이트가 얼마나 급한지
- 루트 입장에서 지금 처리해야 할 가장 높은 우선순위가 무엇인지

를 알 수 있게 된다.

#### 2) 루트 스케줄링 요청

루트를 찾은 뒤에는 `ensureRootIsScheduled`를 호출해  
실제로 이 루트를 어떤 방식으로 실행할지 결정하게 만든다.

즉:

- `scheduleUpdateOnFiber`는 **업데이트를 루트까지 연결하는 함수**
- `ensureRootIsScheduled`는 **그 루트를 실제로 예약하는 함수**

라고 이해하면 된다.

---

## 3. `expirationTime`과의 관계

React 16에서는 업데이트 우선순위를 `lane`이 아니라 **`expirationTime`** 으로 표현한다.

`expirationTime`은 단순한 이벤트 발생 시각이 아니라,  
**현재 시간과 업데이트 우선순위 정책을 반영해 계산된 내부 스케줄링 값**이다.

`scheduleUpdateOnFiber`는 이 `expirationTime`을 현재 Fiber와 상위 경로에 반영하면서  
루트가 어떤 우선순위의 작업을 갖고 있는지 갱신한다.

즉 이 함수는 단순한 연결 함수가 아니라,  
**업데이트 우선순위 정보를 Fiber 트리 전체에 반영하는 역할**까지 함께 수행한다.

---

## 4. `ensureRootIsScheduled`의 역할

### 4-1. 실제 스케줄 결정 함수

`ensureRootIsScheduled`는 이름 그대로  
**현재 루트가 적절하게 스케줄되어 있는지 확인하고, 필요하면 새로 예약하는 함수**다.

이 함수는 루트의 현재 상태를 보고:

- 다음에 처리해야 할 가장 높은 우선순위의 작업이 무엇인지
- 기존에 예약된 callback이 그대로 유효한지
- 새 callback을 등록해야 하는지
- 동기 작업으로 처리해야 하는지, 비동기 작업으로 처리해야 하는지

를 결정한다.

즉 React 16 스케줄링 흐름에서  
실질적인 “예약 판단”의 중심은 `ensureRootIsScheduled`에 있다.

---

### 4-2. 왜 중요한가?

`scheduleUpdateOnFiber`가 업데이트를 루트까지 연결해 줬다면,  
`ensureRootIsScheduled`는 그 다음 단계인  
**“그래서 이 루트를 지금 어떻게 실행할 것인가?”** 를 결정한다.

따라서 React 16 스케줄링을 공부할 때는  
`update 발생 → scheduleUpdateOnFiber → ensureRootIsScheduled`  
이 연결을 하나의 세트로 이해하는 것이 중요하다.

---

## 5. 동기 작업 처리 방식

### 5-1. 동기 작업이란?

React 16에서는 `expirationTime === Sync`인 경우를  
가장 높은 우선순위의 동기 작업 경로로 이해할 수 있다.

이런 작업은 가능한 한 빠르게 처리되어야 하며,  
상황에 따라 즉시 실행되거나, sync callback queue를 통해 곧바로 flush된다.

---

### 5-2. 동기 작업의 처리 흐름

동기 업데이트가 들어오면 루트는 먼저 `ensureRootIsScheduled`를 통해  
동기 작업으로 예약된다.

이후 실행 문맥에 따라 두 가지 방식으로 이어질 수 있다.

#### 1) 즉시 실행 경로

특정 실행 문맥에서는 동기 작업이 바로 실행된다.  
이 경우 루트에서 즉시 동기 렌더링이 수행된다.

#### 2) sync callback queue를 통한 실행

항상 즉시 실행되는 것은 아니다.  
경우에 따라 sync callback queue에 작업이 들어간 뒤,  
적절한 시점에 `flushSyncCallbackQueue()`를 통해 실행된다.

즉, React 16의 sync update는  
**“무조건 즉시 실행”이 아니라, 실행 문맥에 따라 바로 실행되거나 큐를 통해 flush되는 구조**다.

---

### 5-3. `performSyncWorkOnRoot`

동기 작업의 실제 실행 진입점으로 이해하면 되는 함수가  
`performSyncWorkOnRoot`다.

이 함수는 루트에서 동기 렌더링 작업을 수행하는 entry point 역할을 하며,  
특히 동기 경로에서는 Scheduler를 길게 거치지 않고  
바로 실행 단계로 들어가는 흐름이 존재한다.

정리하면:

- `scheduleUpdateOnFiber`: 스케줄링 시작
- `ensureRootIsScheduled`: 루트 예약 결정
- `performSyncWorkOnRoot`: 동기 렌더링 실제 실행

으로 연결된다.

---

## 6. 비동기 작업 처리 방식

### 6-1. 비동기 작업의 기본 방향

동기 작업이 아닌 경우, React는 해당 루트를  
비동기적으로 처리할 수 있는 작업으로 예약한다.

이때도 시작은 동일하다.

1. 업데이트 발생
2. `scheduleUpdateOnFiber`
3. 루트까지 우선순위 전파
4. `ensureRootIsScheduled`

차이는 마지막 단계에서  
**동기 즉시 처리 대신, Scheduler가 나중에 실행할 callback으로 등록된다**는 점이다.

---

### 6-2. 비동기 작업에서 `ensureRootIsScheduled`

비동기 작업일 때 `ensureRootIsScheduled`는  
루트의 현재 `expirationTime` 상태를 보고,  
해당 우선순위에 맞는 callback을 등록한다.

즉 이 함수는 비동기 경로에서:

- 어떤 우선순위로 실행할지
- 기존 callback을 재사용할지
- 새 callback을 등록할지

를 판단하는 역할을 한다.

따라서 비동기 작업 흐름을 이해할 때도  
핵심 함수는 여전히 `ensureRootIsScheduled`다.

---

## 7. React 16 기준 핵심 정리

React 16의 업데이트 스케줄링은 다음처럼 기억하면 가장 깔끔하다.

### 핵심 구조

- 업데이트가 발생하면 `scheduleUpdateOnFiber`가 시작점이 된다.
- 이 함수는 현재 Fiber에서 루트까지 `expirationTime`을 전파한다.
- 루트를 찾은 뒤 `ensureRootIsScheduled`를 호출한다.
- `ensureRootIsScheduled`는 루트에 어떤 작업을 어떤 우선순위로 예약할지 결정한다.
- 동기 작업은 바로 실행되거나 sync queue를 통해 flush된다.
- 비동기 작업은 Scheduler callback으로 등록되어 나중에 실행된다.

---

### 핵심 문장

> React 16에서 `scheduleUpdateOnFiber`는 업데이트 스케줄링의 진입점이고,  
> `ensureRootIsScheduled`는 루트에 대해 실제로 어떤 작업을 예약할지 결정하는 핵심 함수다.

---

## 8. React 16에서 이 구조를 왜 이렇게 나눴는가

이 구조를 보면 역할이 명확히 나뉘어 있다.

- **업데이트 정보 수집과 루트 탐색**: `scheduleUpdateOnFiber`
- **실제 예약 판단**: `ensureRootIsScheduled`
- **동기 실행 진입점**: `performSyncWorkOnRoot`

이렇게 분리하면 React 내부에서:

- 업데이트 전파
- 우선순위 판단
- 실제 실행

을 각각 독립적으로 관리할 수 있다.

즉, Reconciler 내부의 스케줄링 흐름이  
하나의 거대한 함수가 아니라 **단계별 책임 분리 구조**로 되어 있다는 점이 중요하다.

---

## 9. React 18+에서는 무엇이 달라졌는가?

React 18+에서도 큰 흐름 자체는 비슷하다.

- 업데이트 발생
- `scheduleUpdateOnFiber`
- 루트 갱신
- `ensureRootIsScheduled`
- 작업 예약 및 실행

하지만 **우선순위를 표현하는 방식과 스케줄링 모델이 크게 바뀌었다.**

---

### 9-1. `expirationTime` → `lane`

React 16에서는 업데이트 우선순위를 `expirationTime` 숫자로 표현했다.

React 18+에서는 이 방식이 `lane` 기반 모델로 바뀌었다.

즉 이제는:

- React 16: 숫자 하나로 우선순위 표현
- React 18+: 비트마스크 lane으로 우선순위 표현

으로 이해해야 한다.

`lane` 모델은 여러 종류의 작업을 더 정교하게 구분하고  
동시에 관리하기 위해 도입되었다.

---

### 9-2. 왜 `lane` 모델이 필요했는가?

React 18+는 concurrent rendering을 지원한다.  
이 환경에서는 단순히 “이 작업이 더 급하다” 정도만으로는 부족하고,

- 입력 반응처럼 아주 급한 작업
- 일반적인 상태 업데이트
- transition 작업
- retry 작업
- idle 작업

등을 더 세밀하게 구분할 필요가 있다.

`lane`은 이런 다양한 작업 성격을 더 유연하게 표현할 수 있는 구조다.

---

### 9-3. `scheduleUpdateOnFiber`의 의미 변화

React 18+에서도 `scheduleUpdateOnFiber`는 여전히  
업데이트 스케줄링의 진입점이다.

다만 React 16에서는 `expirationTime` 중심으로 이해했다면,  
React 18+에서는 **`lane` 기반 concurrent scheduling의 시작점**으로 이해해야 한다.

즉 함수 이름은 비슷하지만, 내부에서 다루는 우선순위 모델은 크게 달라졌다.

---

### 9-4. `ensureRootIsScheduled`도 더 정교해졌다

React 18+의 `ensureRootIsScheduled`는 단순히  
“새 callback 등록” 정도를 넘어서,

- 현재 root에 남아 있는 lane들 확인
- starvation 방지
- 기존 callback 유지 여부 판단
- priority 재조정
- concurrent 환경에 맞는 task 재등록

같은 더 복잡한 판단을 수행한다.

즉 React 16에서도 중요했지만,  
React 18+에서는 concurrent scheduling의 중심 함수로서  
역할이 더 확장되었다고 볼 수 있다.

---

### 9-5. sync / async 구분도 더 정교해졌다

React 16에서는 비교적 단순하게  
`expirationTime === Sync` 같은 기준으로 sync 경로를 이해할 수 있었다.

React 18+에서는 단순한 sync/async 구분보다  
**어떤 lane인가**,  
**urgent update인가**,  
**transition인가**,  
**concurrent하게 중단 가능한 작업인가**  
가 더 중요해졌다.

즉 React 18+에서는 “동기냐 비동기냐”보다  
**작업의 성격과 우선순위 그룹**을 기준으로 이해하는 것이 더 적절하다.

---

## 10. React 16 vs React 18+ 비교

| 구분            | React 16                 | React 18+                        |
| --------------- | ------------------------ | -------------------------------- |
| 우선순위 모델   | `expirationTime`         | `lane`                           |
| 스케줄링 진입점 | `scheduleUpdateOnFiber`  | `scheduleUpdateOnFiber`          |
| 실제 예약 판단  | `ensureRootIsScheduled`  | `ensureRootIsScheduled`          |
| 동기 처리 개념  | `Sync` 중심              | `SyncLane` 및 lane priority 중심 |
| 핵심 특징       | expiration 기반 우선순위 | concurrent/lane 기반 우선순위    |

## 11. 복기

### Q. React 16에서 `scheduleUpdateOnFiber`는 어떤 역할을 하는가?

React 16에서 `scheduleUpdateOnFiber`는 업데이트 스케줄링의 진입점이다.  
업데이트가 발생한 Fiber에서 루트까지 `expirationTime`을 전파하고,  
그 루트가 적절히 실행되도록 `ensureRootIsScheduled`를 호출한다.

---

### Q. `ensureRootIsScheduled`는 어떤 역할을 하는가?

`ensureRootIsScheduled`는 현재 루트에 대해  
어떤 우선순위의 작업을 어떤 방식으로 예약할지 결정하는 핵심 함수다.  
동기 작업이면 즉시 실행 또는 sync queue flush 경로로 이어지고,  
비동기 작업이면 Scheduler callback 등록으로 이어진다.

---

### Q. React 18+에서는 무엇이 달라졌는가?

React 18+에서는 `expirationTime` 대신 `lane` 기반 우선순위 모델을 사용한다.  
`scheduleUpdateOnFiber`와 `ensureRootIsScheduled`라는 큰 흐름은 유지되지만,  
이제는 concurrent rendering을 전제로  
작업 종류별 우선순위를 더 정교하게 다룬다.

---

## 12. 최종 결론

React 16의 업데이트 스케줄링은  
`scheduleUpdateOnFiber`와 `ensureRootIsScheduled`를 중심으로 이해하면 가장 깔끔하다.

- `scheduleUpdateOnFiber`는 업데이트를 루트까지 연결하고
- `ensureRootIsScheduled`는 그 루트를 실제로 예약하며
- 동기 작업은 즉시 실행 또는 sync queue를 통해 flush되고
- 비동기 작업은 Scheduler를 통해 나중에 실행된다

그리고 React 18+에서는 이 구조가 유지되면서도  
우선순위 표현 방식이 `expirationTime`에서 `lane`으로 바뀌어  
더 정교한 concurrent scheduling 모델로 확장되었다.

# React 18+에 맞춘 형태로 다시 재구성, React 18+ 업데이트 스케줄링 정리

### `scheduleUpdateOnFiber`, `ensureRootIsScheduled`, 그리고 lane 모델

### 한 줄 요약

React 18+에서 업데이트 스케줄링의 핵심은  
**업데이트 발생 → `scheduleUpdateOnFiber` 진입 → root에 lane 기록 → `ensureRootIsScheduled`로 다음 작업 결정 → concurrent하게 실행**  
으로 이해할 수 있다.

React 16이 `expirationTime` 중심이었다면,  
React 18+는 **lane 기반 우선순위 모델**을 중심으로 이해하는 것이 맞다.

---

## 1. React 18+를 볼 때 가장 먼저 바꿔야 하는 관점

React 18+를 공부할 때 가장 중요한 포인트는  
**더 이상 `expirationTime` 중심으로 생각하지 않는 것**이다.

React 18의 핵심 기능들은 새 concurrent renderer 위에서 동작하며,  
업데이트 우선순위는 숫자 하나가 아니라 **lane(비트마스크 기반 우선순위 그룹)** 으로 관리된다.  
공식 블로그도 React 18의 핵심을 concurrent rendering, automatic batching, `startTransition` 같은 기능으로 설명한다. :contentReference[oaicite:1]{index=1}

즉 React 18+ 학습에서는 이렇게 기억하면 된다.

- React 16: `expirationTime`
- React 18+: `lane`

---

## 2. 전체 흐름 먼저 보기

React 18+에서 업데이트가 발생하면 내부적으로는 대략 다음 순서로 흐른다.

1. 상태 업데이트 발생
2. `scheduleUpdateOnFiber` 호출
3. root에 어떤 lane의 작업이 생겼는지 기록
4. `ensureRootIsScheduled` 호출
5. 현재 root에서 가장 먼저 처리할 lane 집합 결정
6. sync / concurrent 성격에 맞게 callback 예약
7. 렌더링 수행
8. 더 급한 작업이 오면 중단, 재개, 폐기될 수 있음

여기서 가장 중요한 차이는,  
React 18의 렌더링은 **interruptible** 하다는 점이다.  
공식 문서는 concurrent rendering을 “중단 가능하고, 더 급한 작업을 처리하기 위해 멈췄다가 이어서 할 수 있는 렌더링”으로 설명한다. :contentReference[oaicite:2]{index=2}

---

## 3. `scheduleUpdateOnFiber`의 역할

### 3-1. 업데이트 스케줄링의 시작점

React 18+에서도 `scheduleUpdateOnFiber`는  
업데이트 스케줄링의 진입점이다.

이 함수는 단순히 “작업을 넣는 함수”가 아니라,  
**어떤 root에 어떤 lane의 업데이트가 생겼는지 반영하고, 그 root가 적절히 예약되도록 만드는 시작점**이다.

즉 학습할 때는 이렇게 이해하면 된다.

> `scheduleUpdateOnFiber`는  
> **업데이트를 root까지 연결하고, 스케줄링을 시작하게 만드는 함수**다.

---

### 3-2. React 16과 비교했을 때 달라진 핵심

React 16에서는 이 흐름을 설명할 때 `expirationTime` 전파가 중심이었다.

React 18+에서는 같은 위치의 함수를 설명하더라도,  
이제는 `expirationTime` 대신 **lane** 이 중심이다.

즉 업데이트가 발생하면 React는  
“이 작업이 어떤 lane에 속하는가”를 기준으로 root 상태를 갱신하고,  
이후 어떤 작업을 먼저 실행할지 정한다. lane 모델은 React 18 내부에서 우선순위와 배치 대상을 표현하는 핵심 구조다. :contentReference[oaicite:3]{index=3}

---

## 4. lane이란 무엇인가?

### 4-1. React 18+ 우선순위의 핵심 표현 방식

lane은 React 18+에서 업데이트 우선순위를 표현하는 내부 모델이다.

단순히 “큰 숫자 / 작은 숫자”가 아니라,  
**비트마스크 기반의 우선순위 그룹**으로 이해하면 된다.

이 방식의 장점은:

- 여러 작업을 동시에 추적하기 쉽고
- 어떤 작업끼리 함께 처리할지 표현하기 쉽고
- transition, retry, idle 같은 작업 성격을 분리하기 좋고
- concurrent rendering에 더 잘 맞는다는 점이다. :contentReference[oaicite:4]{index=4}

---

### 4-2. lane을 왜 도입했는가?

React 18은 concurrent rendering을 도입하면서  
단순한 sync/async 구분만으로는 부족해졌다.

예를 들어 React는 서로 다른 성격의 작업을 구분해야 한다.

- 바로 반응해야 하는 입력
- 일반 상태 업데이트
- transition 업데이트
- retry 작업
- idle 작업

이런 작업들을 숫자 하나의 만료 시각으로만 관리하는 것보다,  
lane처럼 **우선순위 그룹 자체를 표현하는 모델**이 훨씬 유연하다. :contentReference[oaicite:5]{index=5}

---

## 5. `ensureRootIsScheduled`의 역할

### 5-1. 실제 예약 판단의 중심

React 18+에서도 `ensureRootIsScheduled`는 매우 중요하다.

이 함수는 현재 root를 보고:

- 지금 남아 있는 lane이 무엇인지
- 그중 어떤 lane을 다음에 처리해야 하는지
- 기존 callback을 유지할지
- 새 callback을 등록할지

를 결정한다.

즉 `scheduleUpdateOnFiber`가 스케줄링의 시작점이라면,  
`ensureRootIsScheduled`는 **실제 예약 판단의 중심**이다.

---

### 5-2. React 18+에서 왜 더 중요해졌는가?

React 18+는 concurrent renderer를 사용하기 때문에,  
단순히 “새 작업 하나 등록”으로 끝나지 않는다.

현재 진행 중인 작업이 있더라도:

- 더 급한 lane이 들어오면 우선순위를 재조정할 수 있고
- 기존에 예약된 callback을 취소하고 다시 잡을 수 있고
- 어떤 렌더는 끝까지 가지 못하고 중간에 중단될 수 있다

즉 `ensureRootIsScheduled`는  
**현재 root의 전체 작업 상태를 보고 다음 실행 전략을 재계산하는 함수**로 이해하는 것이 좋다. :contentReference[oaicite:6]{index=6}

---

## 6. React 18+에서 중요한 것은 "동기/비동기"보다 "작업 성격"

React 16 학습에서는  
“이게 sync냐 async냐”가 큰 축이었다.

React 18+에서는 물론 sync 개념도 중요하지만,  
더 중요한 것은 **작업의 성격과 lane 종류**다.

즉 React 18+에서는 이렇게 생각하는 편이 더 맞다.

- 이것은 즉시 반응해야 하는 업데이트인가?
- 이것은 transition인가?
- 이것은 느려도 되는 작업인가?
- 이것은 concurrent하게 중단 가능한가?

이 관점은 공식 문서의 `startTransition`, `useTransition` 설명과도 맞닿아 있다.  
Transitions는 “느린 기기에서도 인터페이스를 responsive하게 유지하도록, 긴급하지 않은 업데이트를 별도로 표시하는 것”으로 설명된다. :contentReference[oaicite:7]{index=7}

---

## 7. `startTransition`이 왜 중요한가?

### 7-1. React 18+의 실전 우선순위 API

React 18+를 학습할 때는 내부 함수 이름만 보는 것보다  
`startTransition`을 함께 이해하는 것이 훨씬 중요하다.

공식 문서에 따르면 `startTransition`은  
업데이트를 **긴급하지 않은 transition** 으로 표시해서,  
UI가 더 중요한 입력 반응을 우선할 수 있게 만든다. :contentReference[oaicite:8]{index=8}

즉 개발자 입장에서는:

- 입력창 값 반영: urgent
- 큰 리스트 필터링 결과 렌더링: transition

처럼 나눠 생각할 수 있다.

이건 React 18+를 공부할 때 반드시 연결해서 이해해야 하는 포인트다.

---

### 7-2. 내부 스케줄링과 연결하면

`startTransition`은 단순한 편의 API가 아니라,  
React가 해당 업데이트를 **덜 급한 lane 그룹**으로 취급하도록 만드는 실전 도구라고 이해할 수 있다.

즉 내부적으로는 lane 기반 스케줄링과 연결되고,  
개발자는 외부 API로 “이 작업은 지금 당장 급하지 않다”는 의도를 전달하게 된다. :contentReference[oaicite:9]{index=9}

---

## 8. Automatic Batching도 함께 이해해야 한다

React 18의 대표 변화 중 하나는 **automatic batching** 이다.

공식 업그레이드 가이드는 React 18에서 batching이 더 넓은 범위로 적용된다고 설명한다.  
이전에는 React 이벤트 핸들러 안에서 주로 batching되던 것이,  
이제는 여러 비동기 경로에서도 batching이 더 자연스럽게 적용된다. :contentReference[oaicite:10]{index=10}

이게 왜 중요하냐면,  
React 18+의 스케줄링은 단순히 “한 update를 언제 실행할까”가 아니라  
**여러 update를 어떤 우선순위로 묶어서 처리할까**까지 포함하기 때문이다.

즉 React 18+를 공부할 때는  
lane과 batching을 함께 보는 것이 좋다.

---

## 9. Concurrent Rendering은 무엇을 바꿨는가?

Concurrent rendering은 React 18의 핵심 개념이다.

공식 문서 기준으로 concurrent rendering은 다음 특징을 가진다.

- 렌더링이 interruptible하다
- 진행 중인 렌더를 멈출 수 있다
- 더 급한 작업을 먼저 처리할 수 있다
- 어떤 렌더는 끝까지 가지 않고 버려질 수 있다 :contentReference[oaicite:11]{index=11}

이것이 의미하는 바는 명확하다.

> React 18+에서 스케줄링은  
> “작업을 한 번 시작하면 끝까지 간다”가 아니라,  
> **중간에 우선순위를 바꿔가며 더 중요한 작업에 먼저 반응하는 방식**이다.

그래서 React 18+를 이해할 때는  
`ensureRootIsScheduled` 같은 함수를 “예약만 하는 함수”가 아니라  
**동적인 우선순위 조정의 중심 함수**로 보는 편이 맞다.

## 10. React 16은 어디까지 알면 충분한가?

React 18+를 공부하는 사람에게 React 16 지식이 전혀 필요 없는 건 아니다.

다만 React 16은 **배경지식** 정도로 보면 충분하다.

알아두면 좋은 포인트는 이 정도다.

- React 16은 `expirationTime` 중심이었다
- 업데이트 우선순위를 시간 기반 숫자로 표현했다
- React 18+는 이 방식을 lane 모델로 바꿨다

즉 React 16을 깊게 파는 목적은  
“현재 모델을 이해하기 위한 역사적 배경” 정도면 충분하다.

React 18+ 학습의 중심은 여전히:

- lane
- concurrent rendering
- automatic batching
- transition
- Suspense와의 연결

쪽에 있어야 한다. :contentReference[oaicite:12]{index=12}

## 11. React 18+ 학습 순서 추천

React 18+를 공부할 때는 아래 순서가 가장 자연스럽다.

### 1단계. 큰 그림

- concurrent rendering이 무엇인지
- automatic batching이 무엇인지
- transition이 왜 필요한지 이해

### 2단계. 우선순위 모델

- `expirationTime` 대신 `lane`을 사용한다는 점 이해
- lane이 “우선순위 그룹”이라는 점 이해

### 3단계. 내부 함수 흐름

- `scheduleUpdateOnFiber`: 시작점
- `ensureRootIsScheduled`: 실제 예약 판단
- root 기준으로 작업을 보고 우선순위를 재계산한다는 점 이해

### 4단계. 실전 API 연결

- `startTransition`
- `useTransition`
- batching
- 필요할 때 `flushSync`

이 순서로 보면 내부 구현과 실제 사용법이 자연스럽게 연결된다. :contentReference[oaicite:13]{index=13}

## 13. 복기

### Q. React 18+에서 `scheduleUpdateOnFiber`는 어떤 역할을 하는가?

React 18+에서 `scheduleUpdateOnFiber`는 업데이트 스케줄링의 시작점이다.  
업데이트가 발생한 Fiber를 기준으로 root 상태를 갱신하고,  
해당 업데이트가 속한 lane을 반영한 뒤 `ensureRootIsScheduled`를 통해 실제 예약 판단으로 연결한다.

---

### Q. React 18+에서 `ensureRootIsScheduled`는 왜 중요한가?

`ensureRootIsScheduled`는 현재 root에 남아 있는 작업들을 보고  
다음에 어떤 lane을 우선 처리할지 결정하는 함수다.  
Concurrent rendering 환경에서는 기존 callback을 유지할지, 바꿀지, 더 높은 우선순위 작업을 먼저 처리할지 판단해야 하므로 특히 중요하다.

---

### Q. React 16과 가장 큰 차이는 무엇인가?

가장 큰 차이는 우선순위 모델이다.  
React 16은 `expirationTime` 중심이었고,  
React 18+는 `lane` 기반 모델을 사용한다.  
또 React 18+는 concurrent rendering, automatic batching, transition 같은 실전 개념이 핵심이다.

---

## 14. 최종 결론

React 18+를 공부하려면  
`scheduleUpdateOnFiber`와 `ensureRootIsScheduled`를  
단순한 내부 함수 이름으로 외우기보다,  
**lane 기반 우선순위와 concurrent rendering의 흐름 속에서 이해하는 것**이 중요하다.

핵심은 다음이다.

- `scheduleUpdateOnFiber`는 업데이트 스케줄링의 시작점이다.
- React 18+는 `expirationTime` 대신 `lane`으로 우선순위를 표현한다.
- `ensureRootIsScheduled`는 root 기준으로 다음 작업을 결정하는 핵심 함수다.
- concurrent rendering 덕분에 렌더링은 중단, 재개, 우선순위 변경이 가능하다.
- 실전에서는 `startTransition`, `useTransition`, automatic batching과 함께 이해해야 한다.
