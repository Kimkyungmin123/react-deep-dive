# React Fiber — `ensureRootIsScheduled`와 워크 우선순위 비교

## 1. `ensureRootIsScheduled` 개요

`ensureRootIsScheduled`는 **새로운 워크가 발생했을 때, 기존에 스케줄링된 워크와 우선순위를 비교하여 어떤 워크를 진행할지 결정하는 함수**이다.

전체 흐름을 먼저 보면:

```
새로운 setState / 이벤트 발생
        │
        ▼
ensureRootIsScheduled(root)
        │
        ├─ 1. 새로운 워크의 우선순위 추론
        │       expirationTime + currentTime → inferPriorityFromExpirationTime
        │
        ├─ 2. 기존 워크와 우선순위 비교
        │       기존 ≥ 새로운  →  함수 종료 (기존 워크 유지)
        │       새로운 > 기존  →  기존 워크 취소 (cancelCallback)
        │
        └─ 3. 새로운 워크 스케줄링
                expirationTime === Sync  →  scheduleSyncCallback
                expirationTime !== Sync  →  scheduleCallback
                반환값(callbackNode)을 루트에 저장
```

---

## 2. 루트(Root)의 개념과 역할

### 2.1 루트란?

- `ReactDOM.render()` 호출 시 생성되는 **FiberRootNode**를 가리킨다.
- 일반적으로 CRA 환경에서 `id="root"`인 `<div>`에 연결되는 React 전용 루트 객체이다.
- **루트 ↔ VDOM은 1:1 관계**입니다. 하나의 루트는 하나의 Fiber 트리를 가진다.

> ⚠️ **주의**: "루트"는 HTML의 `<div id="root">` DOM 노드 자체가 아니라, React가 내부적으로 생성하는 **FiberRootNode** 객체이다. DOM 노드와 혼동하지 말것.

### 2.2 루트에 저장되는 스케줄링 정보

| 필드               | 설명                                           |
| ------------------ | ---------------------------------------------- |
| `callbackNode`     | 현재 스케줄링된 태스크 핸들 (Scheduler가 반환) |
| `callbackPriority` | 현재 스케줄링된 워크의 우선순위                |
| `expirationTime`   | 현재 처리해야 할 워크의 만료 시간              |

루트 하나에 **처리 중인 워크(태스크)는 항상 하나만 존재**한다.  
새로운 워크가 더 높은 우선순위를 가지면 기존 워크를 취소하고 교체한다.

---

## 3. 우선순위 추론: `inferPriorityFromExpirationTime`

### 3.1 필요한 두 가지 정보

```javascript
// 루트에서 다음에 처리해야 할 워크의 expirationTime 가져오기
const expirationTime = getNextRootExpirationTimeToWorkOn(root);

// 현재 시각 기준의 currentTime 가져오기
const currentTime = requestCurrentTimeForUpdate();
```

### 3.2 `inferPriorityFromExpirationTime` 로직

이 함수는 **"이 워크가 언제까지 처리되어야 하는가"** 를 바탕으로 우선순위를 추론한다.

```
expirationTime이 Sync?
    → ImmediatePriority (즉시 처리)

expirationTime이 Never 또는 Idle?
    → IdlePriority (최대한 늦게 처리)

그 외 → msUntil 계산 (expirationTime - currentTime, 단위: ms)
    msUntil < HIGH_PRIORITY_EXPIRATION + BATCH_SIZE?
        → UserBlockingPriority (사용자 인터랙션 수준)
    msUntil < LOW_PRIORITY_EXPIRATION + BATCH_SIZE?
        → NormalPriority (일반 업데이트)
    그 외
        → IdlePriority
```

> ⚠️ **자주 혼동하는 부분**: `msUntil`의 단위는 **밀리초(ms)** 이다. 마이크로초(μs)가 아니다.  
> 또한 "큰 값 = 이후 시점"이 맞지만, **큰 값일수록 우선순위가 낮다** (시간이 많이 남았으므로).

### 3.3 우선순위 레벨 요약

| 우선순위               | 값  | 사용 조건                              |
| ---------------------- | --- | -------------------------------------- |
| `ImmediatePriority`    | 1   | `expirationTime === Sync`              |
| `UserBlockingPriority` | 2   | msUntil이 매우 작음 (긴박한 인터랙션)  |
| `NormalPriority`       | 3   | msUntil이 중간 수준                    |
| `LowPriority`          | 4   | (잘 사용되지 않음)                     |
| `IdlePriority`         | 5   | `Never`, `Idle` 또는 msUntil이 매우 큼 |

## 4. 기존 워크와의 비교 및 처리 로직

### 4.1 비교 흐름

```javascript
const existingCallbackNode = root.callbackNode;

// 현재 스케줄링된 워크가 없으면 바로 새 워크 등록
if (existingCallbackNode === null) {
  // → 새 워크 스케줄링으로 진행
}

const existingCallbackPriority = root.callbackPriority;
const existingExpirationTime = root.expirationTime;

// expirationTime이 같고 기존 우선순위가 같거나 높으면 → 기존 워크 유지
if (
  expirationTime === existingExpirationTime &&
  existingCallbackPriority >= newCallbackPriority
) {
  return; // 함수 종료, 기존 워크 그대로 진행
}

// 새 워크 우선순위가 더 높으면 기존 워크 취소
cancelCallback(existingCallbackNode);
```

> ⚠️ **주의**: 비교 조건은 단순히 우선순위 숫자만 비교하는 것이 아니다.  
> **`expirationTime`이 같은지도 함께 확인**한다. `expirationTime`이 다르면 처리해야 할 워크 자체가 다른 것이므로 기존 워크를 취소하고 새로 등록한다.

### 4.2 새로운 워크 스케줄링

```javascript
let callbackNode;

if (expirationTime === Sync) {
  // Sync 워크: Reconciler의 syncQueue에 등록
  scheduleSyncCallback(performSyncWorkOnRoot.bind(null, root));
  callbackNode = fakeActCallbackNode; // Sync는 Scheduler 태스크 핸들이 없음
} else {
  // Concurrent 워크: Scheduler에 위임
  callbackNode = scheduleCallback(
    priorityLevel,
    performConcurrentWorkOnRoot.bind(null, root),
  );
}

// 루트에 새로운 워크 정보 저장
root.callbackNode = callbackNode;
root.callbackPriority = newCallbackPriority;
```

### 4.3 처리 결과 정리

| 상황                                | 처리 방식                           |
| ----------------------------------- | ----------------------------------- |
| 기존 워크 없음                      | 즉시 새 워크 스케줄링               |
| 기존 ≥ 새로운 (같은 expirationTime) | 함수 종료, 기존 워크 유지           |
| 새로운 > 기존                       | `cancelCallback` → 새 워크 스케줄링 |
| `expirationTime === Sync`           | `scheduleSyncCallback`              |
| `expirationTime !== Sync`           | `scheduleCallback`                  |

## 5. React 18+과의 대응 개념

> React 18에서는 `expirationTime` 기반 우선순위 시스템이 **Lanes 모델**로 완전히 대체되었다.  
> `ensureRootIsScheduled` 함수 자체는 React 18에도 존재하지만 내부 동작이 크게 달라졌다.

| React 16/17                         | React 18+                       | 변화 요약                        |
| ----------------------------------- | ------------------------------- | -------------------------------- |
| `expirationTime` (정수, 시간 기반)  | **Lanes** (비트마스크)          | 복수 우선순위 동시 표현 가능     |
| `inferPriorityFromExpirationTime`   | `getHighestPriorityLane` 등     | Lane 비트 연산으로 우선순위 판단 |
| `getNextRootExpirationTimeToWorkOn` | `getNextLanes`                  | 다음 처리할 Lane 집합 반환       |
| `cancelCallback` + 재스케줄링       | 동일 패턴 유지 (Lane 기준 비교) | 비교 기준이 시간→Lane으로 변경   |
| `callbackPriority` (Priority Level) | `callbackPriority` (Lane)       | 저장 단위가 Lane으로 변경        |
| `scheduleSyncCallback`              | 내부 변경, `SyncLane` 기준 판단 | Lane 값으로 Sync 여부 결정       |

📌 **핵심 차이**: React 16에서는 **"시간이 얼마나 남았는가"** 로 우선순위를 추론했지만,  
React 18에서는 **"어떤 Lane에 속하는가"** 로 우선순위를 직접 표현한다.  
`startTransition`을 사용하면 해당 업데이트는 `TransitionLane`에 배정되어 자동으로 낮은 우선순위로 처리된다.

---

## 6. 전체 흐름 요약

```
setState 호출
    │
    ▼
ensureRootIsScheduled(root)
    │
    ├─[1] expirationTime = getNextRootExpirationTimeToWorkOn(root)
    │     currentTime    = requestCurrentTimeForUpdate()
    │     newPriority    = inferPriorityFromExpirationTime(expirationTime, currentTime)
    │
    ├─[2] existingCallbackNode 존재?
    │       NO  → [3]으로 이동
    │       YES → existingPriority 비교
    │               기존 ≥ 새로운 (같은 expirationTime)  → return (종료)
    │               새로운 > 기존                        → cancelCallback(existingCallbackNode)
    │
    └─[3] 새 워크 스케줄링
            Sync?  → scheduleSyncCallback(performSyncWorkOnRoot)
            비Sync → scheduleCallback(priority, performConcurrentWorkOnRoot)
            root.callbackNode     ← 반환된 callbackNode
            root.callbackPriority ← newPriority
```

# React 18+ — 워크 우선순위 비교와 Lanes 기반 스케줄링

## 1. React 18에서 우선순위는 어떻게 표현되나?

React 16/17은 **expirationTime(만료 시간)** 이라는 숫자로 우선순위를 표현했다.  
React 18은 이를 완전히 폐기하고 **Lanes(레인)** 이라는 비트마스크 모델로 대체했다.

### 1.1 왜 Lanes로 바뀌었나?

**expirationTime의 한계:**

- 하나의 숫자 = 하나의 우선순위만 표현 가능
- 서로 다른 우선순위의 업데이트가 동시에 존재할 때 표현이 어려움
- 업데이트 배칭(batching)과 인터리빙(interleaving) 제어가 복잡

**Lanes의 장점:**

- 비트마스크이므로 **여러 우선순위를 동시에** 하나의 값으로 표현 가능
- 비트 연산(`&`, `|`, `~`)으로 빠른 비교·병합·제거 가능
- Automatic Batching 구현의 기반

```
// 개념적 예시 (실제 상수값과 다를 수 있음)
SyncLane            = 0b000001  // 비트 1
InputContinuousLane = 0b000100  // 비트 3
DefaultLane         = 0b010000  // 비트 5
TransitionLane      = 0b100000  // 비트 6

// "SyncLane과 DefaultLane이 동시에 pending"을 하나의 값으로 표현
pendingLanes = SyncLane | DefaultLane = 0b010001
```

### 1.2 주요 Lane 종류와 대응하는 상황

| Lane                    | 우선순위  | 발생 상황                         |
| ----------------------- | --------- | --------------------------------- |
| `SyncLane`              | 최고      | `flushSync`, 초기 렌더링 일부     |
| `InputContinuousLane`   | 매우 높음 | 연속 입력 이벤트 (드래그, 스크롤) |
| `DefaultLane`           | 보통      | 일반 `setState`                   |
| `TransitionLane` (1~16) | 낮음      | `startTransition` 내부 업데이트   |
| `IdleLane`              | 최저      | 완전히 여유로운 시점에 처리       |

---

## 2. `ensureRootIsScheduled` — React 18에서의 동작

`ensureRootIsScheduled`는 React 18에서도 존재하며, 역할은 동일하다.  
**"루트에 새 업데이트가 생겼을 때 어떤 워크를 스케줄링할지 결정"** 한다.

### 2.1 React 16 vs React 18 비교

| 단계           | React 16/17                                 | React 18+                                 |
| -------------- | ------------------------------------------- | ----------------------------------------- |
| 다음 워크 탐색 | `getNextRootExpirationTimeToWorkOn(root)`   | `getNextLanes(root, workInProgressLanes)` |
| 우선순위 추론  | `inferPriorityFromExpirationTime(...)`      | `lanesToSchedulerPriority(nextLanes)`     |
| 우선순위 비교  | `callbackPriority` (정수) 비교              | `existingCallbackPriority` (Lane) 비교    |
| 기존 워크 취소 | `cancelCallback(existingCallbackNode)`      | 동일                                      |
| Sync 판단      | `expirationTime === Sync`                   | `includesSyncLane(nextLanes)`             |
| 스케줄링       | `scheduleSyncCallback` / `scheduleCallback` | 동일 패턴 유지                            |

### 2.2 핵심 흐름 (React 18)

```
setState 또는 startTransition 호출
        │
        ▼
ensureRootIsScheduled(root)
        │
        ├─[1] nextLanes = getNextLanes(root, ...)
        │     schedulerPriority = lanesToSchedulerPriority(nextLanes)
        │
        ├─[2] 기존 callbackNode 존재?
        │       YES → existingPriority 비교
        │               같으면 → return (기존 워크 재사용)
        │               다르면 → cancelCallback(existingCallbackNode)
        │
        └─[3] SyncLane 포함?
                YES → scheduleSyncCallback(performSyncWorkOnRoot)
                NO  → scheduleCallback(schedulerPriority, performConcurrentWorkOnRoot)
                      root.callbackNode ← 반환된 callbackNode
```

## 3. 개발자 관점: 우선순위를 제어하는 공개 API

내부 Lanes 시스템은 직접 다루지 않아도 된다.  
React 18은 아래 공개 API로 우선순위를 제어할 수 있다.

### 3.1 `startTransition` — 업데이트를 TransitionLane으로 배정

```jsx
import { startTransition, useState } from "react";

function SearchPage() {
  const [input, setInput] = useState("");
  const [query, setQuery] = useState("");

  function handleChange(e) {
    // DefaultLane (긴급) — 입력창 즉시 반영
    setInput(e.target.value);

    // TransitionLane (비긴급) — 검색 결과 필터링
    startTransition(() => {
      setQuery(e.target.value);
    });
  }

  return (
    <>
      <input value={input} onChange={handleChange} />
      <HeavySearchResults query={query} />
    </>
  );
}
```

`startTransition` 내부의 `setQuery`는 `TransitionLane`에 배정된다.  
`ensureRootIsScheduled`는 `input` 업데이트(`DefaultLane`)가 오면 진행 중인 Transition 워크를 중단하고 먼저 처리한다.

### 3.2 `useTransition` — 진행 상태(isPending) 포함

```jsx
import { useTransition } from "react";

function TabBar() {
  const [isPending, startTransition] = useTransition();
  const [tab, setTab] = useState("home");

  return (
    <>
      <button onClick={() => startTransition(() => setTab("feed"))}>
        Feed
      </button>
      {/* TransitionLane 워크가 처리되는 동안 isPending === true */}
      {isPending && <Spinner />}
      <TabContent tab={tab} />
    </>
  );
}
```

### 3.3 `flushSync` — SyncLane으로 강제 처리

```jsx
import { flushSync } from "react-dom";

function handleClick() {
  flushSync(() => {
    setCount((c) => c + 1); // SyncLane으로 즉시 처리
  });
  // 이 시점에 DOM이 이미 업데이트되어 있음
  console.log(divRef.current.textContent);
}
```

> ⚠️ `flushSync`는 `SyncLane`을 강제로 사용하므로 Concurrent 기능(중단·재개)을 우회한다.
> DOM을 즉시 읽어야 하는 경우 외에는 가능하면 사용을 피해야 한다.

### 3.4 `useDeferredValue` — 값 기반 후순위 처리

```jsx
import { useDeferredValue } from "react";

function FilteredList({ items, filter }) {
  // filter 값이 바뀌어도, deferredFilter는 여유가 생길 때 업데이트
  const deferredFilter = useDeferredValue(filter);

  return <HeavyList items={items} filter={deferredFilter} />;
}
```

| API                | Lane 배정               | 사용 시점                   |
| ------------------ | ----------------------- | --------------------------- |
| 일반 `setState`    | `DefaultLane`           | 기본 업데이트               |
| `startTransition`  | `TransitionLane`        | 무거운 상태 업데이트        |
| `useDeferredValue` | `TransitionLane` (파생) | 외부 props 기반 렌더링 지연 |
| `flushSync`        | `SyncLane`              | DOM 즉시 읽기 필요 시       |

---

## 4. Automatic Batching — Lanes가 가능하게 한 것

React 18의 Automatic Batching은 Lanes 모델 덕분에 가능해졌다.

```jsx
// React 17 이하: setTimeout 내부는 배칭 안 됨 → 렌더링 2번
setTimeout(() => {
  setCount((c) => c + 1); // 렌더링 1번
  setFlag((f) => !f); // 렌더링 1번
}, 1000);

// React 18: 어디서든 자동 배칭 → 렌더링 1번
setTimeout(() => {
  setCount((c) => c + 1);
  setFlag((f) => !f);
  // 두 업데이트 모두 같은 Lane에 배정되어 하나의 렌더링으로 처리
}, 1000);
```

`ensureRootIsScheduled`는 두 번 호출되지만, 두 번째 호출 시 **이미 같은 우선순위의 워크가 스케줄링되어 있으므로** 추가 스케줄링 없이 종료된다. 이것이 배칭의 핵심 메커니즘이다.

## 5. 전체 정리

```
[React 18 우선순위 제어 전체 그림]

개발자 코드
    │
    ├── 일반 setState          → DefaultLane
    ├── startTransition        → TransitionLane (낮은 우선순위)
    ├── flushSync              → SyncLane (최고 우선순위)
    └── useDeferredValue       → TransitionLane (파생)
                │
                ▼
    ensureRootIsScheduled(root)
                │
                ├── getNextLanes → 다음 처리할 Lane 결정
                ├── 기존 워크와 Lane 비교
                │       같은 우선순위 → 유지
                │       더 높은 우선순위 → cancelCallback + 재스케줄
                └── scheduleCallback / scheduleSyncCallback
```

### React 18 마이그레이션 / 학습 체크리스트
