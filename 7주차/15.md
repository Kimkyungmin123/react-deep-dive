# React Fiber 구조에서 Concurrent Mode와 Sync Work 처리 방식

## 1. Fiber 아키텍처와 동시성 모드의 등장

### 1.1 이전 Stack 아키텍처의 한계

React 16 이전의 **Stack Reconciler**는 다음과 같은 구조적 한계가 있었다.

- 렌더링 작업이 **콜 스택(Call Stack)을 동기적으로 점유**했다.
- 한 번 시작된 워크(Work)를 **중단하거나 우선순위를 바꿀 수 없었다**.
- 무거운 렌더링이 시작되면 브라우저 이벤트 처리(클릭, 입력 등)가 지연될 수 있었다.

### 1.2 Fiber 아키텍처 도입 후 변화

React 16의 **Fiber Reconciler**는 렌더링 작업을 작은 단위인 **Fiber**로 쪼개어 처리한다.

| 구분           | Stack Reconciler | Fiber Reconciler |
| -------------- | ---------------- | ---------------- |
| 작업 중단      | 불가             | 가능             |
| 우선순위 조정  | 불가             | 가능             |
| 작업 재개/취소 | 불가             | 가능             |
| 처리 방식      | 항상 동기        | 동기 + 비동기    |

이 변화로 두 가지 작업 유형이 생겨났습니다.

- **Sync Work**: 우선순위 조정이 필요 없는 작업. Reconciler가 직접 처리한다.
- **Concurrent Work**: 우선순위 조정이 필요한 작업. Scheduler에 위임하여 처리한다.

> ⚠️ **자주 혼동하는 부분**: "Sync Work = 낮은 품질의 작업"이 아니다. Sync Work는 **우선순위 조정 자체가 불필요한** 작업(예: 초기 렌더링 중 일부, `flushSync` 강제 동기 업데이트)이다.

## 2. Sync Work와 Concurrent Work의 처리 흐름

### 2.1 Concurrent Work — `scheduleCallback`

```
Reconciler
  └─ scheduleCallback(priorityLevel, performConcurrentWorkOnRoot)
        └─ Scheduler
              └─ 우선순위·브라우저 상황을 고려하여 콜백 실행 시점 결정
                    └─ performConcurrentWorkOnRoot 실행
```

- **인자**: `priorityLevel` (우선순위 레벨), `performConcurrentWorkOnRoot` (루트에서 Concurrent Work를 수행하는 콜백)
- **역할**: Reconciler가 **Scheduler에게 실행 시점 결정을 위임**한다.
- **비동기적**으로 동작 — Scheduler는 브라우저의 유휴 시간(`MessageChannel` 또는 `setTimeout`)을 활용해 콜백을 실행한다.
- Reconciler는 Scheduler로부터 **태스크 핸들**을 반환받아 작업을 취소(`cancelCallback`)하거나 추적할 수 있다.

### 2.2 Sync Work — `scheduleSyncCallback`

```
Reconciler
  └─ scheduleSyncCallback(performSyncWorkOnRoot)
        ├─ syncQueue에 콜백 push
        └─ (최초 등록 시) scheduleCallback(ImmediatePriority, flushSyncCallbackQueueImpl)
              └─ Scheduler의 ImmediatePriority로 flush 예약
                    └─ flushSyncCallbackQueueImpl → syncQueue의 모든 콜백 실행
```

**`scheduleSyncCallback`의 두 가지 책임:**

1. **Sync Queue 관리**: Reconciler가 자체적으로 보유하는 큐(`syncQueue`)에 콜백을 쌓는다.
   - `syncQueue`가 없으면 새로 생성 후 첫 콜백을 push한다.
   - `syncQueue`가 이미 있으면 기존 큐에 콜백을 추가한다.

2. **큐 비우기(flush) 예약**: `syncQueue`를 실제로 실행하는 함수(`flushSyncCallbackQueueImpl`)를 **Scheduler의 ImmediatePriority**로 스케줄링한다.
   - 이미 예약된 flush가 있으면 중복 등록하지 않는다.
   - 이 단계에서 Scheduler를 사용하는 이유는 **"우선순위 조정"이 아니라**, 현재 실행 중인 작업이 끝난 직후 동기적으로 flush가 실행되도록 **실행 타이밍을 보장**하기 위해서다.

> ⚠️ **흔한 오해**: `scheduleSyncCallback`이 Scheduler에 flush를 위임하는 것을 "Scheduler가 Sync Work를 관리한다"고 이해하면 안된다. Scheduler는 큐를 **언제 비울지**만 알려줄 뿐이며, 큐 자체와 콜백은 여전히 **Reconciler가 소유**한다.

### 2.3 두 방식 비교

| 항목          | Sync Work                       | Concurrent Work          |
| ------------- | ------------------------------- | ------------------------ |
| 처리 주체     | Reconciler (직접)               | Scheduler (위임)         |
| 큐            | `syncQueue` (Reconciler 소유)   | Scheduler 내부 태스크 큐 |
| 중단/재개     | 불가                            | 가능                     |
| 우선순위 조정 | 없음                            | Scheduler가 담당         |
| 실행 타이밍   | ImmediatePriority flush 후 즉시 | Scheduler 판단에 따라    |

## 3. Scheduler의 역할

**Scheduler(`react-scheduler` 패키지)** 는 Reconciler와 독립된 별도 패키지이다.

핵심 역할:

- 등록된 태스크에 **우선순위**를 부여하고 실행 순서를 결정한다.
- **브라우저 프레임**(16ms) 내에서 처리 가능한 작업량을 계산하여 작업을 분할한다.
- 현재 프레임에서 처리하지 못한 작업은 다음 프레임으로 **양보(yield)** 한다.
- 우선순위 레벨: `ImmediatePriority > UserBlockingPriority > NormalPriority > LowPriority > IdlePriority`

## 4. React 18+과의 대응 개념

> React 18에서는 Concurrent Mode가 **기본값(opt-in → default)** 으로 바뀌었다.  
> 아래 표는 React 16/17 내부 개념과 React 18+의 대응 관계를 정리한 것이다.

| React 16/17 개념               | React 18+ 대응 개념                 | 변화 요약                                                  |
| ------------------------------ | ----------------------------------- | ---------------------------------------------------------- |
| `ReactDOM.render()`            | `ReactDOM.createRoot().render()`    | Concurrent 기능을 활성화하는 새 진입점                     |
| Concurrent Mode (실험적)       | **기본 동작** (createRoot 사용 시)  | 더 이상 별도 opt-in 불필요                                 |
| `scheduleCallback`             | 동일 (scheduler 패키지 유지)        | 내부 구현 일부 개선                                        |
| `scheduleSyncCallback`         | 내부 변경 (`ensureRootIsScheduled`) | Sync/Concurrent 판단 로직이 `ensureRootIsScheduled`로 통합 |
| 우선순위 레벨 (Priority Level) | **Lanes** 모델                      | 비트마스크 기반으로 복수 우선순위 동시 표현 가능           |
| 없음                           | `startTransition` / `useTransition` | Concurrent Work임을 명시적으로 표시                        |
| 없음                           | `useDeferredValue`                  | 특정 값의 업데이트를 후순위로 미룸                         |
| 없음                           | **Automatic Batching**              | 이벤트 핸들러 외부(setTimeout, Promise 등)에서도 자동 배칭 |
| `unstable_batchedUpdates`      | `flushSync` (강제 동기)             | 동기 강제 실행이 필요한 경우에만 사용                      |

> 📌 **핵심 변화 요약**: React 16/17에서는 Concurrent Mode를 개발자가 직접 선택해야 했고 내부적으로 `scheduleCallback`과 `scheduleSyncCallback`이 분리되어 있었습니다. React 18에서는 `createRoot` 하나로 Concurrent 기능이 기본 활성화되며, `startTransition`으로 업데이트의 긴급도를 개발자가 직접 힌트로 줄 수 있게 되었습니다.

```
[React 16/17 Fiber 스케줄링 요약]

setState / dispatchAction 호출
        │
        ▼
ensureRootIsScheduled (Reconciler)
        │
        ├─── Sync Work ─────► scheduleSyncCallback
        │                           │
        │                    syncQueue에 push
        │                           │
        │                    ImmediatePriority로 flush 예약
        │                           │
        │                    flushSyncCallbackQueueImpl 실행
        │
        └─── Concurrent Work ► scheduleCallback(priority, performConcurrentWorkOnRoot)
                                           │

                                           │
                                    performConcurrentWorkOnRoot 실행
```

# React 18+에서의 동시성(Concurrency)

## 1. React 18 이전과 이후: 무엇이 달라졌나?

### 1.1 핵심 변화: Concurrent 기능이 기본값이 되다

React 18의 가장 큰 변화는 **Concurrent 기능이 opt-in에서 기본값**으로 바뀐 것이다.

```jsx
// React 17 이하 — 항상 동기적 렌더링
ReactDOM.render(<App />, document.getElementById("root"));

// React 18 — createRoot 사용 시 Concurrent 기능 활성화
const root = ReactDOM.createRoot(document.getElementById("root"));
root.render(<App />);
```

`createRoot`를 사용하면 React가 내부적으로 **Concurrent Renderer**를 활성화한다.  
이전처럼 `ReactDOM.render()`를 계속 쓰면 "Legacy Mode"로 동작하며 동시성 기능을 쓸 수 없다.

> ⚠️ 단순히 React 18로 업그레이드하는 것만으로는 Concurrent 기능이 활성화되지 않는다. **`createRoot`로 변경해야 합니다.**

### 1.2 동시성(Concurrency)이란 무엇인가?

React에서 동시성이란 **"렌더링 작업에 우선순위를 부여하여, 중요한 업데이트가 덜 중요한 업데이트에 의해 지연되지 않도록 하는 것"**이다.

예시:

- 사용자가 검색창에 타이핑 → **긴급(Urgent)**: 입력값 표시가 즉시 반응해야 함
- 타이핑에 따른 검색 결과 목록 필터링 → **비긴급(Non-urgent)**: 약간 지연되어도 UX에 무방

React 18은 이 두 업데이트를 분리하여, 긴급 업데이트가 비긴급 업데이트를 중단할 수 있도록 한다.

## 2. 업데이트 우선순위 제어: `startTransition` / `useTransition`

### 2.1 `startTransition`

```jsx
import { startTransition } from "react";

function SearchInput() {
  const [inputValue, setInputValue] = useState("");
  const [searchQuery, setSearchQuery] = useState("");

  function handleChange(e) {
    // 긴급 업데이트: 입력창 값은 즉시 반영
    setInputValue(e.target.value);

    // 비긴급 업데이트: 검색 결과 필터링은 지연 허용
    startTransition(() => {
      setSearchQuery(e.target.value);
    });
  }

  return (
    <>
      <input value={inputValue} onChange={handleChange} />
      <SearchResults query={searchQuery} />
    </>
  );
}
```

`startTransition` 내부의 `setState`는 React에게 **"이 업데이트는 급하지 않습니다"** 라는 힌트를 줍니다.  
React는 더 긴급한 업데이트(예: 새 입력)가 들어오면 transition 업데이트를 **중단하고 재시작**할 수 있습니다.

> **React 16/17과의 연결**: `startTransition` 내부 업데이트는 내부적으로 `scheduleCallback`을 통해 Scheduler에 위임되는 **Concurrent Work**입니다. 일반 `setState`는 기본적으로 **Sync Work**에 가까운 높은 우선순위로 처리됩니다.

### 2.2 `useTransition` — 로딩 상태 포함

```jsx
import { useTransition } from "react";

function TabContainer() {
  const [isPending, startTransition] = useTransition();
  const [activeTab, setActiveTab] = useState("home");

  function handleTabClick(tab) {
    startTransition(() => {
      setActiveTab(tab);
    });
  }

  return (
    <>
      <TabButton onClick={() => handleTabClick("photos")}>Photos</TabButton>
      {/* isPending이 true인 동안 로딩 UI 표시 가능 */}
      {isPending && <Spinner />}
      <TabPanel tab={activeTab} />
    </>
  );
}
```

`useTransition`은 `startTransition`에 더해 **`isPending`** 플래그를 제공한다.  
Transition이 아직 처리 중일 때 `isPending`이 `true`가 되어 로딩 상태를 표시할 수 있다.

## 3. 값 업데이트 지연: `useDeferredValue`

`useDeferredValue`는 특정 **값**의 업데이트를 후순위로 미룬다.  
`startTransition`이 **"이 setState는 급하지 않다"** 라면, `useDeferredValue`는 **"이 값의 렌더링은 급하지 않다"** 이다.

```jsx
import { useDeferredValue } from "react";

function SearchResults({ query }) {
  // deferredQuery는 query가 바뀌어도 급한 렌더링이 끝난 후 반영됨
  const deferredQuery = useDeferredValue(query);

  return <HeavyList filter={deferredQuery} />;
}
```

| 항목      | `startTransition`         | `useDeferredValue`         |
| --------- | ------------------------- | -------------------------- |
| 사용 위치 | setState 호출 래핑        | 값(value) 래핑             |
| 제어 대상 | 상태 업데이트             | 파생 렌더링                |
| 주요 용도 | 이벤트 핸들러 내 업데이트 | 외부에서 내려받는 props/값 |

## 4. 자동 배칭(Automatic Batching)

### 4.1 React 17 이하의 배칭

React 17 이하에서는 **이벤트 핸들러 내부**에서만 여러 `setState`가 하나의 렌더링으로 묶였다.

```jsx
// React 17 이하
setTimeout(() => {
  setCount((c) => c + 1); // 렌더링 1번 발생
  setFlag((f) => !f); // 렌더링 1번 더 발생 → 총 2번
}, 1000);
```

### 4.2 React 18의 자동 배칭

React 18(`createRoot` 사용 시)에서는 **어디서든** 여러 `setState`가 자동으로 배칭된다.

```jsx
// React 18
setTimeout(() => {
  setCount((c) => c + 1);
  setFlag((f) => !f);
  // 두 setState가 하나의 렌더링으로 배칭됨 → 총 1번
}, 1000);

// Promise, async/await, fetch 콜백 등도 마찬가지
fetch("/api/data").then(() => {
  setData(result);
  setLoading(false);
  // 1번만 렌더링
});
```

### 4.3 배칭을 강제로 해제: `flushSync`

배칭을 원하지 않고 즉시 동기적으로 DOM을 업데이트해야 할 때는 `flushSync`를 사용한다.

```jsx
import { flushSync } from "react-dom";

function handleClick() {
  flushSync(() => {
    setCount((c) => c + 1);
  });
  // 이 시점에 DOM이 이미 업데이트되어 있음
  console.log(document.getElementById("count").textContent);

  flushSync(() => {
    setFlag((f) => !f);
  });
}
```

> **React 16/17과의 연결**: `flushSync`는 내부적으로 해당 업데이트를 **Sync Work**로 강제 처리한다. `flushSyncCallbackQueueImpl`을 즉시 실행하는 것과 유사한 효과이다.

## 5. Lanes 모델 — 우선순위 시스템의 변화

React 18은 내부적으로 React 16/17의 **Priority Level** 시스템을 **Lanes** 모델로 대체했다.

### 기존 Priority Level (React 16/17)

- 정수 기반 우선순위 레벨 (ImmediatePriority = 1, UserBlockingPriority = 2, ...)
- 하나의 숫자로 하나의 우선순위만 표현 가능

### Lanes (React 18)

- **비트마스크(bitmask)** 기반
- 여러 우선순위를 **동시에** 하나의 값으로 표현 가능
- 서로 다른 업데이트들의 배칭, 인터리빙(interleaving)이 더 정밀하게 제어됨

```
예시 (개념적):
SyncLane           = 0b0000000000000000000000000000001
InputContinuousLane= 0b0000000000000000000000000000100
DefaultLane        = 0b0000000000000000000000000010000
TransitionLane1    = 0b0000000000000000000000001000000
```

> 📌 Lanes는 내부 구현 세부사항입니다. 일반 개발자는 `startTransition`, `useDeferredValue` 등의 **공개 API**를 통해 우선순위를 제어하면 된다.

## 6. React 18 동시성 기능 전체 요약

```
사용자 상호작용 (e.g., 타이핑)
        │
        ▼
  setState 호출
        │
        ├─── 일반 setState ──────────────► 높은 우선순위(Urgent)로 처리
        │                                   브라우저 이벤트 처리와 동등한 수준
        │
        └─── startTransition 내부 ────────► 낮은 우선순위(Transition)로 처리
             useDeferredValue 사용               Scheduler가 여유 시간에 처리
                                                긴급 업데이트가 오면 중단·재시작 가능
```

| 기능               | 역할                             | 사용 시점                           |
| ------------------ | -------------------------------- | ----------------------------------- |
| `createRoot`       | Concurrent 렌더러 활성화         | 앱 진입점                           |
| `startTransition`  | 업데이트를 비긴급으로 표시       | 무거운 상태 업데이트                |
| `useTransition`    | 비긴급 업데이트 + 로딩 상태      | 탭 전환, 페이지 이동 등             |
| `useDeferredValue` | 값의 렌더링을 후순위로 지연      | 외부에서 내려오는 props 기반 필터링 |
| Automatic Batching | 여러 setState를 1번 렌더링으로   | 기본 동작 (별도 코드 불필요)        |
| `flushSync`        | 배칭 해제, 즉시 동기 렌더링 강제 | DOM 즉시 읽기가 필요한 경우         |

## 7. 마이그레이션 체크리스트 (React 17 → 18)

- [ ] `ReactDOM.render` → `ReactDOM.createRoot().render()`로 변경
- [ ] 무거운 상태 업데이트에 `startTransition` 또는 `useTransition` 적용 검토
- [ ] Automatic Batching으로 인해 의도치 않은 배칭이 발생하는지 확인
- [ ] 즉각적인 DOM 접근이 필요한 코드에 `flushSync` 적용 검토
- [ ] `unstable_batchedUpdates` 사용 코드가 있다면 제거 (자동 배칭으로 대체됨)
