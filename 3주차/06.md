# React Fiber — `useState` 내부 구조: 훅 객체, 큐 객체, 업데이트 객체

## 1. `useState` 호출 시 생성되는 세 가지 객체

```jsx
const [state, setState] = useState(initialValue);
//     │        │
//     │        └─ dispatch 함수 (dispatchAction 바인딩)
//     └─ hook.memoizedState (현재 상태값)
```

`useState`를 호출하면 내부적으로 세 가지 객체가 생성·연결된다.

| 객체                       | 역할                                           |
| -------------------------- | ---------------------------------------------- |
| **훅 객체 (Hook)**         | 상태값과 큐를 보관, 링크드 리스트로 연결       |
| **큐 객체 (Queue)**        | setState 호출 시 생성된 업데이트 객체들을 관리 |
| **업데이트 객체 (Update)** | 개별 setState 호출의 정보를 담는 객체          |

---

## 2. 훅 객체 (Hook Object)

### 2.1 구조

```javascript
const hook = {
  memoizedState: null, // 현재 확정된 상태값 (렌더링에 사용)
  baseState: null, // 우선순위 처리를 위한 기준 상태
  baseQueue: null, // 우선순위로 인해 건너뛴 업데이트들
  queue: null, // 업데이트 객체를 담는 큐 객체
  next: null, // 다음 훅 객체를 가리키는 포인터 (링크드 리스트)
};
```

`useState`가 반환하는 값과의 매핑:

```javascript
const [state, setState] = useState(0);
//     └─ hook.memoizedState    └─ hook.queue.dispatch
```

### 2.2 링크드 리스트 구조

컴포넌트 내 모든 훅은 **링크드 리스트**로 연결되며, Fiber 노드의 `memoizedState`에 첫 번째 훅이 저장된다.

```
Fiber.memoizedState
    │
    ▼
 Hook1 (useState)          Hook2 (useEffect)       Hook3 (useState)
 ┌─────────────────┐       ┌──────────────────┐    ┌─────────────────┐
 │ memoizedState:0 │──next►│ memoizedState:fn │────►│ memoizedState:''│
 │ queue: Queue1   │       │ queue: Queue2    │    │ queue: Queue3   │
 └─────────────────┘       └──────────────────┘    └─────────────────┘
```

> ⚠️ **훅이 배열이 아닌 링크드 리스트인 이유:**  
> React는 훅의 **호출 순서**를 기반으로 어떤 훅이 어떤 상태와 대응하는지 추적한다.  
> 링크드 리스트는 삽입·순회가 단순하며 순서가 보장된다.  
> 이것이 "훅은 조건문이나 반복문 안에서 호출하면 안 된다"는 규칙의 근거이다.

### 2.3 `mountWorkInProgressHook` — 훅 객체 생성

```javascript
function mountWorkInProgressHook() {
  const hook = {
    memoizedState: null,
    baseState: null,
    baseQueue: null,
    queue: null,
    next: null,
  };

  if (workInProgressHook === null) {
    // 첫 번째 훅 → Fiber의 memoizedState에 직접 연결
    currentlyRenderingFiber.memoizedState = workInProgressHook = hook;
  } else {
    // 이후 훅 → 이전 훅의 next에 연결
    workInProgressHook = workInProgressHook.next = hook;
  }

  return workInProgressHook;
}
```

## 3. 큐 객체 (Queue Object)

### 3.1 구조

```javascript
const queue = {
  last: null, // 마지막으로 추가된 업데이트 객체 (원형 링크드 리스트의 꼬리)
  dispatch: null, // setState 함수 (dispatchAction 바인딩)
  lastRenderedReducer: basicStateReducer, // 마지막 렌더에 사용된 리듀서
  lastRenderedState: initialState, // 마지막 렌더에 확정된 상태값
};
```

> ⚠️ **`last`가 `first`가 아닌 이유:**  
> 업데이트 객체들은 **원형 링크드 리스트(circular linked list)** 로 구성된다.  
> `last`를 알면 `last.next`로 첫 번째 업데이트에 O(1)로 접근할 수 있어 효율적이다.

### 3.2 `mountState` 내 큐 생성

```javascript
function mountState(initialState) {
  const hook = mountWorkInProgressHook(); // 훅 객체 생성

  // 초기 상태 설정
  hook.memoizedState = hook.baseState = initialState;

  // 큐 객체 생성
  const queue = (hook.queue = {
    last: null,
    dispatch: null,
    lastRenderedReducer: basicStateReducer,
    lastRenderedState: initialState,
  });

  // dispatch 함수 생성 (dispatchAction에 fiber와 queue를 바인딩)
  const dispatch = (queue.dispatch = dispatchAction.bind(
    null,
    currentlyRenderingFiber,
    queue,
  ));

  return [hook.memoizedState, dispatch];
}
```

## 4. 업데이트 객체 (Update Object)

`setState`(= `dispatch`)가 호출될 때마다 업데이트 객체가 생성되어 큐에 추가된다.

```javascript
const update = {
  expirationTime, // 이 업데이트의 우선순위
  suspenseConfig, // Suspense 관련 설정
  action, // setState에 전달된 값 또는 함수
  eagerReducer, // 조기 계산을 위한 리듀서
  eagerState, // 조기 계산된 상태값
  next, // 다음 업데이트 객체 (원형 링크드 리스트)
};
```

### 4.1 업데이트 객체의 원형 링크드 리스트

```
setState(1) → update1
setState(2) → update2
setState(3) → update3

queue.last = update3

update3.next = update1  ← 원형 구조
update1.next = update2
update2.next = update3

순회 시작: queue.last.next = update1 (첫 번째 업데이트)
```

## 5. 전체 구조 한눈에 보기

```
Fiber
└─ memoizedState
      │
      ▼
   Hook1 (useState(0))
   ├─ memoizedState: 0        ← state 반환값
   ├─ queue
   │    ├─ dispatch: fn       ← setState 반환값
   │    └─ last: Update3
   │              └─ next: Update1 → Update2 → Update3 (원형)
   └─ next
         │
         ▼
      Hook2 (useState(''))
      ├─ memoizedState: ''
      ├─ queue
      │    ├─ dispatch: fn
      │    └─ last: null
      └─ next: null
```

## 6. React 18+과의 대응 개념

| React 16/17                       | React 18+                | 변화 요약                 |
| --------------------------------- | ------------------------ | ------------------------- |
| `hook.memoizedState`              | 동일하게 유지            | 변화 없음                 |
| `hook.next` 링크드 리스트         | 동일하게 유지            | 변화 없음                 |
| `queue.last` (원형 링크드 리스트) | `queue.pending`으로 변경 | 필드명 변경, 역할 동일    |
| `update.expirationTime`           | **`update.lane`**        | Lane 기반 우선순위로 변경 |
| `dispatchAction`                  | `dispatchSetState`       | 함수명 변경 (역할 동일)   |
| `basicStateReducer`               | 동일하게 유지            | 변화 없음                 |

📌 **React 18의 주요 변화**: 업데이트 객체의 우선순위 필드가 `expirationTime`에서 `lane`으로 변경되었다.  
이에 따라 여러 `setState` 호출이 서로 다른 Lane에 배정될 수 있으며, Reconciler는 Lane을 기준으로 어떤 업데이트를 현재 렌더링에서 처리할지 결정한다.

## 7. 전체 흐름 요약

```
컴포넌트 첫 렌더링 (mountState)
        │
        ▼
mountWorkInProgressHook()
  → Hook 객체 생성
  → Fiber.memoizedState 링크드 리스트에 추가
        │
        ▼
Queue 객체 생성
  → hook.queue = { last: null, dispatch: null, ... }
        │
        ▼
dispatch = dispatchAction.bind(null, fiber, queue)
  → queue.dispatch = dispatch
        │
        ▼
return [hook.memoizedState, dispatch]
  → [state, setState]
```

# React 18+ — `useState` 내부 구조와 Lane 기반 업데이트

## 1. `useState`가 반환하는 것들의 정체

```jsx
const [count, setCount] = useState(0);
//     │         │
//     │         └─ hook.queue.dispatch (dispatchSetState 바인딩)
//     └─ hook.memoizedState (현재 확정 상태값)
```

`useState`의 반환값은 Fiber 노드에 연결된 **훅 객체(Hook)**의 내부 값이다.  
리렌더링이 발생할 때마다 React는 이 훅 객체에서 최신 `memoizedState`를 읽어 반환한다.

---

## 2. React 18에서 달라진 내부 구조

### 2.1 `queue.pending` (React 16의 `queue.last` 대체)

React 18에서는 대기 중인 업데이트를 `queue.pending`으로 관리한다.

```javascript
// React 16
const queue = {
  last: null, // 마지막 업데이트 (원형 링크드 리스트 꼬리)
  dispatch: null,
};

// React 18
const queue = {
  pending: null, // 대기 중인 업데이트 (원형 링크드 리스트 꼬리, 역할 동일)
  dispatch: null,
  lanes: NoLanes, // 이 큐에 pending된 업데이트들의 Lane 합집합
};
```

### 2.2 `update.lane` (React 16의 `update.expirationTime` 대체)

`setState`가 호출될 때 생성되는 업데이트 객체에 Lane이 기록된다.

```javascript
// React 16
const update = {
  expirationTime: ...,
  action: newState,
  next: null,
};

// React 18
const update = {
  lane: requestUpdateLane(fiber),  // 어떤 Lane인지 결정
  action: newState,
  hasEagerState: false,
  eagerState: null,
  next: null,
};
```

### 2.3 `dispatchSetState` (React 16의 `dispatchAction` 대체)

함수명이 변경되었지만 역할은 동일하다.

```javascript
// 컴포넌트 내부에서 setCount를 호출하면
setCount(1);
// 내부적으로 실행되는 것
dispatchSetState(fiber, queue, 1);
```

## 3. 훅 규칙이 존재하는 이유 — 링크드 리스트 순서 보장

React가 "훅은 조건문/반복문 안에서 호출하면 안 된다"고 규정하는 이유는 **링크드 리스트 순서** 때문이다.

```jsx
// ❌ 잘못된 예
function Bad({ flag }) {
  if (flag) {
    const [a, setA] = useState(0); // 조건부 호출
  }
  const [b, setB] = useState(""); // 순서가 flag에 따라 달라짐
}
```

```
flag=true 첫 렌더:
  Fiber.memoizedState → Hook(a) → Hook(b)

flag=false 리렌더:
  React는 순서대로 Hook을 읽음:
  첫 번째 Hook → b의 값을 기대하지만 → a의 Hook 객체를 읽음 → 불일치!
```

```jsx
// ✅ 올바른 예
function Good({ flag }) {
  const [a, setA] = useState(0); // 항상 첫 번째
  const [b, setB] = useState(""); // 항상 두 번째

  // flag에 따른 분기는 렌더 결과에만 적용
  if (flag) {
    /* ... */
  }
}
```

## 4. 여러 `setState` 호출이 처리되는 방식

### 4.1 단일 이벤트 내 여러 `setState`

```jsx
function Counter() {
  const [a, setA] = useState(0);
  const [b, setB] = useState(0);

  function handleClick() {
    setA(1); // update 객체 생성 → queue.pending에 추가
    setB(2); // update 객체 생성 → queue.pending에 추가
    // 두 호출 모두 DefaultLane → Automatic Batching → 1번 렌더링
  }
}
```

**Automatic Batching 메커니즘:**

```
setA(1) → dispatchSetState → queue_A.pending에 update 추가
          → ensureRootIsScheduled → DefaultLane Task 등록

setB(2) → dispatchSetState → queue_B.pending에 update 추가
          → ensureRootIsScheduled → 동일 Lane Task 이미 존재 → 재사용

→ 렌더링 1번만 발생 (두 큐 모두 소비)
```

### 4.2 `startTransition` 내부 `setState`

```jsx
function Search() {
  const [input, setInput] = useState("");
  const [results, setResults] = useState([]);

  function handleChange(e) {
    setInput(e.target.value); // DefaultLane → update.lane = DefaultLane
    startTransition(() => {
      setResults(search(e.target.value)); // TransitionLane → update.lane = TransitionLane
    });
  }
}
```

**큐 내 Lane별 업데이트:**

```
results_queue.pending:
  Update { lane: TransitionLane, action: [...results] }

렌더링 시 React는 현재 renderLanes와 update.lane을 비교:
  renderLanes에 TransitionLane 포함? YES → 이 update 처리
  renderLanes에 TransitionLane 포함? NO  → 이 update 건너뜀 (다음 렌더에서 처리)
```

## 5. `eagerState` — 불필요한 렌더링 방지 최적화

React 18의 `dispatchSetState`는 **같은 상태로의 업데이트를 조기에 감지**하여 렌더링을 생략한다.

```javascript
// dispatchSetState 내부 (개념적)
const currentState = queue.lastRenderedState;
const eagerState = queue.lastRenderedReducer(currentState, action);

if (Object.is(eagerState, currentState)) {
  // 상태가 변하지 않음 → 렌더링 스킵
  return;
}
// 상태가 변함 → update 객체 생성 → 렌더링 예약
```

```jsx
// 실제 동작 예시
const [count, setCount] = useState(0);

setCount(0); // 현재 상태(0)와 동일 → 렌더링 발생하지 않음 ✅
setCount(1); // 현재 상태(0)와 다름 → 렌더링 발생 ✅
```

> ⚠️ 단, `eagerState` 최적화는 **Idle 상태(렌더 페이즈 밖)**에서만 적용된다.  
> 렌더 페이즈 중 `setState`는 항상 update 객체를 생성한다.

## 6. `useReducer`와 `useState`의 관계

`useState`는 내부적으로 `useReducer`의 특수한 형태이다.

```javascript
// useState는 basicStateReducer를 사용하는 useReducer
function basicStateReducer(state, action) {
  return typeof action === "function" ? action(state) : action;
}

// useState(initialState)는 사실상:
useReducer(basicStateReducer, initialState);
```

이 때문에 `setState`에 **함수(업데이터 함수)** 를 전달할 수 있다.

```jsx
// 직접 값 전달
setCount(5);
// basicStateReducer(currentState, 5) → 5 반환

// 함수 전달 (이전 상태 기반)
setCount((prev) => prev + 1);
// basicStateReducer(currentState, prev => prev + 1) → currentState + 1 반환
```

## 7. 전체 구조 요약

```
[React 18 useState 전체 그림]

Fiber (Work in Progress)
└─ memoizedState (첫 번째 훅)
      │
      ▼
   Hook1 { memoizedState: 0, queue: Queue1, next: Hook2 }
   Hook2 { memoizedState: '', queue: Queue2, next: null }

Queue1 {
  pending: Update { lane: DefaultLane, action: 1, next: (원형) },
  dispatch: dispatchSetState.bind(null, fiber, Queue1),
  lanes: DefaultLane,
}

setCount(1) 호출 시:
  1. dispatchSetState(fiber, Queue1, 1)
  2. eagerState 계산 → 다른 값 → update 생성
  3. Queue1.pending에 update 추가
  4. ensureRootIsScheduled → 렌더링 예약
  5. 렌더링 시 updateState → Queue1.pending 소비 → memoizedState 갱신
  6. 새 memoizedState 반환 → 컴포넌트에 최신 state 전달
```

### 핵심 체크리스트

- [ ] `useState`가 반환하는 `state`는 `hook.memoizedState`이고 `setState`는 `hook.queue.dispatch`임을 이해했다.
- [ ] 훅이 링크드 리스트로 연결되기 때문에 훅 호출 순서가 항상 일정해야 함을 이해했다.
- [ ] React 18에서 `update.lane`으로 업데이트의 우선순위가 기록됨을 이해했다.
- [ ] `queue.pending` 원형 링크드 리스트의 꼬리를 저장하여 O(1)로 첫 요소에 접근할 수 있음을 이해했다.
- [ ] `eagerState` 최적화로 동일 상태 업데이트 시 렌더링이 생략됨을 이해했다.
- [ ] `useState`가 내부적으로 `basicStateReducer`를 사용하는 `useReducer`임을 이해했다.
