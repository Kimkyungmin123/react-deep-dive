# React Fiber — `dispatchAction`과 업데이트 큐 저장·소비 과정

> ⚠️ **용어 정정**: 원문에서 등장하는 오기 및 모호한 표현을 아래와 같이 정정한다.

| 원문 표현                 | 올바른 표현                 | 비고                                                 |
| ------------------------- | --------------------------- | ---------------------------------------------------- |
| "현재 작업 중인 함수"     | **Fiber**                   | 바인딩되는 첫 번째 인자는 현재 컴포넌트의 Fiber 노드 |
| "아이들(idle) 상태"       | **Idle 상태**               | 아무 업데이트도 진행 중이지 않은 기본 상태           |
| `scheduleWork`            | `scheduleUpdateOnFiber`     | React 16 내부에서 `scheduleWork`는 이 함수의 별칭    |
| `queue.lastRenderReducer` | `queue.lastRenderedReducer` | 정확한 필드명                                        |

---

## 1. `setState`와 `dispatchAction`의 관계

```javascript
// mountState 내부 (단순화)
const dispatch = dispatchAction.bind(null, currentlyRenderingFiber, queue);
queue.dispatch = dispatch;
return [hook.memoizedState, dispatch];
//                          └─ 이것이 setState
```

`setState`는 `dispatchAction`에 두 가지를 **바인딩**한 함수이다.

| 바인딩 인자 | 값      | 역할                                     |
| ----------- | ------- | ---------------------------------------- |
| 첫 번째     | `fiber` | 이 setState가 속한 컴포넌트의 Fiber 노드 |
| 두 번째     | `queue` | 이 훅의 업데이트 큐 객체                 |

이 두 값이 바인딩되어 있기 때문에 `setState`를 어디서 호출해도 React는 **어떤 컴포넌트의 어떤 훅**을 업데이트해야 하는지 정확히 알 수 있다.

---

## 2. `dispatchAction`의 4단계

```
setState(newValue) 호출
        │
        ▼ dispatchAction(fiber, queue, newValue)
        │
        ├─ [1단계] 업데이트 객체 생성
        ├─ [2단계] 큐에 업데이트 객체 저장 (원형 링크드 리스트)
        ├─ [3단계] 최적화: 불필요한 렌더링 방지
        └─ [4단계] scheduleUpdateOnFiber 호출 → Work 스케줄링
```

---

## 3. 1단계 — 업데이트 객체 생성

```javascript
const update = {
  expirationTime, // 이 업데이트의 우선순위 (큰 값 = 높은 우선순위)
  action, // setState에 전달된 값 또는 함수
  eagerReducer: null, // 최적화용: 미리 계산한 리듀서
  eagerState: null, // 최적화용: 미리 계산한 상태값
  next: null, // 원형 링크드 리스트의 다음 노드
};
```

- `action`은 `setState(5)` → `5`, `setState(prev => prev + 1)` → `prev => prev + 1`
- `next`는 여러 `setState` 호출이 연결되는 링크드 리스트 포인터

---

## 4. 2단계 — 원형 링크드 리스트로 큐에 저장

업데이트 객체는 `queue.last`를 꼬리로 하는 **원형 링크드 리스트**에 저장된다.

### 4.1 첫 번째 업데이트 저장

```javascript
if (queue.last === null) {
  // 큐가 비어있음 → 자기 자신을 가리키는 원형 구조 시작
  update.next = update; // 자신을 가리킴
  queue.last = update;
}
```

```
queue.last → [update1]
              └─ next: update1 (자기 자신)
```

### 4.2 두 번째 이후 업데이트 저장

```javascript
} else {
  const first = queue.last.next; // 현재 첫 번째 노드 (꼬리.next)
  update.next = first;           // 새 업데이트가 첫 번째를 가리킴
  queue.last.next = update;      // 기존 꼬리가 새 업데이트를 가리킴
  queue.last = update;           // 꼬리를 새 업데이트로 교체
}
```

**세 번의 `setState` 호출 후 결과:**

```
queue.last → [update3]
               └─ next → [update1]
                           └─ next → [update2]
                                      └─ next → [update3] (원형)
```

> ⚠️ **원형 링크드 리스트를 쓰는 이유:**  
> `queue.last`(꼬리)만 알면 `queue.last.next`로 **첫 번째 업데이트에 O(1)로 접근**할 수 있다.  
> 소비 시 항상 처음부터 순서대로 처리하기 위해 첫 번째 노드에 빠르게 접근하는 것이 중요하다.

---

## 5. 3단계 — 불필요한 렌더링 방지 최적화 (`eagerState`)

**조건**: Idle 상태이고, 현재 진행 중인 Work가 없을 때만 수행한다.

```javascript
// Idle 상태 최적화 블록
const lastRenderedReducer = queue.lastRenderedReducer; // basicStateReducer
const currentState = queue.lastRenderedState; // 현재 확정 상태값

// action으로 다음 상태를 미리 계산
const eagerState = lastRenderedReducer(currentState, action);
update.eagerReducer = lastRenderedReducer;
update.eagerState = eagerState;

// 현재 상태와 동일하면 렌더링 생략
if (Object.is(eagerState, currentState)) {
  return; // Work 스케줄링 없이 함수 종료
}
```

```
예시:
  현재 count = 5
  setCount(5) 호출 → eagerState = 5 === currentState(5) → return (렌더링 없음)
  setCount(6) 호출 → eagerState = 6 !== currentState(5) → 4단계로 진행
```

> ⚠️ **`eagerState` 최적화가 Idle 상태에서만 동작하는 이유:**  
> 렌더 페이즈나 커밋 페이즈 중에는 `queue.lastRenderedState`가 최신 상태가 아닐 수 있다.  
> 잘못된 상태값과 비교하여 렌더링을 생략하면 UI 불일치가 발생할 수 있기 때문이다.

---

## 6. 4단계 — Work 스케줄링

```javascript
scheduleUpdateOnFiber(fiber, expirationTime);
// → markUpdateTimeFromFiberToRoot → ensureRootIsScheduled → Scheduler에 Work 등록
```

---

## 7. 여러 번의 `setState` 호출이 하나의 렌더링으로 처리되는 이유

```jsx
function handleClick() {
  setCount(1); // dispatchAction → update1 생성 → queue에 저장 → scheduleUpdateOnFiber
  setCount(2); // dispatchAction → update2 생성 → queue에 저장 → scheduleUpdateOnFiber
  setCount(3); // dispatchAction → update3 생성 → queue에 저장 → scheduleUpdateOnFiber
}
```

세 번 모두 `scheduleUpdateOnFiber`를 호출하지만:

```
첫 번째 호출 → Scheduler에 Task 등록 (callbackNode 생성)
두 번째 호출 → ensureRootIsScheduled: 동일 우선순위 Task 이미 존재 → return (중복 등록 없음)
세 번째 호출 → 동일하게 중복 등록 없음

→ 이벤트 핸들러 종료 후 Scheduler가 Task 실행
→ 렌더링 1번: queue에 쌓인 update1, update2, update3 순서대로 소비
→ 최종 상태 = 3
```

---

## 8. 업데이트 소비: `updateState` (렌더링 시)

렌더링이 시작되면 `updateState` → `updateReducer`가 `queue.last`의 원형 링크드 리스트를 순회하며 업데이트를 소비한다.

```javascript
function updateReducer(reducer, initialArg) {
  // queue.last.next 에서 시작하여 원형 순회
  let update = queue.last.next; // 첫 번째 업데이트

  do {
    const action = update.action;
    newState = reducer(newState, action); // basicStateReducer 적용
    update = update.next;
  } while (update !== queue.last.next); // 원형 순회 종료 조건

  hook.memoizedState = newState; // 최종 상태 확정
  return [newState, dispatch];
}
```

## 9. React 18+과의 대응 개념

| React 16/17                                    | React 18+                                       | 변화 요약                  |
| ---------------------------------------------- | ----------------------------------------------- | -------------------------- |
| `dispatchAction`                               | `dispatchSetState`                              | 함수명 변경, 역할 동일     |
| `update.expirationTime`                        | `update.lane`                                   | Lane 기반 우선순위로 변경  |
| `queue.last` (원형 링크드 리스트 꼬리)         | `queue.pending`                                 | 필드명 변경, 역할 동일     |
| `eagerState` 최적화                            | 동일하게 유지                                   | `Object.is` 비교 로직 유지 |
| `scheduleUpdateOnFiber(fiber, expirationTime)` | `scheduleUpdateOnFiber(fiber, lane, eventTime)` | Lane 기반 인자로 변경      |

---

## 10. 전체 흐름 요약

```
setState(action) 호출
        │
        ▼
dispatchAction(fiber, queue, action)
        │
        ├─ [1] update 객체 생성 { expirationTime, action, next, eagerState, eagerReducer }
        │
        ├─ [2] queue.last 원형 링크드 리스트에 update 추가
        │
        ├─ [3] Idle 상태 + Work 없음?
        │       YES → eagerState 계산
        │               === currentState → return (렌더링 없음)
        │               !== currentState → 4단계 진행
        │       NO  → 4단계 진행
        │
        └─ [4] scheduleUpdateOnFiber(fiber, expirationTime)
                  → ensureRootIsScheduled → Scheduler에 Work 등록
                  → 렌더링 시 queue 소비 → memoizedState 갱신
```

# React 18+ — `dispatchSetState`와 Automatic Batching의 실제 메커니즘

## 1. React 16 `dispatchAction` → React 18 `dispatchSetState`

함수명이 바뀌었지만 핵심 4단계 구조는 동일하다.

| 단계                | React 16                                       | React 18                                        | 변화                    |
| ------------------- | ---------------------------------------------- | ----------------------------------------------- | ----------------------- |
| 1단계 업데이트 객체 | `expirationTime`, `action`                     | **`lane`**, `action`                            | 우선순위 표현 방식 변경 |
| 2단계 큐 저장       | `queue.last` 원형 링크드 리스트                | `queue.pending` 원형 링크드 리스트              | 필드명만 변경           |
| 3단계 최적화        | `eagerState` (`Object.is` 비교)                | 동일하게 유지                                   | 변화 없음               |
| 4단계 스케줄링      | `scheduleUpdateOnFiber(fiber, expirationTime)` | `scheduleUpdateOnFiber(fiber, lane, eventTime)` | Lane 기반으로 변경      |

---

## 2. `update.lane` — 업데이트 하나하나에 우선순위가 붙는다

React 18에서는 개별 업데이트 객체에 **Lane**이 기록된다.

```javascript
// dispatchSetState 내부
const lane = requestUpdateLane(fiber); // 컨텍스트 기반으로 Lane 결정

const update = {
  lane, // 이 업데이트의 우선순위 Lane
  action, // setState에 전달된 값 또는 함수
  hasEagerState: false,
  eagerState: null,
  next: null,
};
```

### 2.1 같은 큐에 다른 Lane의 업데이트가 공존하는 경우

```jsx
function App() {
  const [count, setCount] = useState(0);

  function handleClick() {
    setCount(1); // DefaultLane
    startTransition(() => setCount(2)); // TransitionLane
  }
}
```

```
queue.pending (원형 링크드 리스트):
  Update { lane: DefaultLane,     action: 1 }
  Update { lane: TransitionLane,  action: 2 }
```

렌더링 시 React는 현재 `renderLanes`와 `update.lane`을 AND 비교하여 이번 렌더에서 처리할 업데이트를 결정한다.

```javascript
// updateReducer 내부
if (!isSubsetOfLanes(renderLanes, update.lane)) {
  // renderLanes에 이 update의 lane이 포함되지 않음
  // → 이번 렌더에서 건너뜀 → baseQueue에 보관 → 다음 렌더에서 처리
  newBaseQueueLast.next = update;
} else {
  // renderLanes에 포함됨 → 이번 렌더에서 처리
  newState = reducer(newState, update.action);
}
```

## 3. Automatic Batching — `ensureRootIsScheduled` 재사용의 실체

React 18에서 Automatic Batching이 동작하는 구체적인 메커니즘이다.

```jsx
// React 18 (createRoot 사용)
setTimeout(() => {
  setA(1); // DefaultLane
  setB(2); // DefaultLane
  // → 렌더링 1번
}, 1000);
```

**내부 흐름 추적:**

```
setA(1) → dispatchSetState
  → update_A 생성 (lane: DefaultLane)
  → queue_A.pending에 추가
  → scheduleUpdateOnFiber → ensureRootIsScheduled
      → DefaultLane Task 없음 → scheduleCallback 호출 → callbackNode 저장

setB(2) → dispatchSetState
  → update_B 생성 (lane: DefaultLane)
  → queue_B.pending에 추가
  → scheduleUpdateOnFiber → ensureRootIsScheduled
      → DefaultLane Task 이미 존재 (callbackNode) → return (중복 등록 없음)

→ setTimeout 콜백 종료
→ Scheduler: 저장된 callbackNode 실행
→ 렌더링 1번: queue_A와 queue_B 모두 소비
→ A = 1, B = 2 동시 반영
```

### 3.1 React 17과의 비교

```jsx
// React 17 (Legacy Mode)
setTimeout(() => {
  setA(1); // → scheduleUpdateOnFiber → 렌더링 예약 → 즉시 flush
  setB(2); // → scheduleUpdateOnFiber → 렌더링 예약 → 즉시 flush
  // → 렌더링 2번
}, 1000);
```

React 17에서는 `setTimeout` 내부에서 `executionContext === NoContext`이므로 각 `setState`마다 즉시 `flushSyncCallbackQueue`가 실행되었다.  
React 18의 `createRoot`는 이 즉시 flush 경로를 Concurrent 경로로 대체하여 **항상 batching이 적용**된다.

## 4. `flushSync` — Batching 강제 해제

배칭을 원하지 않고 DOM을 즉시 업데이트해야 할 때 사용한다.

```jsx
import { flushSync } from "react-dom";

function handleClick() {
  flushSync(() => {
    setA(1); // SyncLane → 즉시 flush
  });
  // 이 시점에 A=1이 DOM에 반영되어 있음

  flushSync(() => {
    setB(2); // SyncLane → 즉시 flush
  });
  // 이 시점에 B=2도 DOM에 반영되어 있음
}
// → 렌더링 2번 (flushSync 블록마다 1번)
```

**`flushSync` 내부 메커니즘:**

```
flushSync 진입
  → executionContext에 BatchedContext 추가
  → 콜백 실행 (setState → SyncLane update 생성)
  → 콜백 종료
  → flushSyncCallbacks() 즉시 호출 → 렌더링 완료
  → executionContext 복구
```

> ⚠️ `flushSync` 내부에서 `startTransition`을 사용해도 효과가 없다.  
> `flushSync`는 내부의 모든 업데이트를 `SyncLane`으로 강제하기 때문이다.

## 5. 함수형 업데이터 — 배칭 환경에서 안전한 상태 업데이트

배칭이 적용되는 환경에서 여러 번 같은 상태를 증가시키려면 함수형 업데이터를 사용해야 한다.

```jsx
// ❌ 예상과 다른 결과 (배칭 환경)
function handleClick() {
  setCount(count + 1); // count = 0, eagerState = 1
  setCount(count + 1); // count = 0 (아직 렌더 안 됨), eagerState = 1
  setCount(count + 1); // count = 0, eagerState = 1
}
// → 렌더링 후 count = 1 (3이 아님)
```

```jsx
// ✅ 올바른 결과 (함수형 업데이터)
function handleClick() {
  setCount((prev) => prev + 1); // action: prev => prev + 1
  setCount((prev) => prev + 1); // action: prev => prev + 1
  setCount((prev) => prev + 1); // action: prev => prev + 1
}
// update 소비 과정:
//   0 → prev(0)+1=1 → prev(1)+1=2 → prev(2)+1=3
// → 렌더링 후 count = 3 ✅
```

**함수형 업데이터가 올바르게 동작하는 이유:**  
`updateReducer`가 큐의 update를 순서대로 소비할 때, 각 update의 `action` 함수에 **이전 update의 결과값**을 인자로 넘기기 때문이다.

## 6. `eagerState` 최적화와 함수형 업데이터의 상호작용

```jsx
const [count, setCount] = useState(0);

// 값 전달: eagerState 계산 가능 → 동일하면 렌더링 생략
setCount(0); // eagerState = 0 === currentState(0) → 렌더링 없음

// 함수 전달: eagerState 계산 불가 (이전 렌더 상태만 알고 중간 상태는 모름)
setCount((prev) => prev); // eagerState 계산 시도 → prev=0, 결과=0 → 렌더링 없음 ✅
setCount((prev) => prev + 1); // eagerState = 1 !== 0 → 렌더링 예약 ✅
```

> ⚠️ **`eagerState`의 한계**: 큐에 이미 처리되지 않은 업데이트가 있으면 `eagerState` 최적화가 적용되지 않는다.  
> 최적화는 큐가 비어있는 완전한 Idle 상태에서만 동작한다.

## 7. 전체 흐름 요약

```
[React 18 setState 호출 전체 흐름]

setState(action) 호출
        │
        ▼
dispatchSetState(fiber, queue, action)
        │
        ├─ [1] update 생성 { lane: requestUpdateLane(fiber), action }
        │
        ├─ [2] queue.pending 원형 링크드 리스트에 추가
        │
        ├─ [3] Idle + 큐 비어있음?
        │       YES → eagerState 계산 → 동일하면 return
        │       NO  → 4단계 진행
        │
        └─ [4] scheduleUpdateOnFiber(fiber, lane)
                  → ensureRootIsScheduled
                      → 동일 Lane Task 이미 있음? → return (배칭)
                      → 없음? → scheduleCallback → Task 등록
        │
        ▼ (이벤트 핸들러 종료 후 Scheduler 실행)
        │
updateReducer 실행 (렌더링 시)
  → queue.pending 원형 리스트 순회
  → update.lane ∩ renderLanes → 처리 or 건너뜀
  → 최종 newState 확정 → hook.memoizedState 갱신
```

### 핵심 체크리스트

- [ ] `setState`가 `dispatchSetState(fiber, queue, action)` 바인딩임을 이해했다.
- [ ] 업데이트 객체가 `queue.pending` 원형 링크드 리스트에 저장됨을 이해했다.
- [ ] Automatic Batching이 `ensureRootIsScheduled`에서 동일 Lane Task 재사용으로 구현됨을 이해했다.
- [ ] 배칭 환경에서 여러 번 증가시키려면 함수형 업데이터를 사용해야 함을 이해했다.
- [ ] `eagerState` 최적화가 Idle 상태 + 큐 비어있을 때만 동작함을 이해했다.
- [ ] `flushSync`가 배칭을 해제하고 SyncLane으로 즉시 처리함을 이해했다.
