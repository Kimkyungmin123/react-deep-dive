# React 내부 패키지 구조와 핵심 용어 정의

## 1. React 주요 패키지 구성

React는 단일 라이브러리가 아니라 **역할별로 분리된 여러 패키지**의 조합이다.

```
┌─────────────────────────────────────────────┐
│                   개발자 코드                  │
│  import React from 'react'                   │
│  import ReactDOM from 'react-dom'            │
└───────────┬─────────────────────────────────┘
            │
  ┌─────────▼──────────┐
  │    React Core       │ → 컴포넌트 정의, 훅 인터페이스
  │  (react 패키지)     │   플랫폼 독립적
  └─────────┬───────────┘
            │ shared 패키지를 통해 연결
  ┌─────────▼───────────┐
  │    Reconciler        │ → VDOM 재조정, 컴포넌트 호출
  │ (react-reconciler)   │   훅 실제 구현
  └──────┬──────┬────────┘
         │      │
  ┌──────▼──┐ ┌─▼────────────┐
  │Scheduler│ │  Renderer     │
  │(scheduler│ │ (react-dom   │
  │ 패키지)  │ │  react-native│
  └──────────┘ └──────────────┘
```

---

## 2. 각 패키지의 역할

### 2.1 React Core (`react` 패키지)

- **컴포넌트를 정의**하는 데 필요한 API를 제공한다.
- `useState`, `useEffect` 등 훅의 **인터페이스(껍데기)** 만 여기에 있다. 실제 구현은 Reconciler에 있다.
- **다른 패키지에 직접 의존하지 않는다** → 브라우저, 앱, 서버 등 어떤 플랫폼에서도 동일하게 사용 가능하다.

```javascript
// react 패키지가 제공하는 것들
import React, { useState, useEffect, useRef, createContext } from "react";
```

### 2.2 Renderer (`react-dom`, `react-native` 등)

- Reconciler가 계산한 변경 사항을 **실제 플랫폼(호스트 환경)에 적용**한다.
- 플랫폼마다 별도의 Renderer 패키지가 존재한다.

| Renderer            | 플랫폼      | 역할                       |
| ------------------- | ----------- | -------------------------- |
| `react-dom`         | 웹 브라우저 | DOM 노드 생성·수정·삭제    |
| `react-native`      | iOS/Android | 네이티브 뷰 생성·수정·삭제 |
| `react-three-fiber` | WebGL       | 3D 씬 그래프 조작          |
| `react-pdf`         | PDF         | PDF 요소 생성              |

- **Reconciler와 Legacy Event 패키지에 의존**한다.

### 2.3 Scheduler (`scheduler` 패키지)

- Work(재조정 작업)의 **실행 타이밍을 관리**한다.
- 우선순위에 따라 Task를 정렬하고, 브라우저 여유 시간(5ms 단위)에 실행한다.
- React와 **완전히 독립된 패키지**로, React 없이 단독 사용도 가능하다.

```
Scheduler의 역할:
  "이 Task를 언제 실행할까?"
  "더 급한 Task가 오면 현재 Task를 잠시 멈출까?"
```

### 2.4 Reconciler (`react-reconciler` 패키지)

- **Virtual DOM(VDOM)을 재조정(Reconciliation)** 하는 핵심 패키지이다.
- 컴포넌트 함수를 호출하고, 반환된 React 엘리먼트와 기존 Fiber를 비교하여 변경 사항을 파악한다.
- 훅(`useState`, `useEffect` 등)의 **실제 구현**이 여기에 있다.

> ⚠️ **Reconciler ≠ Renderer**: Reconciler는 "무엇이 바뀌었는가"를 계산하고, Renderer는 "실제로 바꾼다"는 역할이다. 둘은 명확히 분리되어 있다.

### 2.5 Legacy Event 시스템

- 브라우저의 기본 이벤트 시스템을 **React 전용으로 확장**한 래퍼(wrapper)이다.
- 이벤트 위임(Event Delegation), 합성 이벤트(SyntheticEvent), 브라우저 간 호환성 처리 등을 담당한다.

> 📌 React 17부터 이벤트 위임 대상이 `document`에서 **루트 DOM 컨테이너**로 변경되었다.

## 3. 핵심 용어 정의

### 3.1 React 엘리먼트 (React Element)

컴포넌트를 호출할 때 반환되는 **순수 자바스크립트 객체**이다.

```jsx
// JSX 작성
const element = <button className="primary">클릭</button>;

// 컴파일 후 (React.createElement 호출)
const element = React.createElement('button', { className: 'primary' }, '클릭');

// 결과 객체
{
  $$typeof: Symbol(react.element),
  type: 'button',       // 어떤 태그/컴포넌트인가
  key: null,
  ref: null,
  props: {
    className: 'primary',
    children: '클릭',
  },
}
```

**React 엘리먼트의 특징:**

- **불변(immutable)** 이다. 한 번 생성 후 변경 불가.
- UI의 "한 시점에 대한 설명서"일 뿐, 실제 DOM이 아니다.
- `type` 필드로 Host(DOM 태그), Custom(함수·클래스), Static(Symbol) 컴포넌트를 구분한다.

### 3.2 Fiber (파이버)

**Virtual DOM 노드 객체**이다.  
React 엘리먼트에 컴포넌트의 상태, 라이프사이클, 훅 등 **런타임 정보를 추가하여 확장**한 자바스크립트 객체이다.

```javascript
// Fiber 노드 구조 (주요 필드)
{
  // React 엘리먼트 정보
  type,             // 컴포넌트 함수 또는 DOM 태그 문자열
  key,
  props: {},

  // 런타임 상태 (React 엘리먼트에는 없음)
  memoizedState,    // 훅 링크드 리스트 (useState 등의 상태)
  memoizedProps,    // 현재 렌더링에 사용된 props
  updateQueue,      // 대기 중인 업데이트

  // 트리 연결
  child,            // 첫 번째 자식 Fiber
  sibling,          // 다음 형제 Fiber
  return,           // 부모 Fiber

  // 이중 버퍼링
  alternate,        // Current ↔ Work In Progress 상호 참조

  // 스케줄링
  lanes,            // 이 Fiber의 업데이트 우선순위 (React 18)

  // 커밋 정보
  flags,            // 변경 유형 (Placement, Update, Deletion)
  stateNode,        // 실제 DOM 노드 또는 클래스 컴포넌트 인스턴스
}
```

**React 엘리먼트 vs Fiber 비교:**

| 구분           | React 엘리먼트                        | Fiber                             |
| -------------- | ------------------------------------- | --------------------------------- |
| 생성 시점      | `render()` 또는 컴포넌트 함수 호출 시 | Reconciler가 엘리먼트를 처리할 때 |
| 변경 가능 여부 | 불변                                  | 가변 (렌더링 중 업데이트됨)       |
| 상태 보유      | ❌                                    | ✅ (`memoizedState`)              |
| 훅 정보        | ❌                                    | ✅ (훅 링크드 리스트)             |
| 트리 연결      | ❌                                    | ✅ (`child`, `sibling`, `return`) |
| 생명주기       | 렌더마다 새로 생성·폐기               | 재사용 (alternate로 보존)         |

### 3.3 렌더링 (Rendering)

React에서 "렌더링"은 **컴포넌트 함수 호출부터 VDOM 재조정 완료까지**를 의미한다.

```
[렌더링에 포함되는 과정]
  1. 컴포넌트 함수 호출 → React 엘리먼트 반환
  2. React 엘리먼트 → Fiber 노드와 비교 (Diffing)
  3. 변경 사항을 Fiber의 flags에 표시

[렌더링에 포함되지 않는 과정]
  4. DOM에 변경 사항 반영 (커밋 페이즈)
  5. 브라우저 화면 페인트
```

> ⚠️ **흔한 오해**: "화면에 그리는 것"이 렌더링이라고 생각하기 쉽지만,  
> React의 렌더링은 **메모리 내 VDOM 계산 단계**까지만을 의미한다.  
> 실제 DOM 적용은 커밋 페이즈, 화면 페인트는 브라우저의 영역이다.

### 3.4 Reconciliation (재조정)

**이전 VDOM과 새로운 React 엘리먼트를 비교하여 최소한의 변경 사항을 파악하는 과정**이다.

---

## 4. 전체 렌더링 과정

```
setState 호출
        │
        ▼
[Scheduler] 실행 타이밍 결정
        │
        ▼
[Reconciler] 렌더 페이즈 (Render Phase)
  1. 컴포넌트 함수 호출 → React 엘리먼트 반환
  2. React 엘리먼트와 기존 Fiber 비교 (Diffing)
  3. 변경 사항을 Fiber flags에 기록
        │
        ▼
[Reconciler + Renderer] 커밋 페이즈 (Commit Phase)
  4. 변경된 Fiber를 실제 DOM에 반영 (Mount/Update/Unmount)
        │
        ▼
[브라우저]
  5. 변경된 DOM을 화면에 페인트
```

## 5. React 18+과의 대응 개념

| React 16/17                                      | React 18+                  | 변화 요약                                   |
| ------------------------------------------------ | -------------------------- | ------------------------------------------- |
| 패키지 구조 (Core/Reconciler/Scheduler/Renderer) | 동일하게 유지              | 구조 변화 없음                              |
| React 엘리먼트 구조                              | 동일 + `jsx-runtime` 추가  | 새 JSX 변환으로 `import React` 불필요       |
| Fiber 구조                                       | `expirationTime` → `lanes` | 우선순위 필드 변경                          |
| 이벤트 위임 대상                                 | `document`                 | **루트 컨테이너** (React 17부터 변경)       |
| Reconciler 실행                                  | 동기 위주                  | Concurrent 기본 활성 (`createRoot` 사용 시) |
| `Legacy Event`                                   | 유지                       | React Native 등과 공유 구조 유지            |

# React 18+ — 패키지 구조와 핵심 용어 완전 정복

## 1. React의 패키지 구조를 알아야 하는 이유

React를 사용할 때 우리는 두 가지 패키지만 설치한다.

```bash
npm install react react-dom
```

하지만 내부적으로는 훨씬 복잡한 패키지 구조가 동작하고 있다.  
이 구조를 이해하면 **에러 메시지의 출처**, **성능 최적화의 원리**, **훅이 동작하는 이유**가 명확해진다.

---

## 2. 패키지별 역할과 책임

### 2.1 전체 구조 한눈에 보기

```
개발자 코드
  └─ import { useState } from 'react'
  └─ import ReactDOM from 'react-dom'

react (Core)
  └─ 컴포넌트 정의 API
  └─ 훅 인터페이스 (구현 없음)
  └─ JSX → React.createElement / jsx-runtime

react-reconciler (Reconciler)
  └─ VDOM 계산 (Diffing)
  └─ 훅 실제 구현 (mountState, updateState 등)
  └─ renderWithHooks, performConcurrentWorkOnRoot

scheduler (Scheduler)
  └─ Task 우선순위 관리
  └─ 브라우저 여유 시간에 Work 실행
  └─ MessageChannel 기반 비동기 실행

react-dom (Renderer)
  └─ 실제 DOM 노드 생성·수정·삭제
  └─ 이벤트 시스템 (합성 이벤트)
  └─ createRoot, ReactDOM.render
```

### 2.2 패키지 간 의존 관계

```
react-dom
  ├─ 의존: react-reconciler
  └─ 의존: 이벤트 시스템

react-reconciler
  ├─ 의존: scheduler
  └─ shared 패키지를 통해 react와 통신

react
  └─ 의존 없음 (플랫폼 독립)
```

`react` 패키지가 어떤 패키지에도 의존하지 않는 것이 **플랫폼 독립성**의 핵심이다.

---

## 3. React 18의 새로운 진입점: `createRoot`

React 17 이하:

```javascript
import ReactDOM from "react-dom";
ReactDOM.render(<App />, document.getElementById("root"));
// Legacy Mode: 항상 동기 렌더링
```

React 18:

```javascript
import { createRoot } from "react-dom/client";
const root = createRoot(document.getElementById("root"));
root.render(<App />);
// Concurrent Mode: 중단 가능한 렌더링, 자동 배칭 등 활성화
```

`createRoot`를 사용하면 Reconciler가 **Concurrent 모드**로 전환된다.  
같은 `react-reconciler`를 사용하지만, 내부적으로 처리 방식이 달라진다.

## 4. 핵심 용어 — React 18 기준 정의

### 4.1 React 엘리먼트

```jsx
// JSX 작성
<Button onClick={handleClick} disabled={false}>
  제출
</Button>

// React 18 새 변환 (jsx-runtime)
import { jsx as _jsx } from 'react/jsx-runtime';
_jsx(Button, {
  onClick: handleClick,
  disabled: false,
  children: '제출'
})

// 결과 객체 (React 엘리먼트)
{
  $$typeof: Symbol(react.element),
  type: Button,            // 함수 자체 (Custom 컴포넌트)
  key: null,
  ref: null,
  props: {
    onClick: handleClick,
    disabled: false,
    children: '제출',
  },
}
```

**React 엘리먼트는 렌더마다 새로 생성되는 일회성 객체이다.**  
상태나 라이프사이클 정보를 갖지 않는다.

### 4.2 Fiber — React 엘리먼트의 런타임 확장

Fiber는 React 엘리먼트를 처음 처리할 때 Reconciler가 생성하는 **지속적인 작업 단위 객체**이다.

```
React 엘리먼트 (일회성)        Fiber (지속적)
  type: Button            →    type: Button
  props: { ... }          →    memoizedProps: { ... }
  (상태 없음)              →    memoizedState: Hook1 → Hook2 (훅 링크드 리스트)
  (트리 연결 없음)          →    child, sibling, return (트리 포인터)
  (우선순위 없음)           →    lanes: DefaultLane
  (DOM 없음)               →    stateNode: <button> DOM 노드
```

**Fiber의 생명주기:**

```
최초 렌더링: React 엘리먼트 → Fiber 새로 생성 (mountState 등 실행)
리렌더링:   React 엘리먼트 → 기존 Fiber 재사용 (updateState 등 실행)
언마운트:   Fiber 제거 (cleanup 실행)
```

### 4.3 렌더링 (Rendering)

```
[렌더링 = 컴포넌트 호출 + VDOM 재조정]

setState 호출
  → Scheduler: Task 예약
  → Reconciler: 컴포넌트 함수 호출 → React 엘리먼트 반환
  → Reconciler: Fiber와 비교 (Diffing) → 변경 사항 파악
  → 여기까지가 렌더링

[렌더링이 아닌 것]
  → 실제 DOM 변경 (커밋 페이즈)
  → 브라우저 화면 페인트
  → useEffect 실행
```

### 4.4 Reconciliation (재조정)

이전 Fiber 트리와 새로운 React 엘리먼트를 비교하여 **최소한의 DOM 변경 사항을 계산**하는 과정이다.

```
이전 Fiber:  <ul> → <li>A</li> → <li>B</li> → <li>C</li>
새 엘리먼트: <ul> → <li>A</li> → <li>D</li> → <li>C</li>

Reconciler:
  A 비교 → 동일 → 변경 없음
  B vs D → 다름 → Update 플래그
  C 비교 → 동일 → 변경 없음

→ B → D 변경 1회만 실제 DOM에 적용
```

## 5. React 18에서 달라진 개념들

### 5.1 Concurrent 렌더링 — 중단 가능한 Reconciliation

```
[React 17 이하 - 동기 렌더링]
setState → Reconciliation 시작 → 완료까지 메인 스레드 독점
        → 사용자 입력 지연 가능

[React 18 - Concurrent 렌더링]
setState → Reconciliation 시작
        → 5ms 후 shouldYield → 중단
        → 브라우저: 사용자 입력 처리
        → Reconciliation 재개
        → 완료 → 커밋
```

### 5.2 Lane 기반 우선순위

React 18에서 Fiber의 `expirationTime`이 **`lanes`** 로 대체되었다.

```javascript
// React 16/17: 하나의 숫자로 우선순위 표현
fiber.expirationTime = 1073741771;

// React 18: 비트마스크로 복수 우선순위 표현
fiber.lanes = 0b000110; // DefaultLane + TransitionLane 동시 표현
```

### 5.3 이벤트 위임 대상 변경 (React 17부터)

```javascript
// React 16: document에 이벤트 위임
document.addEventListener("click", reactHandler);

// React 17+: 루트 컨테이너에 이벤트 위임
document.getElementById("root").addEventListener("click", reactHandler);
```

이 변경으로 한 페이지에서 여러 버전의 React를 동시에 사용할 수 있게 되었다.

| 패키지             | 역할                         | React 18 변화               |
| ------------------ | ---------------------------- | --------------------------- |
| `react`            | 컴포넌트 정의, 훅 인터페이스 | 새 JSX 변환(`jsx-runtime`)  |
| `react-dom`        | 브라우저 DOM 렌더링, 이벤트  | `createRoot` 추가           |
| `react-dom/client` | Concurrent 진입점            | **React 18 신규**           |
| `react-reconciler` | VDOM 계산, 훅 구현           | Lanes 모델, Concurrent 기본 |
| `scheduler`        | Task 실행 타이밍             | `needsPaint` 플래그 추가    |

---

## 7. 학습 로드맵

이 시리즈를 통해 아래 흐름을 코드 레벨까지 이해하는 것이 목표이다.

```
개발자: useState → setState 호출
        │
        ▼
react: dispatchSetState → update 객체 생성
        │
        ▼
react-reconciler: scheduleUpdateOnFiber
  → markUpdateLaneFromFiberToRoot (Lane 마킹)
  → ensureRootIsScheduled
        │
        ▼
scheduler: scheduleCallback
  → Task 객체 생성 → taskQueue 삽입
  → MessageChannel → performWorkUntilDeadline
        │
        ▼
react-reconciler: performConcurrentWorkOnRoot
  → renderWithHooks → 컴포넌트 호출
  → Diffing → flags 표시
        │
        ▼
react-dom: commitRoot
  → DOM 노드 생성·수정·삭제
  → useLayoutEffect 실행
        │
        ▼
브라우저: 화면 페인트 → useEffect 실행
```

### 핵심 체크리스트

- [ ] `react`, `react-reconciler`, `scheduler`, `react-dom` 각 패키지의 역할을 구분할 수 있다.
- [ ] React 엘리먼트는 불변 객체이며, Fiber는 상태와 라이프사이클을 가진 런타임 객체임을 이해했다.
- [ ] 렌더링 = 컴포넌트 호출 + VDOM 재조정 (DOM 적용·페인트는 렌더링이 아님)을 이해했다.
- [ ] `createRoot`가 Concurrent 모드를 활성화하는 진입점임을 이해했다.
- [ ] React 17부터 이벤트 위임 대상이 루트 컨테이너로 변경되었음을 이해했다.
