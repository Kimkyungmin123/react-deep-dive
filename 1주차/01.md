# React — Fiber 아키텍처 개요: Stack과의 결정적 차이

## 1. Fiber 아키텍처 등장 배경

### 1.1 React 15까지: Stack Reconciler

React 15 이하의 Reconciler는 **콜 스택(Call Stack)** 을 그대로 사용하는 방식으로 동작했다.

```
[컴포넌트 트리]
    App
    ├─ Header
    └─ Main
         ├─ List
         │   ├─ Item A
         │   ├─ Item B
         │   └─ Item C
         └─ Footer
```

Stack Reconciler가 이 트리를 처리하는 방식:

```
renderComponent(App)
  └─ renderComponent(Header)
  └─ renderComponent(Main)
       └─ renderComponent(List)
       │    └─ renderComponent(Item A)
       │    └─ renderComponent(Item B)  ← 여기서 멈출 수 없음
       │    └─ renderComponent(Item C)
       └─ renderComponent(Footer)
```

**Stack 자료구조의 LIFO 특성으로 인해:**

- 한 번 시작한 렌더링을 중간에 멈출 수 없다.
- 더 급한 업데이트(사용자 입력)가 오더라도 현재 렌더링이 완료될 때까지 기다려야 한다.
- 트리가 깊고 크면 메인 스레드를 수백ms 동안 독점할 수 있다.

### 1.2 React 16부터: Fiber Reconciler

Fiber 아키텍처는 React 16에서 도입되었다.  
Stack과의 **결정적 차이**: 처리 순서를 자유롭게 조정할 수 있다.

| 항목          | Stack Reconciler | Fiber Reconciler                  |
| ------------- | ---------------- | --------------------------------- |
| 자료구조      | 콜 스택 (암묵적) | 링크드 리스트 (명시적 Fiber 노드) |
| 작업 중단     | 불가             | 가능                              |
| 우선순위 조정 | 불가             | 가능                              |
| 작업 재시작   | 불가             | 가능                              |
| 처리 순서     | LIFO 고정        | 자유롭게 조정 가능                |

---

## 2. Fiber가 "꺼내는 순서를 바꿀 수 있다"는 것의 의미

React에서 "작업을 꺼낸다"는 것은 **컴포넌트를 렌더링하는 것**이다.

Stack 방식: A → B → C → D 순서로 삽입되면 D → C → B → A 순서로만 처리 가능

```
Stack 큐에 렌더링 작업 삽입
  push(renderA) → push(renderB) → push(renderC) → push(renderD)

처리 순서: D → C → B → A (LIFO, 변경 불가)
```

Fiber 방식: 우선순위에 따라 어떤 순서로든 처리 가능

```
Fiber 작업 큐 (우선순위 기반)
  renderA (NormalPriority)
  renderB (ImmediatePriority) ← 긴급
  renderC (NormalPriority)
  renderD (TransitionPriority) ← 낮음

처리 순서: B → A → C → D (우선순위 기반, 조정 가능)
```

> ⚠️ **"Fiber 구조가 스택과 유사하다"는 원문 설명에 대한 보완:**  
> Fiber가 스택 자료구조와 유사한 것은 **탐색 방식** 때문이다.  
> Fiber 트리를 순회할 때 `child → sibling → return`(부모로 복귀) 패턴이 DFS(깊이 우선 탐색)와 유사한 순서를 만들어낸다.  
> 그러나 Stack과의 핵심 차이는, Fiber는 이 순서를 **언제든 중단하고 재개할 수 있다**는 점이다.

---

## 3. Fiber 아키텍처가 가능하게 한 것들

Fiber는 단순한 자료구조 변경이 아니라, **렌더링의 패러다임 자체를 바꿨다**.

### 3.1 렌더링 중단과 재개 (Interruptible Rendering)

```
[5ms 단위 작업 분할]

Fiber 렌더링 시작
  ├─ renderComponent(Item A) → 2ms 경과
  ├─ renderComponent(Item B) → shouldYield() = true → 중단!
  │
  브라우저: 사용자 클릭 이벤트 처리
  │
  └─ renderComponent(Item B) 재개 → renderComponent(Item C) → 완료
```

### 3.2 우선순위 기반 렌더링

```
낮은 우선순위 렌더링 진행 중
  → 긴급한 업데이트 도착
  → 낮은 우선순위 렌더링 폐기
  → 긴급 업데이트 즉시 처리
  → 낮은 우선순위 렌더링 처음부터 재시작
```

### 3.3 Concurrent Mode (React 18에서 기본 활성화)

Fiber 아키텍처는 React 16에서 도입되었지만, Concurrent 기능들이 **본격 활용된 것은 React 18**부터이다.

| 버전       | 주요 변화                                                                                    |
| ---------- | -------------------------------------------------------------------------------------------- |
| React 16   | Fiber 아키텍처 도입 (내부 구조 재작성)                                                       |
| React 16.6 | `React.lazy`, `Suspense` 실험적 도입                                                         |
| React 17   | 이벤트 시스템 개선 (이벤트 위임 대상 변경)                                                   |
| React 18   | `createRoot` → Concurrent 기본 활성, `useTransition`, `useDeferredValue`, Automatic Batching |

## 4. `useState` → Fiber 아키텍처 연결 고리

이 시리즈가 `useState` 분석에서 시작하는 이유:

```
개발자가 가장 익숙한 API
        │
        ▼
useState → setState 호출
        │ (어떻게 상태가 변경되는가?)
        ▼
react 패키지 → shared 패키지 → reconciler 패키지
        │ (Dispatcher 패턴)
        ▼
dispatchSetState → update 객체 생성 → 큐에 저장
        │ (어떻게 화면이 다시 그려지는가?)
        ▼
scheduleUpdateOnFiber → Scheduler → Fiber 렌더링
        │ (Fiber 아키텍처 전체가 동작)
        ▼
Fiber 트리 재조정 → 커밋 → DOM 업데이트
```

`useState` 하나를 끝까지 추적하면 React의 **전체 패키지 구조와 Fiber 아키텍처**를 모두 이해할 수 있다.

## 5. 주요 패키지 상호작용 개요

```
[시리즈에서 다루는 전체 흐름]

useState (react 패키지)
  └─ Dispatcher 패턴 → Reconciler로 주입
        │
        ▼
dispatchSetState (react-reconciler)
  └─ Lane 마킹 → ensureRootIsScheduled
        │
        ▼
scheduleCallback (scheduler)
  └─ Task 생성 → 우선순위 큐
  └─ MessageChannel → performWorkUntilDeadline
        │
        ▼
performConcurrentWorkOnRoot (react-reconciler)
  └─ renderWithHooks → 컴포넌트 호출
  └─ Diffing → flags 표시
        │
        ▼
commitRoot (react-dom)
  └─ DOM 조작 → 라이프사이클 실행
```

## 6. React 18+과의 대응 개념

| React 16/17                   | React 18+                           | 변화 요약                           |
| ----------------------------- | ----------------------------------- | ----------------------------------- |
| Fiber 아키텍처 (도입)         | Fiber 아키텍처 (완전 활용)          | 구조 동일, 활용도 극대화            |
| Concurrent Mode (실험적)      | **기본 동작** (`createRoot`)        | opt-in → 기본값                     |
| 렌더링 중단 (내부적으로 가능) | `startTransition`으로 제어 가능     | 개발자에게 공개 API 제공            |
| 없음                          | `useTransition`, `useDeferredValue` | Fiber 우선순위 조정을 위한 공개 API |
| `expirationTime` 우선순위     | **Lanes** 비트마스크                | 더 정밀한 우선순위 표현             |

# React 18+ — Fiber 아키텍처와 Concurrent 렌더링의 완성

## 1. Fiber 아키텍처가 React 18에서 "완성"된 이유

Fiber는 React 16에서 내부 아키텍처로 도입되었지만, 개발자가 그 힘을 직접 사용할 수 있게 된 것은 React 18부터이다.

```
React 15: Stack Reconciler
  → 중단 불가 동기 렌더링만 가능

React 16: Fiber Reconciler 도입
  → 내부적으로 중단 가능해짐
  → 그러나 개발자에게 공개된 Concurrent API 없음

React 18: Fiber 위에 구축된 Concurrent API 공개
  → createRoot, useTransition, useDeferredValue
  → Automatic Batching, Suspense for data fetching
```

---

## 2. Fiber가 가능하게 한 세 가지 핵심 능력

### 2.1 중단 (Interrupt)

```jsx
function HeavyList({ items }) {
  // 이 컴포넌트가 렌더링 중일 때 사용자가 입력하면?
  return items.map((item) => <HeavyItem key={item.id} data={item} />);
}
```

```
[Stack 방식 - React 15]
HeavyList 렌더링 시작 → 완료될 때까지 모든 입력 무시
  → 수백ms 동안 화면 멈춤

[Fiber 방식 - React 18]
HeavyList 렌더링 중 → shouldYield() = true → 중단
브라우저: 사용자 입력 처리 → 입력창 즉시 반응
HeavyList 렌더링 재개 → 완료
```

### 2.2 우선순위 조정 (Prioritization)

```jsx
function SearchPage() {
  const [input, setInput] = useState("");
  const [results, setResults] = useState([]);

  function handleChange(e) {
    setInput(e.target.value); // DefaultLane: 즉시 처리
    startTransition(() => {
      setResults(search(e.target.value)); // TransitionLane: 여유있게 처리
    });
  }
}
```

`startTransition` 없이는 두 업데이트가 동일한 우선순위로 처리되어 입력 지연이 발생할 수 있다.  
Fiber의 Lane 시스템이 이 우선순위 구분을 가능하게 한다.

### 2.3 재사용 (Reuse)

```
낮은 우선순위 렌더링 진행 중 (Fiber 트리 계산 중)
  → 새 긴급 업데이트 도착
  → 현재 WIP Fiber 트리 폐기
  → 긴급 업데이트로 새 WIP 생성 → 커밋

다음 렌더링:
  → 이전 Current Fiber를 기반으로 새 WIP 생성
  (Current Fiber가 살아있기 때문에 처음부터 재시작 가능)
```

## 3. React 18 Concurrent API — Fiber 우선순위 조정 도구

### 3.1 `useTransition` — 업데이트를 비긴급으로 표시

```jsx
import { useTransition } from "react";

function TabBar() {
  const [isPending, startTransition] = useTransition();
  const [activeTab, setActiveTab] = useState("home");

  function switchTab(tab) {
    startTransition(() => {
      setActiveTab(tab); // TransitionLane 배정
    });
  }

  return (
    <>
      <button onClick={() => switchTab("feed")}>Feed</button>
      <button onClick={() => switchTab("profile")}>Profile</button>

      {/* Fiber가 TransitionLane 처리 중인 동안 isPending = true */}
      {isPending && <Spinner />}

      <TabContent tab={activeTab} />
    </>
  );
}
```

**`isPending`의 정체**: TransitionLane의 Fiber 렌더링이 5ms 단위로 분할 처리되는 동안 `true`를 유지한다.

### 3.2 `useDeferredValue` — 값 기반 후순위 처리

```jsx
import { useDeferredValue } from "react";

function SearchResults({ query }) {
  // query가 바뀌어도 deferredQuery는 여유 있을 때 업데이트
  const deferredQuery = useDeferredValue(query);

  // deferredQuery 기반으로 렌더링 → TransitionLane
  return <HeavyResultList filter={deferredQuery} />;
}

function SearchPage() {
  const [query, setQuery] = useState("");

  return (
    <>
      <input onChange={(e) => setQuery(e.target.value)} /> {/* 즉시 반응 */}
      <SearchResults query={query} /> {/* 지연 처리 */}
    </>
  );
}
```

| API                | 제어 대상                 | 주요 사용 시점                        |
| ------------------ | ------------------------- | ------------------------------------- |
| `startTransition`  | setState 호출             | 이벤트 핸들러 내 무거운 상태 업데이트 |
| `useTransition`    | setState 호출 + 로딩 상태 | 탭 전환, 페이지 이동                  |
| `useDeferredValue` | 특정 값                   | 외부 props 기반 필터링·검색           |

## 4. Automatic Batching — Fiber 덕분에 가능해진 것

React 18은 어디서든 여러 `setState`를 하나의 렌더링으로 묶는다.

```jsx
// React 17: setTimeout 내부에서 2번 렌더링
setTimeout(() => {
  setA(1); // 렌더링 1
  setB(2); // 렌더링 2
}, 1000);

// React 18 (createRoot): 1번 렌더링
setTimeout(() => {
  setA(1);
  setB(2);
  // Fiber: 두 업데이트 모두 DefaultLane에 배정
  // ensureRootIsScheduled: 동일 Lane → 재사용 → 렌더링 1회
}, 1000);
```

배칭 해제가 필요할 때:

```jsx
import { flushSync } from "react-dom";

flushSync(() => setA(1)); // 즉시 DOM 업데이트
console.log(ref.current.textContent); // 최신값 읽기 가능
flushSync(() => setB(2));
```

## 5. Fiber 학습의 전체 로드맵

Fiber 아키텍처를 이해하기 위한 `useState` 추적 여정:

```
[1] useState 인터페이스 (react 패키지)
    → Dispatcher 패턴으로 Reconciler의 구현을 주입받음

[2] mountState / updateState (react-reconciler)
    → Fiber.memoizedState에 훅 링크드 리스트 연결
    → queue.pending에 업데이트 객체 저장

[3] dispatchSetState (react-reconciler)
    → Lane 배정 → eagerState 최적화 → scheduleUpdateOnFiber

[4] ensureRootIsScheduled (react-reconciler)
    → Lane 비교 → Scheduler에 Task 등록

[5] workLoop (scheduler)
    → 5ms 단위 실행 → shouldYield → MessageChannel 반복

[6] performConcurrentWorkOnRoot (react-reconciler)
    → renderWithHooks → 컴포넌트 호출 → Diffing
    → flags 표시 → continuationCallback 반환(미완성 시)

[7] commitRoot (react-dom)
    → DOM 변경 → Current ↔ WIP 포인터 교체
    → useLayoutEffect → useEffect 예약
```

각 단계가 Fiber 아키텍처의 한 측면을 담당하며, 전체를 추적하면 React 18의 Concurrent 렌더링이 어떻게 작동하는지 완전히 이해할 수 있다.

## 6. Stack vs Fiber — 개발자 관점에서의 차이

| 개발 시나리오            | Stack (React 15) | Fiber (React 18)                    |
| ------------------------ | ---------------- | ----------------------------------- |
| 무거운 목록 렌더 중 입력 | ❌ 입력 지연     | ✅ `startTransition` → 즉시 반응    |
| 여러 setState 동시 호출  | 각각 렌더링      | ✅ Automatic Batching → 1번 렌더    |
| 데이터 로딩 중 UI        | 별도 처리 필요   | ✅ `Suspense` + `use()`             |
| 렌더링 진행 상태 표시    | 구현 복잡        | ✅ `isPending` (`useTransition`)    |
| 탭 전환 시 화면 유지     | 불가             | ✅ `startTransition` + 기존 UI 유지 |

### 핵심 체크리스트

- [ ] Stack Reconciler의 LIFO 구조 때문에 렌더링을 중단할 수 없었음을 이해했다.
- [ ] Fiber는 처리 순서를 자유롭게 조정할 수 있어 중단·재개·우선순위 조정이 가능함을 이해했다.
- [ ] Fiber는 React 16에서 도입되었지만, Concurrent API(`useTransition` 등)는 React 18부터 공개됨을 이해했다.
- [ ] `useTransition`의 `isPending`이 TransitionLane Fiber 처리 중 `true`임을 이해했다.
- [ ] Automatic Batching이 Fiber Lane 시스템 덕분에 가능함을 이해했다.
- [ ] `useState` 하나를 추적하면 Fiber 아키텍처 전체가 연결되어 있음을 이해했다.
