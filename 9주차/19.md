## 1. 왜 메인 스레드를 양보해야 하는가?

### 브라우저 메인 스레드의 특성

브라우저(Chrome 기준)는 멀티 프로세스 아키텍처로 동작한다. 그 중 **렌더러 프로세스**의 **메인 스레드**는 다음 모든 작업을 단일 스레드에서 처리한다.

- HTML 파싱 및 DOM 트리 구축
- JavaScript 실행
- 스타일 계산 및 레이아웃(Reflow)
- 페인트(Paint)
- 사용자 이벤트 처리 (클릭, 키보드 입력 등)

메인 스레드는 하나이기 때문에, **JavaScript가 실행 중이면 페인트나 사용자 이벤트 처리는 대기 상태**가 된다. 콜 스택이 비워지기 전까지 다른 어떤 작업도 실행될 수 없다.

### 블로킹 문제

```
[복잡한 컴포넌트 렌더링 JS 실행] ─────────────────────────────→ 완료
          ↑ 이 구간 동안 사용자 키 입력, 마우스 이벤트, 화면 갱신 모두 대기
```

React 15까지는 렌더링 과정이 **단일 재귀 호출**로 이루어졌다. 한 번 시작된 렌더링은 끝날 때까지 메인 스레드를 독점했고, 그 사이 사용자 입력이 쌓여도 브라우저는 처리하지 못했다.

### React Fiber와 양보 개념의 등장

React 16에서 도입된 **Fiber 아키텍처**의 핵심 목표 중 하나는 렌더링 작업을 잘게 쪼개어 브라우저에게 주기적으로 제어권을 돌려주는 것이다. 이를 **메인 스레드 양보(yielding)**라고 한다.

> 💡 **양보(yield)**: React가 진행 중인 렌더링 작업을 일시 중단하고, 브라우저가 페인트나 사용자 이벤트 처리 등 더 긴급한 작업을 먼저 수행할 수 있도록 메인 스레드의 제어권을 넘기는 행위

## 2. isInputPending API란?

### 등장 배경

React가 양보 타이밍을 결정하려면 "지금 브라우저가 처리해야 할 긴급한 작업이 있는가?"를 알아야 한다. 하지만 기존에는 JavaScript에서 브라우저의 **미처리 사용자 입력 유무**를 알 수 없었다.

`navigator.scheduling.isInputPending()` API는 **Facebook이 W3C에 제안하고 기여한 브라우저 API**로, 이 문제를 해결한다.

### 동작 방식

```javascript
// 현재 처리되지 않은 사용자 입력(클릭, 키보드, 마우스 등)이 있으면 true 반환
const hasPendingInput = navigator.scheduling.isInputPending();
```

- `true`: 브라우저 이벤트 큐에 아직 처리되지 않은 사용자 입력이 존재한다.
- `false`: 현재 대기 중인 사용자 입력이 없다.

이 API 덕분에 React는 JavaScript 실행 중에도 "지금 사용자가 입력하고 있는가?"를 능동적으로 확인할 수 있게 되었다.

> ⚠️ **브라우저 지원**: Chrome 87+에서 공식 지원. 지원하지 않는 환경에서는 폴백 로직이 동작한다.

## 3. shouldYieldToHost 함수 분석

### 함수의 역할

`shouldYieldToHost`는 React 스케줄러가 **"지금 브라우저에게 메인 스레드를 양보해야 하는가?"**를 판단하는 핵심 함수다.

### Case 1: isInputPending API를 지원하는 경우

```javascript
// React 16.x 소스 (packages/scheduler/src/forks/SchedulerHostConfig.default.js)
if (
  enableIsInputPending &&
  navigator?.scheduling?.isInputPending !== undefined
) {
  const scheduling = navigator.scheduling;

  shouldYieldToHost = function () {
    const currentTime = getCurrentTime();

    if (currentTime >= deadline) {
      // 데드라인 초과 → 페인트 대기 또는 사용자 입력 대기 시 즉시 양보
      if (needsPaint || scheduling.isInputPending()) {
        return true; // 양보!
      }
      // 페인트도 없고 입력도 없다면, 최대 허용 시간(maxYieldInterval)까지는 계속 작업
      return currentTime >= maxYieldInterval;
    } else {
      // 데드라인 미초과 → 계속 작업
      return false;
    }
  };
}
```

**양보 조건 정리:**

| 조건                                                            | 결과                            |
| --------------------------------------------------------------- | ------------------------------- |
| 현재 시간 < 데드라인                                            | 양보 안 함 (작업 계속)          |
| 현재 시간 ≥ 데드라인 AND (`needsPaint` 또는 `isInputPending()`) | **양보**                        |
| 현재 시간 ≥ 데드라인 AND 둘 다 없음                             | 최대 허용 시간까지 작업 후 결정 |

- **데드라인**: `currentTime + yieldInterval` (기본값 **5ms**)
- **needsPaint**: `requestPaint()` 호출 시 `true`로 설정됨 (커밋 페이즈에서 호출)
- **maxYieldInterval**: 300ms (일부 버전에서 150ms로 조정)

### Case 2: isInputPending API를 지원하지 않는 경우 (폴백)

```javascript
// isInputPending 미지원 환경
shouldYieldToHost = function () {
  return getCurrentTime() >= deadline;
};
```

단순히 **데드라인 초과 여부만**으로 양보를 결정한다. 매 프레임마다 양보하게 되므로 덜 효율적이다.

### yieldInterval과 FPS

기본 `yieldInterval`은 **5ms**이며, `forceFrameRate(fps)` 함수로 조정할 수 있다.

```javascript
forceFrameRate = function (fps) {
  if (fps > 0) {
    yieldInterval = Math.floor(1000 / fps); // 예: 60fps → 약 16ms
  } else {
    yieldInterval = 5; // 기본값 리셋
  }
};
```

> ⚠️ **주의**: `forceFrameRate`는 내부 실험용 API로, 일반 애플리케이션 코드에서 직접 호출하는 것은 지양해야 한다.

## 4. shouldYieldToHost가 호출되는 시점

### 스케줄러의 workLoop (커밋 페이즈 포함)

스케줄러 내부 `workLoop` 함수에서 `shouldYieldToHost`를 직접 호출한다.

```javascript
// Scheduler.js - workLoop
function workLoop(hasTimeRemaining, initialTime) {
  // ...
  while (
    currentTask !== null &&
    !(enableSchedulerDebugging && isSchedulerPaused)
  ) {
    if (currentTask.expirationTime > currentTime && shouldYieldToHost()) {
      // 데드라인이 지나지 않았지만 양보가 필요하면 중단
      break;
    }
    // 태스크 실행
    const callback = currentTask.callback;
    // ...
  }
}
```

### 리컨실러의 workLoopConcurrent (렌더 페이즈)

리컨실러(Reconciler)는 `unstable_shouldYield`(= 스케줄러의 `shouldYieldToHost` 래퍼)를 통해 렌더 페이즈를 중단할지 결정한다.

```javascript
// ReactFiberWorkLoop.js
function workLoopConcurrent() {
  while (workInProgress !== null && !shouldYield()) {
    performUnitOfWork(workInProgress);
  }
}
```

- `shouldYield()`가 `true`를 반환하면 while 루프가 멈추고 렌더링이 중단된다.
- 중단 시 `performConcurrentWorkOnRoot`는 **자기 자신의 바인딩을 반환**하여 스케줄러가 나중에 재개할 수 있도록 한다.

### workLoopSync vs workLoopConcurrent 비교

| 함수                 | shouldYield 체크 | 중단 가능 | 사용 시점                  |
| -------------------- | ---------------- | --------- | -------------------------- |
| `workLoopSync`       | ❌ 없음          | 불가      | Legacy 모드, 긴급 업데이트 |
| `workLoopConcurrent` | ✅ 있음          | 가능      | Concurrent 모드            |

> **React 16 주의**: React 16에서 Concurrent 모드는 실험적(opt-in)이었다. 기본 모드(`ReactDOM.render`)는 `workLoopSync`를 사용하므로 양보가 발생하지 않는다.

## 5. React 16 vs React 18+ 핵심 비교

### 스케줄러 파일 구조 변경

| 항목                | React 16                         | React 18+             |
| ------------------- | -------------------------------- | --------------------- |
| 스케줄러 설정 파일  | `SchedulerHostConfig.default.js` | `Scheduler.js`로 통합 |
| isInputPending 사용 | ✅ 조건부 사용                   | ✅ 유지 (동일 로직)   |
| `workLoop` 위치     | Scheduler 내부                   | Scheduler 내부 (동일) |

> React 18에서는 `SchedulerHostConfig.default.js`가 `Scheduler.js`에 통합되었다. 핵심 로직 자체는 유사하게 유지된다.

### Concurrent Mode의 전환

```
React 16: Concurrent Mode = 실험적 opt-in (ReactDOM.unstable_createRoot)
React 18: Concurrent Mode = 기본값 (ReactDOM.createRoot)
```

React 18부터 `ReactDOM.createRoot()`를 사용하면 모든 업데이트가 기본적으로 Concurrent 렌더링 경로를 타게 된다. 즉, `workLoopConcurrent`와 `shouldYield()` 기반의 양보 메커니즘이 **기본으로 활성화**된다.

### startTransition / useTransition (React 18 신규)

React 18에서는 **개발자가 직접 업데이트 우선순위를 선언**할 수 있는 API가 추가되었다.

```javascript
import { startTransition, useTransition } from "react";

// 긴급하지 않은 업데이트를 Transition으로 표시
startTransition(() => {
  setSearchResults(newResults); // 이 업데이트는 낮은 우선순위로 처리
});
```

- **Urgent 업데이트** (타이핑, 클릭 등): 즉시 처리
- **Transition 업데이트** (`startTransition` 래핑): 양보 가능, 중단 가능

스케줄러 레벨에서는 Transition이 `NormalPriority` (timeout: 5000ms)로 스케줄링되며, Urgent 업데이트(`UserBlockingPriority`, timeout: 250ms)가 들어오면 중단된다.

### 자동 배칭 (Automatic Batching, React 18 신규)

React 18 이전에는 이벤트 핸들러 내부의 `setState` 호출만 배칭되었다. React 18부터는 `setTimeout`, Promise, 네이티브 이벤트 핸들러 내부의 호출도 자동으로 배칭된다. 이는 불필요한 재렌더링을 줄여 스케줄러의 부담을 낮춘다.

### unstable_scheduleCallback 우선순위 시스템

React 18에서도 Scheduler의 우선순위 시스템은 유지된다.

```javascript
// 우선순위별 timeout (React 18 기준)
ImmediatePriority:    timeout = -1        // 즉시 실행
UserBlockingPriority: timeout = 250ms     // 사용자 인터랙션
NormalPriority:       timeout = 5000ms    // 일반 렌더링
LowPriority:          timeout = 10000ms   // 낮은 우선순위
IdlePriority:         timeout = 최대값    // 유휴 시간에만
```

## 전체 흐름 요약

```
사용자 인터랙션 or setState 호출
        ↓
ensureRootIsScheduled() → scheduleCallback(priority, performConcurrentWorkOnRoot)
        ↓
MessageChannel → performWorkUntilDeadline()
        ↓
workLoop() [Scheduler]
  ├─ shouldYieldToHost() 체크
  │    ├─ currentTime < deadline → 작업 계속
  │    └─ deadline 초과 AND (needsPaint or isInputPending) → 양보
        ↓
performConcurrentWorkOnRoot()
        ↓
workLoopConcurrent() [Reconciler]
  └─ 매 Fiber 단위로 shouldYield() 체크
       └─ true → 렌더 중단, 자기 자신을 반환(continuation)
        ↓
MessageChannel → 다음 틱에서 재개
```

## 1. 문제: 왜 브라우저가 멈추는가?

### 브라우저의 메인 스레드

브라우저는 화면 그리기, JavaScript 실행, 사용자 이벤트 처리를 모두 **단 하나의 스레드(메인 스레드)**에서 처리한다.

```
메인 스레드가 처리하는 일들:
┌─────────────────────────────────────────────────────────┐
│  JavaScript 실행 | DOM 조작 | 화면 그리기 | 이벤트 처리  │
└─────────────────────────────────────────────────────────┘
              ↑ 이 모든 것을 순차적으로, 혼자서!
```

문제는 **JavaScript가 실행되는 동안 다른 모든 작업이 대기**해야 한다는 점이다.

### 실제 문제 상황

예를 들어, 수천 개의 아이템을 렌더링하는 컴포넌트가 있다고 하자.

```javascript
// 렌더링하는 데 200ms가 걸리는 컴포넌트
function HeavyList({ items }) {
  return (
    <ul>
      {items.map((item) => (
        <ComplexItem key={item.id} data={item} />
      ))}
    </ul>
  );
}
```

이 렌더링이 진행되는 200ms 동안, 사용자가 검색창에 타이핑해도 글자가 나타나지 않는다. 브라우저가 React의 JavaScript 실행에 묶여있기 때문이다.

## 2. 해결책: 작업을 쪼개고 양보한다

### React 18의 접근 방식

React 18은 큰 렌더링 작업을 **작은 단위(Fiber)**로 쪼갠 뒤, 중간중간 브라우저에게 "잠깐 먼저 해"라고 제어권을 넘긴다. 이를 **메인 스레드 양보(yielding)**라고 한다.

```
React 18의 렌더링 방식:
[렌더 일부] → 브라우저에 양보 → [이벤트 처리] → [렌더 재개] → 브라우저에 양보 → [렌더 완료]
```

### 양보는 어떻게 이루어지는가?

React는 브라우저의 이벤트 루프를 활용한다. **MessageChannel**을 사용해 현재 JavaScript 실행을 끊고, 새로운 매크로태스크(macrotask)로 작업을 재예약한다.

```javascript
// React 18 Scheduler 내부 (단순화된 버전)
const channel = new MessageChannel();
channel.port1.onmessage = performWorkUntilDeadline; // 작업 재개 지점

function scheduleWork() {
  channel.port2.postMessage(null); // 현재 실행 끊고, 다음 틱에 재개 예약
}
```

`MessageChannel`을 쓰는 이유는 `setTimeout(fn, 0)`이 최소 4ms 지연이 있는 반면, `MessageChannel`은 지연 없이 다음 매크로태스크를 예약할 수 있기 때문이다.

## 3. 스케줄러: 작업 우선순위 관리자

### 스케줄러의 역할

React 18의 **Scheduler**는 "어떤 작업을 언제 실행할지"를 관리하는 내부 모듈이다. 모든 React 업데이트는 Scheduler를 통해 우선순위가 부여된 태스크로 등록된다.

### 우선순위 종류

```
ImmediatePriority    (-1ms)   → 즉시 실행 (동기)
UserBlockingPriority (250ms)  → 사용자 입력 관련 (타이핑, 클릭)
NormalPriority       (5000ms) → 일반 업데이트
LowPriority          (10000ms)→ 낮은 우선순위 업데이트
IdlePriority         (무제한) → 브라우저가 한가할 때만
```

숫자는 **만료 시간(timeout)**이다. 빨리 만료될수록 먼저 처리된다.

### 태스크 큐 구조

Scheduler는 내부적으로 **min-heap(최소 힙)** 기반의 우선순위 큐를 사용한다. 만료 시간이 가장 짧은(= 가장 긴급한) 태스크가 항상 앞에 온다.

```
태스크 큐 예시:
┌──────────────────────────────────────────┐
│ [즉시] 버튼 클릭 처리                     │  ← 가장 먼저 실행
│ [250ms] 텍스트 입력 반영                  │
│ [5000ms] 검색 결과 목록 렌더링            │
│ [10000ms] 추천 섹션 렌더링                │
└──────────────────────────────────────────┘
```

---

## 4. shouldYieldToHost: 언제 양보할지 결정

### 핵심 함수

`shouldYieldToHost`는 Scheduler가 "지금 브라우저에게 양보할 타이밍인가?"를 판단하는 함수다.

### 5ms 데드라인

React 18 스케줄러는 기본적으로 **5ms**마다 양보를 고려한다.

```javascript
// 단순화된 shouldYieldToHost (React 18 기준)
function shouldYieldToHost() {
  const currentTime = getCurrentTime(); // performance.now()

  if (currentTime >= deadline) {
    // 데드라인(시작 시간 + 5ms) 초과
    if (needsPaint || navigator.scheduling?.isInputPending()) {
      return true; // 화면 그리기 또는 사용자 입력 대기 중 → 양보!
    }
    // 둘 다 없으면 최대 허용 시간까지는 계속 작업
    return currentTime >= maxYieldInterval;
  }
  return false; // 아직 데드라인 전 → 계속 작업
}
```

### isInputPending API

`navigator.scheduling.isInputPending()`은 브라우저 이벤트 큐에 **처리되지 않은 사용자 입력이 있는지** 확인하는 API다.

```
브라우저 이벤트 큐에 키보드 입력 대기 중?
     ↓ isInputPending() → true
React: "사용자가 타이핑하고 있구나, 렌더링 잠깐 멈추고 양보!"
```

- Facebook이 제안하여 브라우저 표준이 된 API
- Chrome 87+에서 지원
- 지원하지 않는 환경에서는 단순히 데드라인 초과 시 양보

### 양보 흐름 전체 그림

```
workLoopConcurrent() 실행 중...
    ↓
각 Fiber 처리 후 shouldYield() 체크
    ↓
shouldYieldToHost() 호출
    ├─ 데드라인 전 → false → 다음 Fiber 계속 처리
    └─ 데드라인 초과 + (needsPaint or isInputPending)
         → true → while 루프 중단
              ↓
         MessageChannel로 다음 틱에 재개 예약
              ↓
         브라우저: 화면 그리기, 이벤트 처리
              ↓
         React: 중단된 지점부터 렌더링 재개
```

## 5. React 18의 Concurrent 기능과 양보의 관계

### Concurrent Mode가 기본값

React 17까지는 `ReactDOM.render()`가 기본이었고, Concurrent 모드는 실험적이었다.

```javascript
// React 17 이하 (동기 렌더링, 양보 없음)
ReactDOM.render(<App />, document.getElementById("root"));

// React 18 (Concurrent 모드, 양보 있음)
const root = ReactDOM.createRoot(document.getElementById("root"));
root.render(<App />);
```

`createRoot()`를 사용하는 순간, React의 모든 렌더링이 `workLoopConcurrent`를 통해 이루어지고 양보 메커니즘이 활성화된다.

### startTransition / useTransition

React 18의 가장 중요한 신규 기능 중 하나다. **개발자가 직접 업데이트의 긴급도를 선언**할 수 있다.

```javascript
import { startTransition, useTransition } from "react";

function SearchComponent() {
  const [query, setQuery] = useState("");
  const [results, setResults] = useState([]);
  const [isPending, startTransition] = useTransition();

  function handleChange(e) {
    // 타이핑은 긴급: 즉시 처리 (UserBlockingPriority)
    setQuery(e.target.value);

    // 검색 결과 렌더링은 비긴급: 양보 가능, 중단 가능
    startTransition(() => {
      setResults(search(e.target.value));
    });
  }

  return (
    <>
      <input value={query} onChange={handleChange} />
      {isPending ? <Spinner /> : <ResultList results={results} />}
    </>
  );
}
```

**startTransition이 없었다면**: 검색 결과를 렌더링하는 동안 타이핑이 끊겼을 것이다.  
**startTransition 사용 후**: 새 타이핑이 들어오면 검색 결과 렌더링을 중단하고, 타이핑을 먼저 반영한다.

### useDeferredValue

값의 업데이트를 의도적으로 지연시켜, 무거운 렌더링이 빠른 UI 업데이트를 방해하지 않도록 한다.

```javascript
import { useDeferredValue } from "react";

function App() {
  const [query, setQuery] = useState("");
  const deferredQuery = useDeferredValue(query); // 렌더링 부담 없는 시점에 업데이트

  return (
    <>
      <input value={query} onChange={(e) => setQuery(e.target.value)} />
      <HeavyList filter={deferredQuery} /> {/* 지연된 값으로 렌더링 */}
    </>
  );
}
```

### 자동 배칭 (Automatic Batching)

React 18부터는 어디서든 여러 `setState` 호출이 자동으로 하나의 렌더링으로 묶인다. 불필요한 렌더링 횟수가 줄어 스케줄러의 부담도 감소한다.

```javascript
// React 17: setTimeout 안에서 각각 렌더링 발생 (2번)
setTimeout(() => {
  setCount((c) => c + 1);
  setFlag((f) => !f);
}, 100);

// React 18: 자동 배칭으로 1번만 렌더링
setTimeout(() => {
  setCount((c) => c + 1); // 배칭
  setFlag((f) => !f); // 배칭 → 한 번에 렌더링
}, 100);
```

## 6. 개발자 관점: 내가 할 수 있는 것

### createRoot 사용하기

React 18의 모든 성능 이점을 활용하려면 반드시 `createRoot`를 써야 한다.

```javascript
// ✅ 권장
import { createRoot } from "react-dom/client";
const root = createRoot(document.getElementById("root"));
root.render(<App />);
```

### startTransition으로 우선순위 명시하기

무거운 렌더링과 빠른 UI 업데이트가 함께 있을 때 사용한다.

```javascript
// 무거운 필터링/정렬 결과 렌더링을 Transition으로 표시
startTransition(() => {
  setSortedList(heavySort(data));
});
```

### 언제 쓰면 좋은가?

| 상황                        | 권장 방법                                 |
| --------------------------- | ----------------------------------------- |
| 검색 입력 + 결과 렌더링     | `startTransition`                         |
| 무거운 목록 필터링          | `startTransition` 또는 `useDeferredValue` |
| 탭 전환 시 로딩             | `startTransition` + `Suspense`            |
| 컴포넌트 자체가 너무 무거움 | 가상화(react-window) 또는 컴포넌트 분리   |

## 핵심 개념 요약

| 개념                   | 설명                                                        |
| ---------------------- | ----------------------------------------------------------- |
| **Fiber**              | React의 렌더링 작업 최소 단위. 쪼개고 재개할 수 있다.       |
| **Scheduler**          | 어떤 Fiber 작업을 언제 실행할지 우선순위로 관리한다.        |
| **shouldYieldToHost**  | 지금 브라우저에 양보할 타이밍인지 판단한다. (5ms 단위)      |
| **isInputPending**     | 브라우저에 처리 안 된 사용자 입력이 있는지 감지한다.        |
| **MessageChannel**     | 렌더링을 끊고 다음 틱에 재개하기 위한 메커니즘.             |
| **workLoopConcurrent** | 양보를 지원하는 렌더 루프. `shouldYield()`를 매번 체크한다. |
| **startTransition**    | 개발자가 "이 업데이트는 비긴급"이라고 선언하는 API.         |
| **useDeferredValue**   | 값 업데이트를 여유 있을 때 처리하도록 지연하는 훅.          |
