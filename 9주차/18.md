# React Scheduler — 브라우저에게 스레드를 양보하는 원리

## 1. Work와 렌더링의 차이

Scheduler에서 말하는 **Work(워크)** 는 React의 재조정(Reconciliation) 작업이다.

| 구분                | 설명                                             |
| ------------------- | ------------------------------------------------ |
| **Work**            | VDOM 계산, Fiber 트리 업데이트 — **메모리 작업** |
| **브라우저 렌더링** | Layout → Paint → Composite — **브라우저 작업**   |

Work는 순수한 자바스크립트 연산이므로 **브라우저 렌더링 파이프라인과 무관**하다.  
단, Work가 메인 스레드를 너무 오래 점유하면 브라우저가 사용자 이벤트나 화면 갱신을 처리하지 못한다.

---

## 2. Yield Interval — 왜 5ms인가?

```
브라우저 한 프레임 = 16.67ms (60Hz 기준)

[브라우저 프레임 예산 분배]
  사용자 이벤트 처리   ─┐
  JS 실행              ─┤ → 이 구간 안에서 React Work 수행
  스타일 계산          ─┤
  Layout               ─┤
  Paint                ─┘

React Work 허용 시간: 약 5ms (Yield Interval)
```

React가 5ms를 기준으로 양보하는 이유:

1. **V-Sync 불필요**: Work는 메모리 작업이므로 16.67ms 프레임에 딱 맞출 필요가 없다.
2. **이벤트 응답성**: 5ms마다 브라우저에 제어권을 돌려주면 사용자 입력이 지연 없이 처리된다.
3. **보수적 설계**: 16.67ms를 모두 쓰면 브라우저의 Layout·Paint를 위한 여유가 없어진다.

> ⚠️ **5ms는 고정값이 아니다.** 브라우저 환경과 React 버전에 따라 달라질 수 있으며, React 18에서는 `frameInterval` 변수로 관리된다.

---

## 3. 전체 스케줄링 흐름

```
scheduleCallback(priority, callback, options?)
        │
        ├─ startTime > currentTime (delay 있음)
        │       → requestHostTimeout(handleTimeout, delay)
        │               └─ setTimeout(handleTimeout, delay) 예약
        │               └─ delay 경과 후 handleTimeout 실행
        │                       → advanceTimers → taskQueue로 이동
        │                       → requestHostCallback(flushWork) 호출
        │
        └─ startTime ≤ currentTime (즉시 처리)
                → requestHostCallback(flushWork)
                        └─ port2.postMessage() 호출
                                → (비동기) port1.onmessage 발화
                                → performWorkUntilDeadline 실행
```

---

## 4. `requestHostTimeout` — 딜레이된 Task 처리

```javascript
function requestHostTimeout(callback, ms) {
  taskTimeoutID = localSetTimeout(() => {
    callback(getCurrentTime()); // handleTimeout 호출
  }, ms);
}

function cancelHostTimeout() {
  localClearTimeout(taskTimeoutID);
  taskTimeoutID = -1;
}
```

- `setTimeout`으로 delay만큼 대기 후 `handleTimeout` 호출
- `cancelHostTimeout`은 `clearTimeout`으로 취소

---

## 5. `requestHostCallback`과 `MessageChannel`

### 5.1 왜 `setTimeout(fn, 0)`이 아닌 `MessageChannel`인가?

| 방식                         | 최소 지연 | 이유                                           |
| ---------------------------- | --------- | ---------------------------------------------- |
| `setTimeout(fn, 0)`          | **1~4ms** | 브라우저 스펙상 최소 지연 존재                 |
| `MessageChannel.postMessage` | **~0ms**  | 현재 태스크 종료 즉시 다음 매크로태스크로 진입 |

고빈도로 호출되는 Scheduler에서 1~4ms 지연이 반복되면 성능에 영향이 생긴다.

### 5.2 `MessageChannel` 구조

```javascript
// Scheduler 내부
const channel = new MessageChannel();
const port = channel.port2; // 메시지를 보내는 포트

channel.port1.onmessage = performWorkUntilDeadline; // 메시지를 받는 포트

function requestHostCallback(callback) {
  scheduledHostCallback = callback; // flushWork 저장

  if (!isMessageLoopRunning) {
    isMessageLoopRunning = true;
    port.postMessage(null); // port2로 메시지 전송 → port1.onmessage 발화
  }
}
```

```
port2.postMessage(null)
        │ (현재 동기 실행 스택이 비워진 후)
        ▼ 다음 매크로태스크로 진입
port1.onmessage 발화
        └─ performWorkUntilDeadline 실행
```

### 5.3 `cancelHostCallback`

```javascript
function cancelHostCallback() {
  scheduledHostCallback = null; // 콜백 참조 제거
  // (이미 예약된 postMessage는 취소 불가, 대신 scheduledHostCallback이 null이므로 performWorkUntilDeadline에서 일찍 종료)
}
```

## 6. `performWorkUntilDeadline` — 데드라인 기반 Work 실행

```javascript
const yieldInterval = 5; // ms
let deadline = 0;

function performWorkUntilDeadline() {
  if (scheduledHostCallback !== null) {
    const currentTime = getCurrentTime();

    // 데드라인 설정: 현재 시간 + 5ms
    deadline = currentTime + yieldInterval;

    const hasTimeRemaining = true;
    let hasMoreWork = true;

    try {
      // flushWork 호출 → workLoop 실행
      hasMoreWork = scheduledHostCallback(hasTimeRemaining, currentTime);
    } finally {
      if (hasMoreWork) {
        // 처리할 Work가 더 있으면 postMessage 재호출
        port.postMessage(null); // 다음 매크로태스크에서 재개
      } else {
        // 더 이상 Work 없음 → 메시지 루프 종료
        isMessageLoopRunning = false;
        scheduledHostCallback = null;
      }
    }
  } else {
    isMessageLoopRunning = false;
  }
}
```

### 6.1 데드라인이 사용되는 곳

```javascript
// Scheduler workLoop 내부
function shouldYieldToHost() {
  return getCurrentTime() >= deadline;
  // 현재 시간이 deadline(currentTime + 5ms)을 넘으면 yield
}

// Reconciler workLoopConcurrent 내부
while (workInProgress !== null && !shouldYield()) {
  performUnitOfWork(workInProgress); // Fiber 하나 처리
}
// shouldYield()가 true → 루프 탈출 → 브라우저에 제어권 반환
```

### 6.2 `hasMoreWork`가 `true`일 때의 재귀 흐름

```
performWorkUntilDeadline 실행
    → flushWork → workLoop → (5ms 초과) → shouldYield = true → 루프 탈출
    → flushWork 반환값: hasMoreWork = true (처리 못한 Task 있음)
    → port.postMessage(null) 재호출
    → (브라우저가 이벤트·페인트 처리할 기회)
    → 다음 매크로태스크: performWorkUntilDeadline 다시 실행
    → 반복
```

## 7. 메인 스레드 양보의 전체 구조

```
[메인 스레드 타임라인]

│←──── 16.67ms (1 frame) ────────────────────────────────►│
│                                                          │
│  매크로태스크 A        매크로태스크 B        브라우저    │
│  ┌────────────────┐   ┌────────────────┐   ┌────────┐   │
│  │ performWork    │   │ performWork    │   │ Layout │   │
│  │ (5ms, Work 처리)│   │ (5ms, Work 처리)│   │ Paint  │   │
│  └────────────────┘   └────────────────┘   └────────┘   │
│          ↑                    ↑                          │
│      postMessage          postMessage                    │
│      (다음 태스크 큐)       (다음 태스크 큐)               │
│                                                          │
│  └── 사용자 이벤트, 타이머 등 다른 태스크 처리 가능 ──┘    │
```

## 8. React 18+과의 대응 개념

| React 16/17                         | React 18+                       | 변화 요약                   |
| ----------------------------------- | ------------------------------- | --------------------------- |
| `yieldInterval = 5ms`               | `frameInterval = 5ms`           | 변수명 변경, 값 동일        |
| `deadline` 전역 변수                | `deadline` 전역 변수 유지       | 변화 없음                   |
| `MessageChannel` 사용               | 동일하게 유지                   | 변화 없음                   |
| `requestHostTimeout` (`setTimeout`) | 동일하게 유지                   | 변화 없음                   |
| `performWorkUntilDeadline`          | 동일하게 유지                   | 내부 구현 일부 개선         |
| `shouldYieldToHost()`               | 동일 + `needsPaint` 플래그 추가 | 페인트 필요 시 더 빨리 양보 |

📌 **React 18의 추가 사항**: `needsPaint` 플래그가 추가되었다.  
브라우저가 페인트를 해야 하는 상황이라면 5ms가 지나지 않아도 Work를 중단하고 양보한다.  
이를 통해 애니메이션·스크롤 등 시각적 업데이트의 응답성이 더 향상되었다.

---

## 9. 전체 흐름 요약

```
scheduleCallback
        │
        ├─ delay 있음 → requestHostTimeout → setTimeout → handleTimeout
        │                                                      │
        └─ delay 없음 → requestHostCallback                    │
                              │ ◄─────────────────────────────┘
                              ├─ scheduledHostCallback = flushWork
                              └─ port2.postMessage()
                                        │ (비동기, 현재 스택 비워진 후)
                                        ▼
                              performWorkUntilDeadline
                                ├─ deadline = currentTime + 5ms
                                ├─ flushWork(hasTimeRemaining, currentTime)
                                │     └─ workLoop
                                │           ├─ Fiber 처리
                                │           └─ shouldYieldToHost() 체크
                                │               deadline 초과 → break
                                │
                                ├─ hasMoreWork = true → port2.postMessage() 재호출
                                └─ hasMoreWork = false → 루프 종료
```

# React 18+ — 메인 스레드 양보와 Concurrent 렌더링의 부드러움

## 1. 왜 React가 브라우저에 스레드를 양보하는가?

### 1.1 자바스크립트와 메인 스레드

브라우저의 메인 스레드는 **한 번에 한 가지 일**만 처리한다.

```
메인 스레드가 처리하는 것들:
  ├─ JS 실행 (React Work 포함)
  ├─ 사용자 이벤트 (클릭, 키보드, 스크롤)
  ├─ 타이머 콜백 (setTimeout, setInterval)
  ├─ 렌더링 (Layout, Paint, Composite)
  └─ 네트워크 응답 처리
```

React Work가 메인 스레드를 **독점**하면:

```
React Work 실행 중 (50ms 소요)
│░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░│
                                   ↑
             사용자 클릭 → 50ms 동안 응답 없음 → 화면 멈춤
```

### 1.2 5ms 양보 전략

```
React Work (5ms)│ 브라우저 │ React Work (5ms)│ 브라우저 │
                │이벤트 처리│                 │페인트    │
──────────────────────────────────────────────────────►
                ↑                             ↑
           사용자 클릭 즉시 처리            화면 갱신
```

5ms마다 양보하면 사용자는 입력 지연을 느끼지 못한다.  
인간이 지연을 인식하는 임계값은 약 **100ms**이기 때문이다.

---

## 2. `MessageChannel` — 즉각적인 비동기 실행

### 2.1 매크로태스크 종류별 지연 비교

```
setTimeout(fn, 0)  →  1~4ms 최소 지연 (브라우저 스펙)
setInterval(fn, 0) →  동일
requestAnimationFrame → 다음 프레임 시작까지 대기 (~16ms)
MessageChannel     →  현재 태스크 종료 즉시 (~0ms 지연)
```

React Scheduler가 `MessageChannel`을 선택한 이유: **지연 없이** 다음 매크로태스크로 진입하면서도 현재 태스크를 완전히 끝내고 브라우저에 제어권을 돌려줄 수 있기 때문이다.

### 2.2 매크로태스크 vs 마이크로태스크

```
Promise.then, queueMicrotask (마이크로태스크)
  → 현재 태스크 종료 즉시 실행
  → 브라우저에 제어권 돌아가기 전에 실행
  → 브라우저 이벤트 처리 불가 ← React에 부적합

MessageChannel.postMessage (매크로태스크)
  → 현재 태스크 + 모든 마이크로태스크 완료 후 실행
  → 브라우저가 이벤트·페인트를 처리할 기회 생김 ← React에 적합
```

---

## 3. React 18의 `needsPaint` — 더 스마트한 양보

React 18은 단순한 5ms 타이머 기반 양보를 넘어 **브라우저의 페인트 필요 여부**도 감지한다.

```javascript
// React 18 shouldYieldToHost (개념적)
function shouldYieldToHost() {
  const timeElapsed = getCurrentTime() - startTime;

  if (timeElapsed < frameInterval) {
    // frameInterval = 5ms
    // 아직 5ms 미만 → 계속 Work 수행
    return false;
  }

  if (needsPaint) {
    // 브라우저가 페인트를 해야 함 → 즉시 양보
    return true;
  }

  // 5ms 초과 → 양보
  return true;
}
```

`needsPaint`가 동작하는 방식:

```
사용자 스크롤 발생
  → 브라우저: "페인트가 필요해"
  → needsPaint = true
  → React Work가 3ms 진행됐어도 즉시 양보
  → 브라우저가 스크롤 화면을 즉시 업데이트 → 부드러운 스크롤
```

## 4. `startTransition`과 스레드 양보의 관계

`startTransition`을 사용하면 해당 업데이트는 `TransitionLane`에 배정되어 낮은 우선순위로 처리된다.

```jsx
function SearchPage() {
  const [input, setInput] = useState("");
  const [results, setResults] = useState([]);

  function handleChange(e) {
    setInput(e.target.value); // 즉시 반영 필요
    startTransition(() => {
      setResults(heavySearch(e.target.value)); // 지연 허용
    });
  }
}
```

**스레드 양보와의 연결:**

```
[입력 시 타임라인]

1. setInput → SyncLane/DefaultLane → 높은 우선순위
2. startTransition → TransitionLane → 낮은 우선순위

   Work 시작 (TransitionLane 렌더)
   │← 5ms →│ 양보
              │ 새 입력 이벤트 도착!
              │ 더 높은 우선순위 Task 추가
              ▼
   기존 TransitionLane Work 중단
   새 입력 반영 즉시 처리 (setInput → UI 업데이트)
              ▼
   TransitionLane Work 재시작 (최신 입력값 기준으로)
```

5ms 양보가 없었다면 TransitionLane Work가 메인 스레드를 점유하여 입력 이벤트가 지연됐을 것이다.

---

## 5. `useTransition`으로 로딩 상태 표시

```jsx
function TabContainer() {
  const [isPending, startTransition] = useTransition();
  const [tab, setTab] = useState("home");

  return (
    <>
      <button onClick={() => startTransition(() => setTab("feed"))}>
        Feed
      </button>

      {/* TransitionLane Work가 진행되는 동안 isPending = true */}
      {/* 여러 번의 5ms 양보가 반복되는 동안 스피너 표시 */}
      {isPending && <Spinner />}

      <TabContent tab={tab} />
    </>
  );
}
```

`isPending`이 `true`인 구간은 정확히 **TransitionLane Work가 5ms 단위로 나뉘어 실행되는 기간**이다.

## 6. `Suspense`와 스레드 양보

React 18의 Suspense는 Work를 **일시 중단(suspend)** 하여 브라우저에 제어권을 돌려준다.

```jsx
<Suspense fallback={<Loading />}>
  <SlowComponent /> {/* 데이터 fetching 중 throw Promise */}
</Suspense>
```

```
SlowComponent 렌더 중 → Promise throw
        │
        ▼
React: Work 중단 → Suspense fallback 렌더 → 브라우저에 제어권
        │ (비동기 데이터 도착)
        ▼
React: 중단된 Work 재개 → SlowComponent 렌더 완료
```

이 중단도 `MessageChannel`을 통해 비동기로 재개된다.

## 7. 개발자가 체감하는 변화: React 17 vs React 18

| 시나리오                 | React 17 (Legacy) | React 18 (Concurrent)          |
| ------------------------ | ----------------- | ------------------------------ |
| 무거운 목록 렌더 중 입력 | 입력 지연 발생    | 5ms 양보 → 즉시 반응           |
| 탭 전환 중 스크롤        | 스크롤 버벅임     | `needsPaint` → 즉시 양보       |
| 페이지 로딩 중 클릭      | 클릭 무반응       | 양보 후 클릭 즉시 처리         |
| `startTransition`        | 효과 없음         | TransitionLane → 인터럽트 가능 |

## 8. 전체 흐름 요약

```
[React 18 스레드 양보 전체 그림]

setState / startTransition
        │ Lane 배정
        ▼
scheduleCallback → taskQueue 삽입 → requestHostCallback
        │
        ▼ MessageChannel.postMessage (매크로태스크)
        │ (현재 스택 비워진 후 실행)
        ▼
performWorkUntilDeadline
  deadline = currentTime + 5ms (frameInterval)
        │
        ▼
flushWork → workLoop
  while (workInProgress !== null) {
    if (shouldYieldToHost()) break; // 5ms 초과 or needsPaint
    performUnitOfWork(workInProgress);
  }
        │
        ├─ hasMoreWork = true → port.postMessage() 재호출
        │     │ (브라우저: 이벤트 처리, 페인트 기회)
        │     ▼
        │   performWorkUntilDeadline 재실행
        │
        └─ hasMoreWork = false → 루프 종료, isMessageLoopRunning = false
```

### 핵심 체크리스트
