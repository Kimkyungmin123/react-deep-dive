# React Fiber — 렌더 페이즈에서 `setState` 호출 시 업데이트 저장·소비 메커니즘

## 1. Idle 상태 vs 렌더 페이즈에서의 `setState` 차이

### 1.1 두 상태의 핵심 차이

| 구분           | Idle 상태                                    | 렌더 페이즈                                 |
| -------------- | -------------------------------------------- | ------------------------------------------- |
| Work 등록 여부 | 스케줄러에 Work를 새로 등록해야 함           | 이미 Work가 등록·실행 중                    |
| 최적화 로직    | `ensureRootIsScheduled`로 우선순위 비교 필요 | 우선순위 비교 로직 불필요                   |
| 업데이트 처리  | 큐에 저장 → 스케줄러 → 렌더 페이즈 진입      | 임시 저장소에 저장 → 컴포넌트 재호출로 소비 |

### 1.2 렌더 페이즈 `setState`의 처리 방향

렌더 페이즈에서 `setState`가 호출되면 다음 세 가지를 순서대로 수행한다.

1. **업데이트 임시 저장**: 진행 중인 렌더링을 방해하지 않도록 Map 기반 임시 저장소에 보관한다.
2. **업데이트 소비**: 컴포넌트를 재호출하며 저장된 업데이트를 순서대로 소비한다.
3. **재귀적 업데이트 처리**: 소비 중 다시 `setState`가 호출될 경우 25회 제한으로 무한 루프를 방지한다.

---

## 2. 업데이트 임시 저장 메커니즘

### 2.1 `didScheduleRenderPhaseUpdate` 플래그

렌더 페이즈 `setState` 경로로 진입하면 가장 먼저 전역 변수 `didScheduleRenderPhaseUpdate`를 `true`로 설정한다.

```javascript
didScheduleRenderPhaseUpdate = true;
```

이 플래그는 `renderWithHooks` 함수 내 `while` 루프의 조건으로 사용된다.  
렌더 페이즈 업데이트가 발생했음을 알리는 **신호등** 역할이다.

### 2.2 임시 저장소: `renderPhaseUpdates` Map 객체

업데이트를 저장할 임시 저장소로 **Map 객체**를 사용한다.

```javascript
if (renderPhaseUpdates === null) {
  renderPhaseUpdates = new Map();
}
```

> ⚠️ **왜 일반 객체가 아닌 Map을 사용하는가?**  
> Map은 **객체(Object)를 키(Key)로 사용할 수 있다.** 여기서 키로 사용하는 것은 훅의 **큐 객체(`queue`)** 자체이므로, 객체를 키로 허용하는 Map이 적합하다.

### 2.3 Map에 업데이트를 저장하는 방식

키는 **훅의 큐 객체(`queue`)**, 값은 **업데이트 객체(링크드 리스트)** 이다.

```
renderPhaseUpdates = {
  queue1: update1 → update2 → update3 → (circular),
  queue2: update4,
}
```

**첫 번째 업데이트 저장:**

```javascript
const firstRenderPhaseUpdate = renderPhaseUpdates.get(queue);
if (firstRenderPhaseUpdate === undefined) {
  // 해당 큐에 대한 첫 업데이트 → 바로 set
  renderPhaseUpdates.set(queue, update);
}
```

**두 번째 이후 업데이트 저장 (링크드 리스트 꼬리에 추가):**

```javascript
} else {
  // 이미 저장된 업데이트가 있으면 → 링크드 리스트 꼬리를 찾아 연결
  let lastRenderPhaseUpdate = firstRenderPhaseUpdate;
  while (lastRenderPhaseUpdate.next !== null) {
    lastRenderPhaseUpdate = lastRenderPhaseUpdate.next;
  }
  lastRenderPhaseUpdate.next = update; // 꼬리에 새 업데이트 연결
}
```

> ⚠️ **흔한 오해**: 여기서 사용하는 링크드 리스트는 Idle 상태의 훅 큐와 **별개의 임시 저장소**다.  
> 렌더 페이즈가 끝나면 `renderPhaseUpdates`는 소비 후 초기화된다.

## 3. 업데이트 소비 과정과 무한 루프 방지

### 3.1 `renderWithHooks` 내부의 소비 루프

업데이트 소비는 `renderWithHooks` 함수 내 `do...while` 루프에서 이루어진다.

```javascript
do {
  didScheduleRenderPhaseUpdate = false; // 플래그 초기화
  numberOfReRenders += 1; // 재렌더 횟수 증가

  // 훅 상태 초기화
  currentHook = null;
  workInProgressHook = null;
  workInProgress.memoizedState = null;

  // 컴포넌트 재호출 → 이 안에서 setState가 다시 호출될 수 있음
  children = Component(props, secondArg);
} while (didScheduleRenderPhaseUpdate); // 재호출 중 또 setState가 불렸으면 반복
```

**루프 진행 흐름:**

```
컴포넌트 재호출
    │
    ├─ setState 호출 없음 → didScheduleRenderPhaseUpdate = false → 루프 종료
    │
    └─ setState 호출 있음 → didScheduleRenderPhaseUpdate = true → 루프 반복
```

### 3.2 25회 재렌더 제한

루프 진입 시마다 `numberOfReRenders`에 `+1`을 하고, 25를 초과하면 에러를 던진다.

```javascript
if (numberOfReRenders >= RE_RENDER_LIMIT) {
  // RE_RENDER_LIMIT = 25
  throw new Error(
    "Too many re-renders. React limits the number of renders " +
      "to prevent an infinite loop.",
  );
}
```

> ⚠️ **`numberOfReRenders`는 렌더 페이즈 재진입 횟수만 카운트한다.**  
> 일반적인 렌더링(Idle 상태 `setState`로 시작된)은 이 카운터와 무관하다.  
> 즉, "25회 제한"은 **하나의 렌더 사이클 내에서 렌더 페이즈 업데이트가 연쇄적으로 발생하는 횟수**에 대한 제한이다.

### 3.3 "Too many re-renders" 에러가 발생하는 전형적인 패턴

```jsx
// ❌ 잘못된 예: 렌더 중 setState를 직접 호출
function Bad() {
  const [count, setCount] = useState(0);
  setCount(count + 1); // 렌더마다 호출 → 무한 루프
  return <div>{count}</div>;
}

// ✅ 올바른 예: 이벤트 핸들러에서 setState 호출
function Good() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}

// ⚠️ 주의: 이것도 렌더 페이즈 setState
function AlsoBad() {
  const [flag, setFlag] = useState(false);
  if (!flag) {
    setFlag(true); // 조건 없는 렌더 중 setState
  }
  return <div />;
}
```

## 4. 소비 완료 후 처리

렌더 페이즈 업데이트 소비가 끝나면 두 가지 후속 처리가 이루어진다.

1. **디스패처 교체**: `ReactCurrentDispatcher.current`를 `HooksDispatcherOnUpdate`로 교체한다.  
   마운트용 디스패처(`HooksDispatcherOnMount`)와 업데이트용 디스패처는 훅 내부 구현이 다르기 때문이다.

2. **루프 탈출 및 다음 단계 진행**: `didScheduleRenderPhaseUpdate === false` 조건으로 루프를 탈출하고, 이후 커밋 페이즈(Commit Phase)로 진행한다.

## 5. React 18+과의 대응 개념

| React 16/17                           | React 18+                        | 변화 요약                             |
| ------------------------------------- | -------------------------------- | ------------------------------------- |
| `renderPhaseUpdates` (Map)            | 동일한 Map 기반 임시 저장소 유지 | 구조 변화 없음                        |
| `numberOfReRenders` (25 제한)         | 동일하게 유지                    | 제한 횟수 동일                        |
| `didScheduleRenderPhaseUpdate` 플래그 | 동일하게 유지                    | 내부 구현 일부 개선                   |
| `HooksDispatcherOnUpdate`             | 동일하게 유지                    | 구조 변화 없음                        |
| 렌더 페이즈 업데이트 자체             | **가능하면 사용 자제 권고**      | `useEffect` + 상태 파생으로 대체 권장 |

📌 **React 18의 추가 안전장치**: React 18의 Strict Mode에서는 개발 환경에서 컴포넌트를 **의도적으로 두 번 렌더링**하여 렌더 페이즈 부수효과(side effect)를 조기에 감지한다. 렌더 페이즈 `setState`가 포함된 컴포넌트는 Strict Mode에서 예상치 못한 동작을 보일 수 있다.

## 6. 전체 흐름 요약

```
렌더 페이즈 중 setState 호출
        │
        ▼
dispatchAction (renderPhase 경로)
        │
        ├─ didScheduleRenderPhaseUpdate = true
        ├─ renderPhaseUpdates Map에 업데이트 저장 (링크드 리스트)
        │
        ▼
renderWithHooks 내 do...while 루프
        │
        ├─ [루프 진입 시]
        │     didScheduleRenderPhaseUpdate = false
        │     numberOfReRenders += 1
        │     numberOfReRenders > 25? → "Too many re-renders" 에러
        │
        ├─ 훅 변수 초기화
        │
        ├─ 컴포넌트 재호출 (renderPhaseUpdates 소비)
        │     │
        │     ├─ setState 또 호출됨 → didScheduleRenderPhaseUpdate = true → 루프 반복
        │     └─ setState 없음     → didScheduleRenderPhaseUpdate = false → 루프 탈출
        │
        └─ HooksDispatcherOnUpdate로 디스패처 교체 → 커밋 페이즈 진행
```

# React 18+ — 렌더 페이즈 `setState`와 무한 루프 방지

## 1. 렌더 페이즈 `setState`란?

### 1.1 렌더 페이즈가 무엇인가?

React의 렌더링은 크게 두 단계로 나뉜다.

| 단계            | 역할                           | 부수효과 허용 여부                       |
| --------------- | ------------------------------ | ---------------------------------------- |
| **렌더 페이즈** | 컴포넌트 함수 호출 → VDOM 계산 | ❌ 허용하지 않음                         |
| **커밋 페이즈** | 실제 DOM 업데이트              | ✅ 허용 (`useEffect`, `useLayoutEffect`) |

### 1.2 렌더 페이즈에서 `setState`를 호출하면?

컴포넌트 함수 본문(렌더 페이즈)에서 `setState`를 호출하면 React는 다음 과정을 거친다.

```
컴포넌트 함수 실행 중 setState 호출
        │
        ├─ 업데이트를 임시 저장소(Map)에 보관
        ├─ 컴포넌트를 재호출하여 업데이트 소비
        └─ 재호출 중 또 setState가 불리면? → 최대 25회까지 반복, 초과 시 에러
```

이 과정에서 Idle 상태와 달리 **스케줄러에 새로운 Work를 등록하지 않는다.**  
이미 진행 중인 렌더 사이클 안에서 처리하기 때문이다.

---

## 2. "Too many re-renders" 에러의 실체

### 2.1 에러 발생 조건

React는 렌더 페이즈 재진입 횟수를 `numberOfReRenders` 변수로 추적한다.  
이 값이 **25를 초과**하면 다음 에러를 던진다.

```
Error: Too many re-renders.
React limits the number of renders to prevent an infinite loop.
```

### 2.2 에러를 유발하는 전형적인 패턴

```jsx
// ❌ 패턴 1: 렌더 중 무조건적 setState 호출
function InfiniteLoop() {
  const [count, setCount] = useState(0);
  setCount(count + 1); // 렌더마다 호출 → 무한 루프
  return <div>{count}</div>;
}

// ❌ 패턴 2: 이벤트 핸들러에 함수 호출 결과를 전달 (흔한 실수)
function MistakeHandler() {
  const [value, setValue] = useState("");
  // onClick에 함수 참조가 아니라 즉시 호출 결과를 전달
  return <button onClick={setValue("clicked")}>Click</button>;
  //                     ^^^^^^^^^^^^^^^^ 렌더마다 실행됨
}

// ❌ 패턴 3: 조건이 항상 참인 렌더 중 setState
function AlwaysTrue({ data }) {
  const [processed, setProcessed] = useState(null);
  if (data) {
    setProcessed(transform(data)); // data가 항상 존재하면 무한 루프
  }
  return <div>{processed}</div>;
}
```

### 2.3 올바른 대안 패턴

```jsx
// ✅ 대안 1: useEffect로 부수효과 이동
function Fixed1({ data }) {
  const [processed, setProcessed] = useState(null);
  useEffect(() => {
    if (data) setProcessed(transform(data));
  }, [data]); // data 변경 시에만 실행
  return <div>{processed}</div>;
}

// ✅ 대안 2: 렌더 중 상태를 파생(derive)하여 setState 자체를 없앰
function Fixed2({ data }) {
  // 별도의 상태 없이 렌더 시점에 계산
  const processed = data ? transform(data) : null;
  return <div>{processed}</div>;
}

// ✅ 대안 3: useMemo로 비용이 큰 연산 최적화
function Fixed3({ data }) {
  const processed = useMemo(() => transform(data), [data]);
  return <div>{processed}</div>;
}

// ✅ 대안 4: 이벤트 핸들러에는 함수 참조 전달
function Fixed4() {
  const [value, setValue] = useState("");
  return <button onClick={() => setValue("clicked")}>Click</button>;
  //                     ^^^ 화살표 함수로 감싸서 참조 전달
}
```

## 3. 렌더 페이즈 `setState`가 허용되는 경우

모든 렌더 페이즈 `setState`가 잘못된 것은 아니다.  
React가 공식적으로 허용하는 패턴이 있다.

### 3.1 이전 렌더의 값과 비교하여 상태를 동기화

React 공식 문서에서 안내하는 패턴으로, **이전 props와 현재 props를 비교하여 상태를 업데이트**할 때 사용한다.

```jsx
function CountLabel({ count }) {
  const [prevCount, setPrevCount] = useState(count);
  const [trend, setTrend] = useState(null);

  // 렌더 중 setState — 이전 값과 다를 때만 호출하므로 무한 루프 없음
  if (prevCount !== count) {
    setPrevCount(count);
    setTrend(count > prevCount ? "increasing" : "decreasing");
  }

  return (
    <>
      <h1>{count}</h1>
      {trend && <p>The count is {trend}</p>}
    </>
  );
}
```

**이 패턴이 안전한 이유:**

- `prevCount !== count` 조건이 결국 `false`가 되는 시점이 반드시 존재한다.
- 재호출이 무한히 반복되지 않으므로 25회 제한에 걸리지 않는다.

> ⚠️ **이 패턴은 `useEffect`보다 효율적이다.**  
> `useEffect`를 사용하면 DOM 업데이트 이후 한 번 더 렌더링이 발생하지만,  
> 렌더 페이즈 `setState`는 현재 렌더 사이클 안에서 처리되어 **화면에 중간 상태가 노출되지 않는다.**

---

## 4. React 18 Strict Mode와 렌더 페이즈

React 18의 Strict Mode는 개발 환경에서 컴포넌트를 **의도적으로 두 번 렌더링**한다.

```jsx
// main.jsx
root.render(
  <React.StrictMode>
    <App />
  </React.StrictMode>,
);
```

**목적**: 렌더 페이즈에서 발생하는 **부수효과(side effect)를 조기에 발견**하기 위함이다.

| 렌더 페이즈에서 허용되지 않는 것 | 이유                              |
| -------------------------------- | --------------------------------- |
| 외부 변수 직접 변경              | 두 번 렌더 시 예상치 못한 값이 됨 |
| API 호출                         | 두 번 요청이 발생함               |
| DOM 직접 조작                    | 렌더 페이즈는 순수해야 함         |
| 조건 없는 `setState`             | 무한 루프 발생                    |

## 5. 정리 및 실전 가이드

```
렌더 페이즈 setState 발생
        │
        ├─ 무한 루프 위험 패턴?
        │     YES → useEffect 또는 상태 파생(derive)으로 리팩토링
        │     NO  → 이전값 비교 조건이 있는 경우 → 허용되는 패턴
        │
        └─ 최대 25회 재렌더 제한
              초과 시 "Too many re-renders" 에러
```

### 렌더 페이즈 `setState` 체크리스트

- [ ] 렌더 함수 본문에 조건 없이 `setState`를 호출하고 있지는 않은가?
- [ ] 이벤트 핸들러에 함수를 **호출한 결과**가 아닌 **함수 참조**를 전달하고 있는가?
- [ ] props 변화에 따른 상태 동기화가 필요하다면 `useEffect` 또는 **이전값 비교 렌더 패턴** 중 적합한 것을 선택했는가?
- [ ] 비용이 큰 계산은 `useMemo`로 최적화했는가?
- [ ] Strict Mode(개발 환경 이중 렌더) 하에서도 정상 동작하는가?
