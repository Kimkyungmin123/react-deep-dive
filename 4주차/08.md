# React Fiber — Idle 상태와 렌더 페이즈를 구분하는 조건문

## 1. Idle 상태와 렌더 페이즈의 정의

### 1.1 두 상태의 개념

| 상태                          | 정의                                          | React 라이프사이클 위치 |
| ----------------------------- | --------------------------------------------- | ----------------------- |
| **Idle 상태**                 | 진행 중인 업데이트가 없는 상태                | 렌더 페이즈 이전        |
| **렌더 페이즈(Render Phase)** | 컴포넌트 함수가 호출되어 VDOM을 계산하는 단계 | Reconciliation 진행 중  |

- **Idle 상태**는 특정 페이즈(Phase)에 속하지 않는다. 렌더 페이즈 진입 이전의 구간이다.
- `setState` 호출 후 스케줄러에 Work 등록이 완료되는 순간 Idle 상태가 종료되고, 이후 렌더 페이즈로 진입한다.

### 1.2 두 상태에서 `setState` 동작의 차이

```
[Idle 상태에서 setState 호출]
setState → dispatchAction → 업데이트 객체 생성
        → 큐에 저장 → ensureRootIsScheduled → Scheduler에 Work 등록
        → (나중에) 렌더 페이즈 진입

[렌더 페이즈에서 setState 호출]
setState → dispatchAction → 업데이트 객체 생성
        → renderPhaseUpdates Map에 임시 저장
        → 컴포넌트 재호출로 소비 (Scheduler 등록 없음)
```

## 2. 두 상태를 구분하는 코드: `dispatchAction` 내 조건문

`dispatchAction` 함수는 `useState`의 setter가 내부적으로 호출하는 함수이다.  
이 함수 안에서 **현재 Idle 상태인지 렌더 페이즈인지**를 판단하는 조건문이 핵심이다.

### 2.1 조건문의 구조

```javascript
// dispatchAction 내부
function dispatchAction(fiber, queue, action) {
  // ...
  const currentlyRenderingFiber = ReactCurrentlyRenderingFiber;

  if (
    fiber === currentlyRenderingFiber ||
    fiber.alternate === currentlyRenderingFiber
  ) {
    // 렌더 페이즈 업데이트 경로
    didScheduleRenderPhaseUpdate = true;
    // renderPhaseUpdates Map에 임시 저장
    // ...
  } else {
    // Idle 상태 업데이트 경로
    // 큐에 저장 → ensureRootIsScheduled → Scheduler 등록
    // ...
  }
}
```

조건문이 참(`true`)이면 **렌더 페이즈 경로**, 거짓(`false`)이면 **Idle 상태 경로**로 분기한다.

### 2.2 `currentlyRenderingFiber`란?

| 항목                 | 내용                                                                     |
| -------------------- | ------------------------------------------------------------------------ |
| 변수명               | `ReactCurrentlyRenderingFiber` (코드에 따라 `currentlyRenderingFiber$1`) |
| 타입                 | `Fiber \| null`                                                          |
| 초기값               | `null`                                                                   |
| 값이 설정되는 시점   | 컴포넌트 함수 호출 직전 (`renderWithHooks` 진입 시)                      |
| 값이 초기화되는 시점 | 컴포넌트 함수 호출 완료 후 (`renderWithHooks` 종료 시)                   |

즉, 이 변수는 **현재 렌더링 중인 컴포넌트의 Work in Progress Fiber를 가리킨다.**  
렌더 페이즈 밖(Idle 상태)에서는 항상 `null`이다.

> ⚠️ **주의**: `currentlyRenderingFiber`는 이름과 달리 실제로는 **Work in Progress Fiber**이다.  
> React 소스코드 주석에도 "이 변수는 사실 Work in Progress Fiber이며, 구별을 위해 이름을 다르게 지정했다"고 명시되어 있다.

---

## 3. 왜 `fiber`와 `fiber.alternate` 둘 다 비교하는가?

### 3.1 Fiber의 이중 버퍼링 구조

React의 Virtual DOM은 항상 두 개의 Fiber 트리를 유지한다.

| 트리                       | 역할                              |
| -------------------------- | --------------------------------- |
| **Current Fiber**          | 현재 화면에 렌더링된 상태         |
| **Work in Progress Fiber** | 다음 렌더링을 위해 계산 중인 상태 |

두 트리의 Fiber 노드는 서로의 **`alternate`** 필드를 통해 상호 참조한다.

```
CurrentFiber.alternate          → WorkInProgressFiber
WorkInProgressFiber.alternate   → CurrentFiber
```

### 3.2 재조정(Reconciliation) 완료 후의 역할 교체

업데이트가 완료되면 Work in Progress가 Current로 승격되고, 기존 Current는 다음 렌더링의 Work in Progress 후보가 된다.

```
[업데이트 전]
Current        ←alternate→  WorkInProgress

[업데이트 완료 후]
(기존 WIP가 Current로 승격)
새 Current     ←alternate→  (기존 Current, 다음 WIP 후보)
```

이 역할 교체 때문에 어느 시점에 `fiber` 변수가 Current인지 Work in Progress인지 확정할 수 없다.

### 3.3 양쪽을 모두 비교해야 하는 이유

`dispatchAction`에 전달되는 `fiber` 인자는 훅이 속한 Fiber 노드인데, 이 노드가 현재 **Current인지 Work in Progress인지 알 수 없다.**

```javascript
// fiber가 WIP일 때를 커버
fiber === currentlyRenderingFiber;

// fiber가 Current이고, 그 alternate(WIP)가 currentlyRenderingFiber일 때를 커버
fiber.alternate === currentlyRenderingFiber;
```

두 조건 중 하나라도 참이면 현재 렌더 페이즈가 진행 중임을 확인할 수 있다.

```
경우 1: fiber가 WIP인 경우
  fiber === currentlyRenderingFiber → true ✅

경우 2: fiber가 Current인 경우
  fiber !== currentlyRenderingFiber → false
  fiber.alternate === currentlyRenderingFiber → true ✅

경우 3: 렌더 페이즈 아님 (Idle)
  currentlyRenderingFiber === null
  fiber === null → false
  fiber.alternate === null → false ✅
```

## 4. 렌더 페이즈 업데이트가 발생하는 실제 코드 예시

```jsx
function Counter() {
  const [a, setA] = useState(0);

  // ⚠️ 렌더 페이즈에서 setState 호출 예시
  // a가 1이 되는 순간 렌더 중에 setA(0)이 다시 호출됨
  if (a === 1) {
    setA(0); // → 렌더 페이즈 업데이트 발생
  }

  return <button onClick={() => setA(1)}>Click</button>;
}
```

**흐름 추적:**

```
버튼 클릭 → setA(1) → Idle 경로 → 렌더 페이즈 진입
→ Counter() 호출, a === 1 조건 만족
→ setA(0) 호출 → dispatchAction 내 조건문 검사
→ fiber === currentlyRenderingFiber (true) → 렌더 페이즈 경로
→ renderPhaseUpdates Map에 저장
→ 컴포넌트 재호출로 소비
```

> ⚠️ 이 패턴은 무한 루프 위험이 있다. 실무에서는 렌더 페이즈 `setState`를 최소화하고, 이전값 비교 조건을 반드시 포함해야 한다.

## 5. React 18+과의 대응 개념

| React 16/17                           | React 18+     | 변화 요약                       |
| ------------------------------------- | ------------- | ------------------------------- |
| `currentlyRenderingFiber` 비교 조건문 | 동일하게 유지 | 구조 변화 없음                  |
| `fiber.alternate` 상호 참조 구조      | 동일하게 유지 | Fiber 이중 버퍼링 구조 유지     |
| `didScheduleRenderPhaseUpdate` 플래그 | 동일하게 유지 | 내부 구현 일부 개선             |
| Idle vs 렌더 페이즈 분기              | 동일하게 유지 | 분기 로직 자체는 변화 없음      |
| Current ↔ WIP 역할 교체               | 동일하게 유지 | Lanes 모델 도입과 무관하게 유지 |

📌 **React 18의 차이점**: Concurrent 기능이 기본화되면서 Work in Progress 트리가 **중단되고 재시작될 수 있다.** 이 경우 `currentlyRenderingFiber`가 중간에 `null`로 초기화되는 시점이 더 자주 발생한다. 그러나 `dispatchAction` 내 조건문의 구조 자체는 동일하다.

## 6. 전체 흐름 요약

```
dispatchAction(fiber, queue, action) 호출
        │
        ▼
  조건문 검사
  fiber === currentlyRenderingFiber
  OR
  fiber.alternate === currentlyRenderingFiber
        │
        ├─ true  → [렌더 페이즈 경로]
        │            didScheduleRenderPhaseUpdate = true
        │            renderPhaseUpdates Map에 저장
        │            컴포넌트 재호출로 소비
        │
        └─ false → [Idle 상태 경로]
                     업데이트 큐에 저장
                     ensureRootIsScheduled
                     Scheduler에 Work 등록
```

# React 18+ — Fiber 이중 버퍼링과 렌더 페이즈 감지

## 1. Fiber란 무엇인가?

### 1.1 Fiber의 역할

React 16부터 도입된 **Fiber**는 각 컴포넌트(또는 DOM 노드)에 대응하는 **내부 작업 단위 객체**이다.

```
<App>          →  App Fiber
  <Header>     →  Header Fiber
  <Main>       →  Main Fiber
    <List>     →  List Fiber
```

Fiber 객체에는 다음 정보가 담긴다.

| 필드                         | 내용                        |
| ---------------------------- | --------------------------- |
| `type`                       | 컴포넌트 함수 또는 DOM 태그 |
| `memoizedState`              | 훅의 상태값 (useState 등)   |
| `memoizedProps`              | 현재 props                  |
| `alternate`                  | 이중 버퍼링의 대응 Fiber    |
| `child`, `sibling`, `return` | Fiber 트리 연결 포인터      |

### 1.2 이중 버퍼링 구조

React는 항상 두 개의 Fiber 트리를 유지한다.

```
┌─────────────────────┐         ┌─────────────────────┐
│   Current Tree      │◄───────►│  Work in Progress   │
│  (화면에 렌더된 상태) │ alternate│  (계산 중인 상태)    │
└─────────────────────┘         └─────────────────────┘
```

- **Current Tree**: 현재 화면에 표시된 상태
- **Work in Progress Tree**: 새 업데이트를 계산 중인 트리

업데이트가 완료되면 Work in Progress가 Current로 **승격(flip)** 되고, 기존 Current는 다음 렌더링의 Work in Progress 후보가 된다.

> ⚠️ **이중 버퍼링을 사용하는 이유**: 업데이트 중인 트리(WIP)와 화면에 표시된 트리(Current)를 분리함으로써, 렌더링 중간 상태가 화면에 노출되지 않는다. 계산이 완전히 끝난 후에만 Current로 교체한다.

## 2. `currentlyRenderingFiber` — 렌더 페이즈 감지의 핵심

### 2.1 변수의 생애 주기

```
renderWithHooks 진입
        │
        ▼
currentlyRenderingFiber = workInProgress  ← 렌더 시작 시 설정
        │
        ▼
컴포넌트 함수 실행 (이 구간에서만 null이 아님)
        │
        ▼
renderWithHooks 종료
        │
        ▼
currentlyRenderingFiber = null  ← 렌더 종료 시 초기화
```

- 이 변수가 `null`이 아닌 구간 = 렌더 페이즈
- `setState`(내부적으로 `dispatchAction`)가 이 변수를 참조하여 현재 렌더 페이즈인지 판단한다.

### 2.2 `useState` setter 호출 시 분기 로직

```javascript
// useState의 setter를 호출하면 내부적으로 dispatchAction이 실행됨
const [state, setState] = useState(0);
setState(1); // → dispatchAction(fiber, queue, 1) 호출

// dispatchAction 내부 조건문
if (
  fiber === currentlyRenderingFiber || // WIP Fiber인 경우
  fiber.alternate === currentlyRenderingFiber // Current Fiber인 경우
) {
  // 렌더 페이즈 → renderPhaseUpdates에 임시 저장
} else {
  // Idle 상태 → Scheduler에 Work 등록
}
```

## 3. Concurrent 렌더링에서 이중 버퍼링이 중요한 이유

React 18의 Concurrent 기능에서는 렌더링이 **중단(interrupt)되고 재시작**될 수 있다.

```
렌더링 시작 (WIP 트리 계산 중)
        │
        ├─ 더 긴급한 업데이트 도착!
        │
        ▼
현재 WIP 트리 계산 중단
(Current 트리는 그대로 화면에 유지됨)
        │
        ▼
긴급 업데이트 처리 완료
        │
        ▼
WIP 트리 계산 재시작 (또는 처음부터 다시 시작)
```

이중 버퍼링이 없다면 중단 시점의 반쯤 계산된 트리가 화면에 노출될 수 있다.  
Current 트리를 유지하면서 WIP 트리만 조작하기 때문에 화면은 항상 안정적인 상태를 유지한다.

## 4. 개발자가 알아야 할 실전 규칙

### 4.1 렌더 페이즈는 순수해야 한다

컴포넌트 함수 본문(렌더 페이즈)은 **순수 함수(Pure Function)처럼 동작해야 한다.**

```jsx
// ❌ 렌더 페이즈에서 외부 상태 변경
let externalCount = 0;
function Bad() {
  externalCount++; // 렌더 페이즈 부수효과 → Strict Mode에서 2회 실행 시 문제
  return <div>{externalCount}</div>;
}

// ✅ 부수효과는 useEffect로 분리
function Good() {
  useEffect(() => {
    externalCount++; // 커밋 페이즈에서 실행
  }, []);
  return <div />;
}
```

### 4.2 React 18 Strict Mode의 이중 실행

React 18 Strict Mode는 개발 환경에서 컴포넌트를 **의도적으로 두 번 렌더링**한다.

```jsx
root.render(
  <React.StrictMode>
    <App />
  </React.StrictMode>,
);
```

**이중 실행의 목적**: 렌더 페이즈가 순수하지 않은 컴포넌트를 조기에 발견하기 위함이다.

```
Strict Mode 렌더링 순서 (개발 환경)

1차 렌더 → 결과 버림 (부수효과 감지용)
2차 렌더 → 실제 결과 사용
```

> ⚠️ 두 번의 렌더에서 결과가 다르면 Strict Mode가 경고를 표시한다.  
> 이는 컴포넌트가 순수하지 않다는 신호다.

### 4.3 `alternate` 구조의 실전적 이해

직접 `alternate`를 조작할 일은 없지만, 다음 두 가지를 이해하면 디버깅에 도움이 된다.

```
1. React DevTools에서 보이는 Fiber 정보는 Current Tree 기준이다.
   렌더 중간 상태(WIP)는 보이지 않는다.

2. 컴포넌트가 "리렌더"되면 WIP Fiber가 새로 계산되고,
   업데이트 완료 후 Current로 승격된다.
   이때 이전 Current는 다음 렌더의 WIP 재활용 후보가 된다.
```

## 5. 정리

```
[React 18 Fiber 이중 버퍼링 전체 그림]

Idle 상태
    │
    ▼ setState 호출 (Idle 경로)
Work in Progress 트리 계산 시작
currentlyRenderingFiber = WIP Fiber   ← 렌더 페이즈 시작
    │
    ├─ 렌더 중 setState 호출?
    │     dispatchAction 조건문 → 렌더 페이즈 경로 (임시 저장)
    │
    ▼
계산 완료 (또는 중단·재시작 가능 — React 18 Concurrent)
currentlyRenderingFiber = null         ← 렌더 페이즈 종료
    │
    ▼
커밋 페이즈 — WIP를 Current로 승격 (화면 업데이트)
    │
    ▼
Idle 상태로 복귀
```

### 핵심 정리 체크리스트

- [ ] Fiber = 컴포넌트에 대응하는 내부 작업 단위 객체임을 이해했다.
- [ ] Current Tree ↔ Work in Progress Tree 이중 버퍼링 구조를 이해했다.
- [ ] `currentlyRenderingFiber`가 null이 아닌 구간 = 렌더 페이즈임을 이해했다.
- [ ] `fiber === currentlyRenderingFiber || fiber.alternate === currentlyRenderingFiber` 조건이 Idle vs 렌더 페이즈를 구분함을 이해했다.
- [ ] Concurrent에서 WIP가 중단·재시작 가능하며, Current가 화면 안정성을 보장함을 이해했다.
- [ ] 렌더 페이즈는 순수해야 하며, 부수효과는 `useEffect`로 분리해야 함을 이해했다.
