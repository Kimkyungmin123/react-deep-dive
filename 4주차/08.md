# setState 를 호출했을 때, render phase 에서 벌어지는 일?!(render phase 에서 update 를 저장하고 소비하는 과정)

## 1. 기본 필기

### render phase에서 setState

이미 idle에서 WORK 를 scheduler에게 등록했으므로, 다시 WORK를 등록할 필요 없고, 최적화할 필요도 없음 → render pahse update가 더 이상 발생하지 않을 때까지 컴포넌트를 재호출하고 action(setState의 인자)을 소비하면 됨

*최적화 -> WORK 를 scheduler에게 등록할지 말지를 판단해서 불필요한 등록을 하지 않으려고 최적화 로직 필요 *등록하면 비용크므로

### render phase에서 update의 저장

> 해야하는 이유: 계속해서 추가적인 update가 발생할텐데, 발생한 update에 대해서 소비하기 전에 새로운 update가 발생해버리니까 새로 발생해버린 update와 이전꺼를 모두 소비하기 위해선 이전 것 들을 임시저장소에 저장해둘 필요가 있다.

didScheduleRenderPhaseUpdate
기존에 진행중이던 update를 잠깐 담아둘 임시 저장소 필요 → renderPhaseUpdate (Map)
Map은 객체와 비슷하게 키 벨류 형태로 값을 저장. 이때 키의 타입으로 string 뿐만 아니라 객체도 가능
linked list 형태로 renderPhaseUpdates를 저장

- didScheduleRenderPhaseUpdate(renderphas를 스케줄에 등록?)에 true 할당 -> 19에서는 react-server 폴더 하위에서 하고 있음..
- upate 객체 만듦
- renderPhaseUpdate가 null일 경우 map 할당
- new Map() → key 다양한 타입 허용.객체도 key 가능.
- `const firstRenderPhaseUpdate = renderPhaseUpdate.get(queue);`

### reder phase 에서 update의 소비

renderWithHooks()에서 renderPhaseUpdates에 저장한 update 소비
didScheduleRenderPhaseUpdate 로 발생 여부 판단
numberOfRenders
훅 업데이트 구현체에서 render phase update를 소비하기 위해 필요한 변수들의 값을 할당
nextCurrentHook
훅 업데이트 구현체를 didspatcher에 주입
ReactCurrentDispatcher.current = HooksDispatcherOnUpdate

### update 소비 중에 다시 update 발생

컴포넌트 재호출 → 여기서 다시 update 발생한다면 didScheduleRenderPhaseUpdate 값 true됨
위 과정 다시 반복
numberOfReRenders가 +1

```tsx
function dispatchAction<A>(
  componentIdentity: Object,
  queue: UpdateQueue<A>,
  action: A,
): void {
  if (numberOfReRenders >= RE_RENDER_LIMIT) {
    // RE_RENDER_LIMIT:25
    throw new Error(
      "Too many re-renders. React limits the number of renders to prevent " +
        "an infinite loop.",
    );
  }

  if (componentIdentity === currentlyRenderingComponent) {
    // This is a render phase update. Stash it in a lazily-created map of
    // queue -> linked list of updates. After this render pass, we'll restart
    // and apply the stashed updates on top of the work-in-progress hook.
    didScheduleRenderPhaseUpdate = true;
    const update: Update<A> = {
      action,
      next: null,
    };
    if (renderPhaseUpdates === null) {
      renderPhaseUpdates = new Map(); // renderPhaseUpdates목적: renderphase에서 발생한 update들을 임시로 저장해두는 저장소
    }
    const firstRenderPhaseUpdate = renderPhaseUpdates.get(queue); // queue객체를 키로 가져옴
    if (firstRenderPhaseUpdate === undefined) {
      // $FlowFixMe[incompatible-use] found when upgrading Flow
      renderPhaseUpdates.set(queue, update); // 첫번째 renderPhaseupdate 저장
    } else {
      // Append the update to the end of the list.
      let lastRenderPhaseUpdate = firstRenderPhaseUpdate;

      // linkedList로 update 저장는 로직.
      while (lastRenderPhaseUpdate.next !== null) {
        lastRenderPhaseUpdate = lastRenderPhaseUpdate.next;
      }
      lastRenderPhaseUpdate.next = update;
    }
  } else {
    // This means an update has happened after the function component has
    // returned. On the server this is a no-op. In React Fiber, the update
    // would be scheduled for a future render.
  }
}
```

```tsx
// 16에서 renderWithHooks함수 내부 동작 -> 19에서renderWithHooksAgain
function renderWithHooksAgain<Props, SecondArg>(
  workInProgress: Fiber,
  Component: (p: Props, arg: SecondArg) => any,
  props: Props,
  secondArg: SecondArg,
): any {
  // This is used to perform another render pass. It's used when setState is
  // called during render, and for double invoking components in Strict Mode
  // during development.
  //
  // The state from the previous pass is reused whenever possible. So, state
  // updates that were already processed are not processed again, and memoized
  // functions (`useMemo`) are not invoked again.
  //
  // Keep rendering in a loop for as long as render phase updates continue to
  // be scheduled. Use a counter to prevent infinite loops.

  currentlyRenderingFiber = workInProgress;

  let numberOfReRenders: number = 0;
  let children;
  do {
    if (didScheduleRenderPhaseUpdateDuringThisPass) {
      // It's possible that a use() value depended on a state that was updated in
      // this rerender, so we need to watch for different thenables this time.
      thenableState = null;
    }
    thenableIndexCounter = 0;
    didScheduleRenderPhaseUpdateDuringThisPass = false;

    if (numberOfReRenders >= RE_RENDER_LIMIT) {
      throw new Error(
        'Too many re-renders. React limits the number of renders to prevent ' +
          'an infinite loop.',
      );
    }

    numberOfReRenders += 1; // 리렌더 숫자
    if (__DEV__) {
      // Even when hot reloading, allow dependencies to stabilize
      // after first render to prevent infinite render phase updates.
      ignorePreviousDependencies = false;
    }

    // Start over from the beginning of the list
    // update를 소비하기 위해 필요한 변수들 값을 null 할당
    currentHook = null;
    workInProgressHook = null;

    if (workInProgress.updateQueue != null) {
      resetFunctionComponentUpdateQueue((workInProgress.updateQueue: any));
    }

    if (__DEV__) {
      // Also validate hook order for cascading updates.
      hookTypesUpdateIndexDev = -1;
    }

    ReactSharedInternals.H = __DEV__
      ? HooksDispatcherOnRerenderInDEV
      : HooksDispatcherOnRerender; // hook 구현체

    children = __DEV__
      ? callComponentInDEV(Component, props, secondArg)
      : Component(props, secondArg); // 컴포넌트 재호출 -> 하는 즉시 위의 과정 다시 반복(didScheduleRenderPhaseUpdate가 true) , numberOfReRenders가 RE_RENDER_LIMIT 넘어가면 에러 발생
  } while (didScheduleRenderPhaseUpdateDuringThisPass);
  return children;
}
```
