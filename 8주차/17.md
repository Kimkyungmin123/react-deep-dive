# React Scheduler — `taskQueue`와 `timerQueue`의 이중 큐 구조

## 1. 왜 큐가 두 개인가?

### 1.1 단일 큐의 문제

과거에 모든 Task를 하나의 `taskQueue`로 관리할 때의 문제:

```
taskQueue (단일 큐, expirationTime 기준 정렬)
  [Task A: exp=1000] [Task B: exp=2000] [Task C: exp=5000]

→ Task B에 delay를 줘서 나중에 실행하고 싶다면?
  큐에서 꺼내기 → 대기 → 다시 삽입 → 정렬 재조정
  → 복잡하고 불필요한 연산 발생
```

### 1.2 이중 큐로 해결

| 큐           | 역할                                    | 정렬 기준                   | 포함되는 Task             |
| ------------ | --------------------------------------- | --------------------------- | ------------------------- |
| `timerQueue` | delay가 있어 아직 실행 불가한 Task 대기 | `startTime` (오름차순)      | `startTime > currentTime` |
| `taskQueue`  | 즉시 실행 가능한 Task 처리              | `expirationTime` (오름차순) | `startTime ≤ currentTime` |

```
timerQueue (startTime 기준)          taskQueue (expirationTime 기준)
  [delay=500ms, start=1500]            [exp=1050, Normal]
  [delay=1000ms, start=2000]           [exp=1100, UserBlocking]
  [delay=2000ms, start=3000]           [exp=6000, Idle]
         │
         │ delay 경과 시 advanceTimers가 이동
         ▼
       taskQueue
```

## 2. Task 등록 흐름

### 2.1 일반 Task (`startTime ≤ currentTime`)

```javascript
if (startTime <= currentTime) {
  // 즉시 실행 가능 → taskQueue에 삽입
  task.sortIndex = expirationTime;
  push(taskQueue, task);

  // 현재 호스트 콜백이 없으면 등록
  if (scheduledHostCallback === null) {
    requestHostCallback(flushWork);
  }
}
```

- `sortIndex = expirationTime` → 만료가 임박한 Task가 큐의 앞에 위치
- `requestHostCallback(flushWork)` → 브라우저에게 `flushWork`(Task 소비 함수) 실행을 요청

### 2.2 타이머 Task (`startTime > currentTime`)

```javascript
if (startTime > currentTime) {
  // 아직 실행 불가 → timerQueue에 삽입
  task.sortIndex = startTime;
  push(timerQueue, task);

  // taskQueue가 비어있으면 타임아웃 예약
  if (peek(taskQueue) === null) {
    requestHostTimeout(handleTimeout, startTime - currentTime);
  }
}
```

- `sortIndex = startTime` → 가장 빨리 시작되어야 할 Task가 큐의 앞에 위치
- `requestHostTimeout` → `setTimeout`을 통해 delay가 지난 후 `handleTimeout` 실행을 예약

## 3. 타이머 만료 처리: `handleTimeout`과 `advanceTimers`

### 3.1 `advanceTimers` — 만료된 타이머를 `taskQueue`로 이동

```javascript
function advanceTimers(currentTime) {
  let timer = peek(timerQueue); // timerQueue의 맨 앞 Task 확인

  while (timer !== null) {
    if (timer.callback === null) {
      // 취소된 Task(callback이 null) → 제거
      pop(timerQueue);
    } else if (timer.startTime <= currentTime) {
      // delay가 경과하여 실행 가능한 상태 → taskQueue로 이동
      pop(timerQueue);
      timer.sortIndex = timer.expirationTime; // 정렬 기준을 expirationTime으로 변경
      push(taskQueue, timer);
    } else {
      // 아직 delay가 남은 Task → 루프 종료
      break;
    }
    timer = peek(timerQueue);
  }
}
```

### 3.2 `handleTimeout` — 타임아웃 발생 시 처리

```javascript
function handleTimeout(currentTime) {
  advanceTimers(currentTime); // 만료된 타이머를 taskQueue로 이동

  if (scheduledHostCallback === null) {
    if (peek(taskQueue) !== null) {
      // taskQueue에 처리할 Task가 있으면 hostCallback 등록
      requestHostCallback(flushWork);
    } else {
      // taskQueue는 비어있지만 timerQueue에 남은 Task가 있으면
      // 다음 타이머를 위한 타임아웃 재예약
      const firstTimer = peek(timerQueue);
      if (firstTimer !== null) {
        requestHostTimeout(handleTimeout, firstTimer.startTime - currentTime);
      }
    }
  }
}
```

**`handleTimeout`의 재귀적 호출 구조:**

```
requestHostTimeout(handleTimeout, delay)
        │ (delay ms 후)
        ▼
handleTimeout(currentTime)
  → advanceTimers: 만료된 타이머 이동
  → taskQueue 비어있고 timerQueue에 남은 것 있음?
        YES → requestHostTimeout(handleTimeout, 다음 delay) ← 재귀
        NO  → requestHostCallback(flushWork) → 실행
```

## 4. HostConfig와 스케줄링 환경

**HostConfig**는 React가 다양한 호스트 환경(브라우저, React Native 등)에서 동작할 수 있도록 **환경별 스케줄링 구현을 분리**하는 코드 모음이다.

| 함수                  | 브라우저 구현                  | 역할                      |
| --------------------- | ------------------------------ | ------------------------- |
| `requestHostCallback` | `MessageChannel.postMessage()` | Task 큐 소비 요청         |
| `requestHostTimeout`  | `setTimeout(fn, delay)`        | delay 후 타이머 처리 예약 |
| `cancelHostTimeout`   | `clearTimeout()`               | 타임아웃 취소             |
| `getCurrentTime`      | `performance.now()`            | 현재 시간 획득            |

> ⚠️ **`requestHostCallback`이 `setTimeout`이 아닌 `MessageChannel`을 사용하는 이유:**  
> `setTimeout(fn, 0)`은 브라우저 스펙상 최소 1~4ms의 지연이 발생한다.  
> `MessageChannel`은 이 지연 없이 현재 동기 실행 스택이 비워지면 즉시 실행된다.

---

## 5. 전체 스케줄링 흐름

```
scheduleCallback(priorityLevel, callback, options?)
        │
        ├─ startTime 계산
        │     delay 있음 → currentTime + delay  → timerQueue
        │     delay 없음 → currentTime          → taskQueue
        │
        ├─ [timerQueue 경로]
        │     timerQueue에 push (sortIndex = startTime)
        │     requestHostTimeout(handleTimeout, delay)
        │           │ (delay ms 후)
        │           ▼
        │     handleTimeout
        │           ├─ advanceTimers → 만료 타이머를 taskQueue로 이동
        │           └─ requestHostCallback(flushWork) 또는 재귀 타임아웃
        │
        └─ [taskQueue 경로]
              taskQueue에 push (sortIndex = expirationTime)
              requestHostCallback(flushWork)
                    │ (MessageChannel → 다음 태스크 큐)
                    ▼
              flushWork → workLoop → Task.callback 실행
```

## 6. React 18+과의 대응 개념

| React 16/17                              | React 18+     | 변화 요약                  |
| ---------------------------------------- | ------------- | -------------------------- |
| `taskQueue` / `timerQueue` 이중 구조     | 동일하게 유지 | 구조 변화 없음             |
| `advanceTimers`                          | 동일하게 유지 | 변화 없음                  |
| `handleTimeout`                          | 동일하게 유지 | 변화 없음                  |
| `requestHostCallback` (`MessageChannel`) | 동일하게 유지 | 변화 없음                  |
| `requestHostTimeout` (`setTimeout`)      | 동일하게 유지 | 변화 없음                  |
| HostConfig 개념                          | 동일하게 유지 | 환경별 구현 분리 원칙 유지 |

📌 **React 18에서의 변화**: Scheduler 자체의 큐 구조와 타이밍 메커니즘은 React 16/17과 동일하다.  
달라진 것은 Reconciler 측이다. React 18에서는 Lane 기반 우선순위가 `lanesToSchedulerPriority`로 변환되어 Scheduler에 전달되며, Scheduler는 이를 그대로 Task 객체에 담아 기존과 동일하게 처리한다.

# React 18+ — Scheduler 이중 큐와 Concurrent 렌더링 타이밍

## 1. Scheduler 큐와 Concurrent 렌더링의 관계

`startTransition`이나 일반 `setState`가 호출되면 최종적으로 Scheduler의 큐에 Task가 등록된다.  
Scheduler가 이 Task를 **언제 실행하느냐**가 곧 Concurrent 렌더링의 "인터럽트 가능성"을 결정한다.

```
startTransition(() => setState(value))
        │
        ▼  (TransitionLane → NormalPriority)
scheduleCallback(NormalPriority, performConcurrentWorkOnRoot)
        │
        ▼
taskQueue에 Task 삽입
        │
        ▼  (MessageChannel → 브라우저 스택 비워진 후)
workLoop() 실행 → 5ms 예산 내에서 렌더링
        │
        ├─ 예산 소진 → yield → 브라우저에 제어권 반환
        └─ 더 긴급한 Task 도착 → 현재 Task 중단 → 긴급 Task 먼저 처리
```

---

## 2. 두 큐의 역할 (React 18에서도 동일)

Scheduler의 이중 큐 구조는 React 18에서도 그대로 유지된다.

```
timerQueue                          taskQueue
(delay가 있어 대기 중)              (즉시 실행 가능)
  sortIndex = startTime               sortIndex = expirationTime
  ┌──────────────────┐               ┌──────────────────┐
  │ delay=200ms Task │──만료 시 이동──►│ exp=1050 Task    │ ← 다음 실행 대상
  │ delay=500ms Task │               │ exp=2000 Task    │
  └──────────────────┘               └──────────────────┘
```

### 2.1 실무에서 `timerQueue`를 직접 건드리는 경우

일반 `setState`나 `startTransition`은 delay 없이 즉시 `taskQueue`에 등록된다.  
`timerQueue`를 사용하는 경우는 다음과 같이 제한적이다.

```javascript
// Scheduler를 직접 사용할 때만 delay 옵션을 줄 수 있음
import { scheduleCallback, NormalPriority } from "scheduler";

scheduleCallback(NormalPriority, myTask, { delay: 500 });
// → 500ms 후 timerQueue에서 taskQueue로 이동
```

React의 공개 API(`setState`, `startTransition` 등)는 delay를 사용하지 않는다.  
따라서 일반 React 개발에서 `timerQueue`는 간접적으로만 관련된다.

## 3. `workLoop` — Task 실행의 핵심

`requestHostCallback(flushWork)`가 호출되면 `MessageChannel`을 통해 다음 태스크 큐에서 `workLoop`가 실행된다.

```javascript
function workLoop(hasTimeRemaining, initialTime) {
  let currentTime = initialTime;
  advanceTimers(currentTime); // timerQueue → taskQueue 이동 먼저 처리

  currentTask = peek(taskQueue);

  while (currentTask !== null) {
    if (
      currentTask.expirationTime > currentTime && // 아직 만료 전이고
      (!hasTimeRemaining || shouldYieldToHost()) // 시간 예산 소진됨
    ) {
      break; // 브라우저에 제어권 반환 (yield)
    }

    const callback = currentTask.callback;
    if (typeof callback === "function") {
      currentTask.callback = null;
      const continuationCallback = callback(
        currentTask.expirationTime <= currentTime,
      );
      //                                     ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
      //                                     didTimeout: 만료 여부를 콜백에 전달

      if (typeof continuationCallback === "function") {
        // 콜백이 함수를 반환하면 → 작업이 아직 남아있음 → 다음 프레임에서 재개
        currentTask.callback = continuationCallback;
      } else {
        pop(taskQueue); // 작업 완료 → 큐에서 제거
      }
    } else {
      pop(taskQueue); // callback이 null(취소됨) → 제거
    }

    currentTask = peek(taskQueue);
  }
}
```

### 3.1 `shouldYieldToHost` — 브라우저에 양보하는 기준

```javascript
function shouldYieldToHost() {
  const timeElapsed = getCurrentTime() - startTime;
  return timeElapsed >= frameInterval; // 기본 5ms
}
```

- 한 프레임 예산(기본 **5ms**)을 초과하면 `true`를 반환
- `workLoop`는 이 시점에 `break`하여 브라우저에 제어권을 돌려줌
- 다음 `MessageChannel` 이벤트에서 남은 Task 이어서 처리

## 4. Concurrent 렌더링에서 "중단"이 일어나는 실제 메커니즘

"Concurrent 렌더링은 중단 가능하다"는 말의 정확한 의미를 구체적으로 이해한다.

### 4.1 Fiber 렌더링 중 `shouldYield` 호출

`performConcurrentWorkOnRoot` (Reconciler의 렌더링 함수)는 각 Fiber 처리 사이에 `shouldYield()`를 호출한다.

```javascript
function workLoopConcurrent() {
  while (workInProgress !== null && !shouldYield()) {
    // Fiber 하나 처리
    performUnitOfWork(workInProgress);
  }
  // shouldYield()가 true를 반환하면 루프 탈출 → 렌더링 일시 중단
}
```

```
shouldYield() → Scheduler의 shouldYieldToHost() 호출
              → 5ms 초과 여부 반환
```

### 4.2 중단 후 재개 흐름

```
workLoopConcurrent 루프 탈출 (shouldYield = true)
        │
        ▼
performConcurrentWorkOnRoot가 continuationCallback 반환
        │
        ▼
Scheduler workLoop: continuationCallback을 currentTask.callback에 저장
        │
        ▼
브라우저에 제어권 반환 (페인트, 이벤트 처리 가능)
        │
        ▼
다음 MessageChannel 이벤트 → workLoop 재개 → 남은 Fiber 처리
```

## 5. `startTransition`으로 인터럽트 가능한 렌더링 만들기

```jsx
function SearchPage() {
  const [input, setInput] = useState("");
  const [results, setResults] = useState([]);

  function handleChange(e) {
    setInput(e.target.value); // SyncLane → 즉시 처리 (인터럽트 불가)

    startTransition(() => {
      setResults(search(e.target.value)); // TransitionLane → 인터럽트 가능
    });
  }
}
```

**우선순위 충돌 시 동작:**

```
[상황] TransitionLane 렌더링 중 새 키 입력 발생

1. workLoopConcurrent: shouldYield() = true → 렌더링 중단
2. 새 SyncLane Task가 taskQueue 앞에 삽입
3. workLoop: SyncLane Task 먼저 처리 (입력값 즉시 반영)
4. 기존 TransitionLane Task의 continuationCallback 실행 → 처음부터 재시작
   (이때 setResults의 search는 최신 input값 기준으로 재계산됨)
```

---

## 6. Task 만료(starvation) 방지

`startTransition` Task는 `NormalPriority`(timeout = 5000ms)를 가진다.  
5초 동안 계속 인터럽트되면 만료(`expirationTime` 초과) 상태가 된다.

```javascript
// workLoop에서 만료 Task 처리
if (currentTask.expirationTime <= currentTime) {
  // 만료된 Task → didTimeout = true → 더 이상 yield 없이 강제 실행
  callback(true /* didTimeout */);
}
```

만료된 Task는 `shouldYield()`가 `true`여도 중단되지 않고 **강제로 완료**된다.

## 7. 전체 흐름 요약

```
[React 18 Concurrent 렌더링 타이밍 전체 그림]

startTransition(() => setState)
        │ TransitionLane → NormalPriority
        ▼
scheduleCallback(NormalPriority, performConcurrentWorkOnRoot)
        │ startTime = currentTime (delay 없음)
        ▼
taskQueue에 삽입 (sortIndex = expirationTime)
        │
        ▼
requestHostCallback(flushWork)
        │ MessageChannel.postMessage()
        ▼  [다음 태스크 큐]
workLoop()
        ├─ 5ms 예산 내 → Fiber 처리 (workLoopConcurrent)
        ├─ 예산 소진 → yield → continuationCallback 저장 → 브라우저 반환
        ├─ 더 긴급한 Task → 현재 Task 인터럽트 → 긴급 Task 처리 후 재시작
        └─ expirationTime 초과 → 강제 완료 (starvation 방지)
```

### 핵심 체크리스트

- [ ] `taskQueue`와 `timerQueue`의 역할 차이를 이해했다.
- [ ] `workLoop`가 5ms 예산 기준으로 브라우저에 제어권을 반환함을 이해했다.
- [ ] `workLoopConcurrent`에서 `shouldYield()`로 Fiber 처리를 중단함을 이해했다.
- [ ] `continuationCallback`으로 렌더링이 재개됨을 이해했다.
- [ ] `expirationTime` 초과 시 starvation 방지를 위해 강제 실행됨을 이해했다.
- [ ] `startTransition` Task가 5초 후 만료되어 강제 처리됨을 이해했다.
