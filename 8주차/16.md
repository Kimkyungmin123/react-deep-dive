# React Scheduler — Task 객체의 구조와 시간 계산

## 1. Scheduler의 역할과 Task 객체의 등장 배경

### 1.1 Scheduler가 하는 일

Scheduler(`react-scheduler` 패키지)는 **콜백을 언제 실행할지**를 결정하는 패키지이다.  
Reconciler로부터 콜백을 전달받으면, 이를 즉시 실행하는 것이 아니라 **우선순위와 시간 기준**에 따라 적절한 시점에 실행한다.

```
Reconciler
    └─ scheduleCallback(priorityLevel, performConcurrentWorkOnRoot)
                │
                ▼
           Scheduler
               └─ Task 객체 생성
               └─ Task 큐(우선순위 큐)에 삽입
               └─ 적절한 시점에 Task.callback 실행
```

### 1.2 Task 객체가 필요한 이유

콜백 함수 하나만으로는 **언제 실행해야 하는지, 얼마나 긴급한지** 알 수 없다.  
Scheduler는 전달받은 콜백을 **Task 객체**로 감싸 필요한 메타데이터를 함께 관리한다.

---

## 2. Task 객체의 주요 속성

```javascript
const task = {
  id, // Task 고유 식별자 (생성 순서 기반 정수)
  callback, // 실행할 함수 (performConcurrentWorkOnRoot 바인딩)
  priorityLevel, // Reconciler가 전달한 우선순위 레벨
  startTime, // Task 실행이 시작되어야 하는 시간
  expirationTime, // Task가 만료되는 시간 (이 시점이 지나면 강제 처리)
  sortIndex, // 우선순위 큐 정렬 기준값
};
```

### 2.1 속성 출처 분류

| 속성             | 출처                | 설명                                                                                 |
| ---------------- | ------------------- | ------------------------------------------------------------------------------------ |
| `callback`       | Reconciler가 전달   | 실행할 작업 함수                                                                     |
| `priorityLevel`  | Reconciler가 전달   | ImmediatePriority ~ IdlePriority                                                     |
| `startTime`      | Scheduler 내부 계산 | `currentTime + delay`                                                                |
| `expirationTime` | Scheduler 내부 계산 | `startTime + timeout`                                                                |
| `sortIndex`      | Scheduler 내부 계산 | 큐 정렬 기준 (아직 시작 안 된 Task는 `startTime`, 진행 중인 Task는 `expirationTime`) |
| `id`             | Scheduler 내부 생성 | 순서 보장을 위한 고유 번호                                                           |

---

## 3. `startTime`과 `expirationTime` 계산

### 3.1 계산 공식

```
startTime      = currentTime + delay   (delay가 없으면 currentTime)
expirationTime = startTime + timeout
```

- **`currentTime`**: `performance.now()`로 얻는 현재 시각(ms)
- **`delay`**: Task 실행을 미룰 시간. Concurrent 모드에서 `options.delay`로 전달됨
- **`timeout`**: 이 시간이 지나면 Task가 만료되어 강제 처리됨

### 3.2 `options` 전달 여부에 따른 분기

```javascript
function scheduleCallback(priorityLevel, callback, options) {
  const currentTime = getCurrentTime(); // performance.now()

  // startTime 계산
  let startTime;
  if (options !== null && options !== undefined && options.delay > 0) {
    startTime = currentTime + options.delay; // delay가 있으면 나중에 시작
  } else {
    startTime = currentTime; // delay 없으면 즉시 시작 가능
  }

  // timeout 결정
  let timeout;
  if (
    options !== null &&
    options !== undefined &&
    options.timeout !== undefined
  ) {
    timeout = options.timeout; // 명시적 timeout 사용
  } else {
    timeout = timeoutForPriorityLevel(priorityLevel); // 우선순위 기반 timeout
  }

  const expirationTime = startTime + timeout;

  const task = {
    id: taskIdCounter++,
    callback,
    priorityLevel,
    startTime,
    expirationTime,
    sortIndex: -1,
  };
  // ...
}
```

> ⚠️ **`options`가 전달되는 경우**: Concurrent Work(`scheduleCallback` 경로)에서만 `options`가 전달될 수 있다.  
> Sync Work(`scheduleSyncCallback` 경로)는 `options` 없이 호출된다.

---

## 4. `timeoutForPriorityLevel` 함수

`options.timeout`이 없을 때 우선순위 레벨에 따라 적절한 `timeout` 값을 반환하는 함수이다.

```javascript
function timeoutForPriorityLevel(priorityLevel) {
  switch (priorityLevel) {
    case ImmediatePriority:
      return IMMEDIATE_PRIORITY_TIMEOUT; // -1
    case UserBlockingPriority:
      return USER_BLOCKING_PRIORITY_TIMEOUT; // 250ms
    case NormalPriority:
      return NORMAL_PRIORITY_TIMEOUT; // 5000ms
    case LowPriority:
      return LOW_PRIORITY_TIMEOUT; // 10000ms
    case IdlePriority:
      return IDLE_PRIORITY_TIMEOUT; // maxSigned31BitInt (사실상 무한)
  }
}
```

### 4.1 우선순위별 timeout 값

| 우선순위 레벨          | timeout 값  | 의미                            |
| ---------------------- | ----------- | ------------------------------- |
| `ImmediatePriority`    | **-1ms**    | 생성 즉시 만료 → 가장 먼저 처리 |
| `UserBlockingPriority` | 250ms       | 사용자 인터랙션 응답 기준       |
| `NormalPriority`       | 5000ms      | 일반 업데이트                   |
| `LowPriority`          | 10000ms     | 느긋하게 처리 가능              |
| `IdlePriority`         | 사실상 무한 | 모든 Work가 끝난 후 처리        |

### 4.2 `ImmediatePriority`의 timeout이 -1인 이유

```
startTime      = currentTime (예: 1000ms)
timeout        = -1
expirationTime = startTime + timeout = 1000 + (-1) = 999ms

→ expirationTime(999) < startTime(1000)
→ Task가 생성되는 순간 이미 만료 상태
→ Scheduler는 이 Task를 즉시 처리해야 한다고 판단
```

`timeout = -1`은 "이 Task는 기다릴 여유가 없다. 지금 당장 실행하라"는 의미이다.

---

## 5. Task 큐와 `sortIndex`

Task 객체는 생성 후 **우선순위 큐(min-heap)** 에 삽입된다.  
큐에서 정렬 기준은 `sortIndex`이다.

| Task 상태                                   | `sortIndex` 값   | 속하는 큐                   |
| ------------------------------------------- | ---------------- | --------------------------- |
| `startTime > currentTime` (아직 시작 안 됨) | `startTime`      | `timerQueue` (지연 Task 큐) |
| `startTime ≤ currentTime` (실행 가능)       | `expirationTime` | `taskQueue` (실행 대기 큐)  |

```
timerQueue: delay가 있어 아직 실행할 수 없는 Task 모음
taskQueue:  즉시 실행 가능한 Task 모음 (expirationTime이 작을수록 앞에 위치)
```

---

## 6. React 18+과의 대응 개념

| React 16/17                               | React 18+     | 변화 요약                                                     |
| ----------------------------------------- | ------------- | ------------------------------------------------------------- |
| Task 객체 구조                            | 동일하게 유지 | 구조 변화 없음                                                |
| `priorityLevel` (Scheduler 자체 우선순위) | 동일하게 유지 | Reconciler의 Lane → `lanesToSchedulerPriority`로 변환 후 전달 |
| `timeoutForPriorityLevel`                 | 동일하게 유지 | 함수명 및 timeout 값 동일                                     |
| `ImmediatePriority timeout = -1`          | 동일하게 유지 | 즉시 만료 패턴 유지                                           |
| `options.delay`                           | 동일하게 유지 | 사용 빈도는 낮음                                              |

📌 **React 18의 변화**: React 18에서 Reconciler가 Scheduler에 전달하는 `priorityLevel`은 Lane에서 변환된 값이다.  
`lanesToSchedulerPriority(lanes)` 함수가 Lane 비트마스크를 Scheduler의 우선순위 레벨로 변환한다.  
Scheduler 자체의 Task 객체 구조와 시간 계산 방식은 React 16/17과 거의 동일하다.

---

## 7. 전체 흐름 요약

```
scheduleCallback(priorityLevel, performConcurrentWorkOnRoot, options?)
        │
        ├─ currentTime = getCurrentTime()        // performance.now()
        │
        ├─ startTime 계산
        │     options.delay > 0  →  currentTime + delay
        │     그 외              →  currentTime
        │
        ├─ timeout 결정
        │     options.timeout 있음  →  options.timeout
        │     없음                 →  timeoutForPriorityLevel(priorityLevel)
        │                               ImmediatePriority → -1
        │                               UserBlocking      → 250ms
        │                               Normal            → 5000ms
        │                               ...
        │
        ├─ expirationTime = startTime + timeout
        │
        ├─ Task 객체 생성 { id, callback, priorityLevel, startTime, expirationTime, sortIndex }
        │
        └─ startTime > currentTime?
              YES → timerQueue에 삽입 (sortIndex = startTime)
              NO  → taskQueue에 삽입  (sortIndex = expirationTime)
```

# React 18+ — Scheduler Task 객체와 우선순위 실행 타이밍

## 1. Scheduler 패키지의 위치

React의 렌더링 파이프라인에서 Scheduler는 다음 위치에 있다.

```
개발자 코드 (setState, startTransition)
        │
        ▼
Reconciler (react-reconciler)
  → ensureRootIsScheduled
  → scheduleCallback(priorityLevel, callback)
        │
        ▼
Scheduler (scheduler 패키지)   ← 이번 문서의 범위
  → Task 객체 생성
  → 우선순위 큐 관리
  → 브라우저 여유 시간에 callback 실행
        │
        ▼
performConcurrentWorkOnRoot 실행 → 렌더링 시작
```

Scheduler는 **React와 완전히 독립된 패키지**이므로, React 없이 단독으로 사용하는 것도 가능하다.

---

## 2. Task 객체 구조

Scheduler가 콜백을 관리하기 위해 생성하는 내부 객체이다.

```javascript
const task = {
  id, // 생성 순서 기반 고유 번호
  callback, // 실행할 함수
  priorityLevel, // ImmediatePriority ~ IdlePriority
  startTime, // 실행 가능 시작 시점 (ms)
  expirationTime, // 만료 시점 (ms) — 지나면 강제 처리
  sortIndex, // 큐 내 정렬 기준
};
```

---

## 3. React 18에서 `priorityLevel`이 결정되는 방식

React 18의 Reconciler는 Lane을 Scheduler의 `priorityLevel`로 변환하여 전달한다.

```javascript
// Reconciler 내부 (개념적 표현)
function ensureRootIsScheduled(root) {
  const nextLanes = getNextLanes(root, ...);
  const schedulerPriorityLevel = lanesToSchedulerPriority(nextLanes);

  scheduleCallback(schedulerPriorityLevel, performConcurrentWorkOnRoot);
}
```

### 3.1 Lane → Scheduler Priority 변환표

| Lane                  | Scheduler Priority     | timeout          |
| --------------------- | ---------------------- | ---------------- |
| `SyncLane`            | `ImmediatePriority`    | -1ms (즉시 만료) |
| `InputContinuousLane` | `UserBlockingPriority` | 250ms            |
| `DefaultLane`         | `NormalPriority`       | 5000ms           |
| `TransitionLane`      | `NormalPriority`       | 5000ms           |
| `IdleLane`            | `IdlePriority`         | 사실상 무한      |

> ⚠️ `startTransition`으로 발생한 업데이트(`TransitionLane`)도 Scheduler에서는 `NormalPriority`로 처리된다.  
> 차이는 **Lane 비교 단계(Reconciler)**에서 발생한다. Scheduler는 그 결과만 받아 실행 타이밍을 관리한다.

---

## 4. `expirationTime`과 Task 만료의 의미

Scheduler의 `expirationTime`은 React 16의 Reconciler `expirationTime`과 **완전히 다른 개념**이다.

| 구분          | Reconciler의 `expirationTime` (React 16) | Scheduler의 `task.expirationTime` |
| ------------- | ---------------------------------------- | --------------------------------- |
| 목적          | Work의 우선순위 판단 기준                | Task를 언제까지 처리해야 하는가   |
| React 18 대체 | `Lanes` 비트마스크                       | 동일하게 유지                     |
| 값이 크면     | 높은 우선순위 (먼저 발생한 이벤트)       | 만료 시점이 더 늦음               |

### 4.1 Task 만료 시 동작

Task가 만료(`currentTime > task.expirationTime`)되면 Scheduler는 해당 Task를 **즉시 처리**한다.  
브라우저 프레임 예산(보통 5ms)이 남지 않아도 강제로 실행한다.

```
일반 처리:
  브라우저 프레임 여유 있음 → Task 실행 → 여유 없으면 다음 프레임으로 양보

만료된 Task 처리:
  expirationTime 초과 → 여유 없어도 강제 실행 (starvation 방지)
```

---

## 5. 두 개의 큐: `taskQueue`와 `timerQueue`

Scheduler는 Task를 두 개의 **min-heap** 큐로 관리한다.

```
timerQueue (지연 Task)          taskQueue (실행 가능 Task)
  startTime > currentTime         startTime ≤ currentTime
  sortIndex = startTime           sortIndex = expirationTime
  ┌─────────────────┐             ┌─────────────────┐
  │ delay가 있는     │             │ 지금 바로 실행   │
  │ Task들 대기      │──도달──────►│ 가능한 Task들    │
  └─────────────────┘             └─────────────────┘
```

### 5.1 `sortIndex`의 의미

- `taskQueue`에서는 **`expirationTime`이 작은 Task가 앞에 위치** (더 빨리 만료될수록 먼저 처리)
- `timerQueue`에서는 **`startTime`이 작은 Task가 앞에 위치** (더 빨리 시작될 Task가 앞에)

---

## 6. 브라우저와 Scheduler의 협력: `MessageChannel`

Scheduler는 브라우저 메인 스레드와 협력하기 위해 `MessageChannel`을 사용한다.

```
taskQueue에 Task 삽입
        │
        ▼
MessageChannel.postMessage()  ← 비동기로 브라우저에 신호
        │
        ▼ (현재 동기 실행 스택이 비워진 후)
MessageChannel onmessage 콜백 실행
        │
        ▼
workLoop() 시작
  → 브라우저 프레임 예산(5ms) 내에서 Task 처리
  → 예산 소진 → 양보(yield)
  → 다음 MessageChannel 이벤트에서 재개
```

> ⚠️ `setTimeout(fn, 0)` 대신 `MessageChannel`을 사용하는 이유:  
> `setTimeout`은 브라우저마다 최소 1~4ms의 지연이 있어 고빈도 Task 처리에 부적합하다.  
> `MessageChannel`은 이 지연 없이 **다음 태스크 큐**에 즉시 진입할 수 있다.

---

## 7. `ImmediatePriority` Task의 특별 처리

`ImmediatePriority`(timeout = -1)의 Task는 생성 즉시 만료 상태이므로, `workLoop`가 돌기 전에 **동기적으로 바로 실행**된다.

```javascript
// flushSyncCallbacks 호출 시나리오
// SyncLane 업데이트 → scheduleSyncCallback → ImmediatePriority Task 생성
// → executionContext === NoContext → 즉시 flushSyncCallbacks 실행
// → Task.callback 동기 실행 → 렌더링 완료
```

## 8. 전체 정리

```
[React 18 Scheduler Task 생명주기]

Reconciler
  └─ scheduleCallback(NormalPriority, performConcurrentWorkOnRoot)
                │
                ▼
         Scheduler
           ├─ currentTime = performance.now()
           ├─ startTime = currentTime (delay 없음)
           ├─ timeout = 5000ms (NormalPriority)
           ├─ expirationTime = startTime + 5000
           ├─ Task 객체 생성
           └─ taskQueue에 삽입 (sortIndex = expirationTime)
                │
                ▼
         MessageChannel.postMessage()
                │ (현재 스택 비워진 후)
                ▼
         workLoop()
           ├─ 브라우저 프레임 예산(5ms) 내에서 Task 처리
           ├─ 예산 소진 → yield → 다음 프레임에서 재개
           └─ expirationTime 초과 Task → 강제 실행 (starvation 방지)
```

### 핵심 체크리스트

- [ ] Scheduler의 Task 객체는 `callback`, `priorityLevel`, `startTime`, `expirationTime`, `sortIndex`를 포함함을 이해했다.
- [ ] React 18에서 Lane이 `lanesToSchedulerPriority`로 변환되어 Scheduler에 전달됨을 이해했다.
- [ ] Scheduler의 `task.expirationTime`은 Reconciler의 `expirationTime`(React 16)과 다른 개념임을 이해했다.
- [ ] `ImmediatePriority`의 timeout이 -1인 이유(생성 즉시 만료 → 즉각 처리)를 이해했다.
- [ ] `taskQueue`(실행 가능)와 `timerQueue`(지연 대기)가 분리된 이유를 이해했다.
- [ ] Scheduler가 `MessageChannel`을 통해 브라우저 메인 스레드와 협력함을 이해했다.
